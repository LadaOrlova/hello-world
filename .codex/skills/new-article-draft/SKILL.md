# Скилл: New Article Draft (Double Diamond — СТРУКТУРИРОВАНИЕ)

Ключевой принцип: **суть каждого тезиса = каркас секции**. Полировать, обогащать и адаптировать под стиль издания, а не перепридумывать логику тезиса.

## Рабочие пути и инициализация

Перед стартом вычислить корень репозитория и рабочие каталоги:

```bash
ROOT="$(git rev-parse --show-toplevel 2>/dev/null || pwd)"

# Canonical dirs for PR-articles (always write results here)
THESIS_DIR="$ROOT/3-Marketing/2-PR-Articles/2-Theses"
DRAFT_DIR="$ROOT/3-Marketing/2-PR-Articles/3-Drafts"

# Optional legacy dirs (read-only, if you need to look up older files)
THESIS_DIR_LEGACY="$ROOT/3-Marketing/2-Writing-Book/2-Theses"
DRAFT_DIR_LEGACY="$ROOT/3-Marketing/2-Writing-Book/3-Drafts"

BOOKCONCEPT_PATH="$ROOT/3-Marketing/2-Writing-Book/bookconcept.md"

mkdir -p "$THESIS_DIR" "$DRAFT_DIR"
```

Собрать slug и temp-пути:

```bash
SLUG="$(printf '%s' "{название-статьи}" \
  | tr '[:upper:]' '[:lower:]' \
  | sed -E 's/[^a-z0-9]+/-/g; s/^-+|-+$//g; s/-+/-/g')"

TEMP_DIR="$ROOT/tmp/draft-${SLUG}"
THESIS_TEMP_DIR="$ROOT/tmp/thesis-${SLUG}"
mkdir -p "$TEMP_DIR"
```

`temp-папка` ниже всегда означает `$TEMP_DIR`.

## Входные данные

Пользователь даёт:
1. Путь к файлу тезисов (`Theses-*.md`) или название статьи.
2. Опционально: целевое издание (название + URL).
3. Опционально: дополнительные требования к языку/объёму/структуре.
4. Для итерации: путь к текущему `Draft-*.md` и комментарии автора.

## Определение режима

Сразу определить режим:

- `CREATION`:
  - Есть готовый файл тезисов и нужно сделать новый черновик.
- `ITERATION`:
  - Есть существующий файл `Draft-*.md` и запрос на доработку.
  - Или в черновике заполнено поле с авторским фидбэком.

Маршрутизация:

- `CREATION`: Этапы 0 → 1 → 2 → 3 → 4 → 5 → 6.
- `ITERATION`: Этапы И1 → И2 → И3.

## Anti-Compaction протокол

После каждого этапа записывать промежуточный результат в temp-папку и обновлять `_spring-context.md`.

Файлы состояния:

- `00-must-include.md` — полный список обязательных элементов из тезисов.
- `01-publication-signal-pack.md` — топ-материалы издания + сигналы популярности.
- `02-style-fingerprint.md` — tone of voice, best practices, anti-patterns.
- `03-case-pack.md` — подтверждённые кейсы по тезисам.
- `04-narrative-plan.md` — hook, порядок секций, переходы, word budget.
- `05-section-briefs.md` — секционные брифы для сборки текста.
- `06-draft-v1.md` — собранный черновик.
- `07-quality-report.md` — результаты quality gates.
- `_spring-context.md` — текущее состояние и следующий шаг.

Если есть `$THESIS_TEMP_DIR`, прочитать `_spring-context.md` и `agent-*.md` от thesis-этапа перед началом.

## ЭТАП 0: Извлечь MUST-INCLUDE из тезисов

Цель: не потерять аргументацию.

1. Прочитать файл тезисов целиком.
2. Для каждого тезиса извлечь дословно:
   - `СУТЬ` (полный текст).
   - `DATA` (все цифры/факты).
   - `QUOTE` (точные цитаты с URL).
   - `CONCEPT` (фреймворки/термины).
   - `KB` (цитаты из базы знаний с путём).
   - `COUNTER` (контраргументы с URL).
3. Добавить элементы из секций "Дебаты команды" и "Примеры и кейсы".
4. Использовать чекбоксы `- [ ]` для каждого элемента.

Правила:

- Ничего не фильтровать на этапе извлечения.
- Каждый claim без источника помечать `MISSING_SOURCE`.
- Каждому элементу дать уникальный ID: `T{n}-E{m}`.

Сохранить в `temp-папка/00-must-include.md`.

## ЭТАП 1: Reverse-engineering целевого издания

Цель: не просто "стиль похож", а воспроизводимый `tone + structure + virality` профиль.

### Шаг 1.1: Собрать top/most-read корпус

Обязательно использовать веб-поиск и страницу издания:

- `web.search_query` по домену издания: `most read`, `popular`, `trending`, `best`, `top`.
- `web.open` главной страницы и разделов `Popular/Trending/Most Read`.
- Дополнительные запросы по теме статьи внутри домена.

Собрать минимум 8 материалов:

- 5+ из явных сигналов популярности (Most Read/Trending/Popular).
- До 3 из смежных high-performance материалов (если явных сигналов мало).

Для каждого материала зафиксировать:

- URL, заголовок, автор, дата.
- Тип сигнала (`popular_list`, `trending_badge`, `editor_pick`, `high_engagement_proxy`).
- Тему/угол.

Сохранить в `01-publication-signal-pack.md`.

### Шаг 1.2: Выделить voice fingerprint и best practices

На основе корпуса построить fingerprint:

- Структура:
  - тип заголовков;
  - средняя длина абзаца;
  - частота подзаголовков;
  - формат hook и финала;
  - доля списков и how-to блоков.
- Тон:
  - формальность (1-10);
  - первое лицо / обращение к читателю;
  - степень экспертности vs разговорности;
  - наличие самоиронии/иронии.
- Контент:
  - плотность цифр, цитат, кейсов;
  - как подаются контраргументы;
  - как формулируются actionable выводы.

Дополнительно выделить признаки "цитируемых/вирусных" материалов:

- какой конфликт открывает текст;
- какой формат promise в заголовке;
- какой тип инсайта чаще всего репостят;
- что повышает quotability (короткие формулы, парадоксы, четкие тезисы).

Собрать:

- `Do` список (10-20 правил);
- `Don't` список (антипаттерны);
- `Top-5` приёмов, которые нужно воспроизвести.

Сохранить в `02-style-fingerprint.md`.
Для структуры использовать шаблон: `references/style-fingerprint-template.md`.

### Шаг 1.3: Зафиксировать anti-plagiarism guardrail

- Не копировать исходные формулировки длиннее 8-10 слов подряд.
- Переносить паттерны, а не фразы.
- Все прямые цитаты сохранять как цитаты с источником.

## ЭТАП 2: Подготовить narrative и материалы под сегменты/JTBD

### Шаг 2.1: Прочитать контекст аудитории

Прочитать `bookconcept.md` и секции тезисов про сегменты/jobs.

Зафиксировать в `04-narrative-plan.md`:

- Целевые сегменты.
- JTBD каждого сегмента.
- Главный конфликт статьи.
- Какой behavior change ожидается от читателя после чтения.

### Шаг 2.2: Обновить кейсы

Для каждого тезиса подобрать минимум:

- 1 основной кейс (поддерживает тезис);
- 1 контркейс/ограничение (показывает границы применимости).

Требования:

- Реальный URL обязателен.
- Дата/контекст обязателен.
- Кейс должен быть связан с конкретным сегментом и JTBD.

Сохранить в `03-case-pack.md`.

### Шаг 2.3: Спроектировать арку статьи

В `04-narrative-plan.md` описать:

- 3 варианта hook (`shock`, `story`, `tension`).
- 2 варианта порядка секций (без потери тезисов).
- Переходы между секциями.
- Word budget по секциям.
- Формат финала (`callback`, `call to action`, `provocative close`).

## ЭТАП 3: Собрать Section Briefs

Цель: сделать из тезисов и исследований исполнимый blueprint для текста.

Для каждой секции создать бриф в `05-section-briefs.md`:

- `Section` / `Тезис` / `Word budget` / `Позиция в статье`.
- Полный блок `СУТЬ` (как каркас).
- Список MUST-INCLUDE элементов по ID.
- Кейс + контркейс с URL.
- Primary segment + JTBD.
- Переход от предыдущей и к следующей секции.
- Style constraints из `02-style-fingerprint.md`.

После сборки посчитать покрытие:

- Для глубоких тезисов: в брифы попадает минимум 60% элементов.
- Для средних: минимум 40%.
- Всё исключённое перечислить в `Dropped` с причиной.

## ЭТАП 4: Написать черновик (3 прохода)

Собирать текст в `06-draft-v1.md`.

### Проход A: Skeleton Preservation

- Взять `СУТЬ` каждого тезиса как старт секции.
- Сохранить причинно-следственную логику тезиса.
- Не заменять аргумент новой логикой.

### Проход B: Publication Voice Transposition

- Применить `tone + structure + best practices` из `02-style-fingerprint.md`.
- Вплести DATA/QUOTE/CONCEPT/COUNTER/кейсы по брифу.
- Настроить ритм абзацев и подзаголовков под паттерн издания.

### Проход C: Humanization Pass

Пройти чеклист из `references/humanity-pass-checklist.md`:

- Убрать шаблонные AI-обороты.
- Добавить конкретику (имена, числа, контекст решений).
- Выровнять "человеческий" ритм: сочетать короткие и длинные фразы.
- Добавить точки интеллектуального трения (контраргумент, ограничение, trade-off).
- Проверить, что текст звучит как автор, а не как нейтральный "обзор".

## ЭТАП 5: Quality Gates (обязательные)

Сохранить отчёт в `07-quality-report.md`.

Оценить по шкалам:

1. `Must-Include Coverage` = включено / всего из section-briefs.
   - Порог: `>= 95%`.
2. `Thesis Integrity` = узнаваемость логики каждой "Сути".
   - Порог: `100%` тезисов узнаваемы.
3. `Publication Style Match` (0-100).
   - Порог: `>= 80`.
4. `Segment/JTBD Fit` (0-100).
   - Порог: `>= 80`.
5. `Humanity Risk` (0-100, меньше лучше).
   - Порог: `<= 20`.
6. `Source Completeness`.
   - Порог: `100%` у claims с внешними фактами.

Если хотя бы один порог не пройден — сделать точечный rewrite и повторить gate.

## ЭТАП 6: Выпустить итоговый draft-файл

Создать/обновить финальный файл:

```bash
"$DRAFT_DIR/Draft-{Название-статьи}.md"
```

Рекомендуемый формат:

```markdown
---
## 📝 Инструкция автора для итерации

> *[Заполнить комментарии для следующего прогона draft-этапа]*

---

# {Название статьи}

> **Статус:** Черновик
> **Дата:** {YYYY-MM-DD HH:MM}
> **Целевое издание:** {название}
> **Язык:** {язык}
> **Слов:** ~{N}
> **Тезисов:** {N}
> **Must-include покрытие:** {N}/{M} ({%})
> **Publication Style Match:** {score}/100
> **Humanity Risk:** {score}/100

[Hook]

## {Секция 1}
...

## {Секция N}
...

## How-to / Actionable Takeaways
...

## Итог
...

## Источники
- [Название](URL)
- ...
```

Дополнительно выдать пользователю:

- 3 альтернативы заголовка в стиле издания.
- 2 альтернативы hook.
- Краткий changelog: что взято из тезисов, что добавлено из внешнего корпуса.

## ITERATION режим (И1-И3)

### И1: Разобрать фидбэк

Классифицировать каждый пункт:

- `Light`: переформулировать, сократить, переставить, уточнить стиль.
- `Deep`: добавить новый кейс, усилить аргумент, проверить актуальность, поменять угол для сегмента.

### И2: Выбрать стратегию

- Только `Light` → править сразу в draft.
- Есть `Deep` → точечно дозапросить данные:
  - либо из тезисов/temp-файлов;
  - либо через веб-поиск по конкретному вопросу.

Не перезапускать полный цикл без необходимости.

### И3: Обновить draft in-place

- Обновить текст и метаданные (дата, версия/итерация).
- Очистить поле "Инструкция автора для итерации" до placeholder.
- Зафиксировать изменения в `07-quality-report.md`.

## Чеклист завершения

- [ ] Все тезисы получили секции (или явно отражены в структуре).
- [ ] Логика "Сути" каждого тезиса узнаваема.
- [ ] `Most-read`/`popular` корпус издания реально собран и проанализирован.
- [ ] Style fingerprint и best practices зафиксированы.
- [ ] Показаны признаки вирусности/цитируемости и применены в тексте.
- [ ] Для каждого claim есть источник (URL или путь в KB).
- [ ] Must-include coverage >= 95%.
- [ ] Segment/JTBD fit >= 80.
- [ ] Humanity risk <= 20.
- [ ] Финальный файл создан в `$DRAFT_DIR`.

## Принцип работы

`Тезисы -> signal pack издания -> style fingerprint -> section briefs -> 3-pass writing -> quality gates -> финальный draft`.

Если цель не пройдена по quality gates, этап написания не считается завершённым.
