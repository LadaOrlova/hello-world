# Критерии оценки экзамена

## Часть 1: Теория (52 балла)

### Автоматическая проверка (закрытые вопросы)

| Тип | Вопросы | Баллов/вопрос | Всего | Проверка |
|-----|---------|---------------|-------|----------|
| Multiple choice | 2, 5, 9, 12, 15, 23 | 1.5 | 9 | Авто |
| Multiple select | 3, 7, 11, 18, 20, 25 | 1.5 | 9 | Авто |
| **Итого авто** | 12 вопросов | | **18** | |

**Правила оценки Multiple select:**
- Все верные ответы + ни одного лишнего = 1.5 балла
- 60%+ верных без ошибок = 0.75 балла
- Остальное = 0 баллов

### Ручная проверка (первичная — AI, финальная — преподаватель)

| Тип | Вопросы | Баллов/вопрос | Всего |
|-----|---------|---------------|-------|
| Короткий открытый ответ | 4, 8, 16, 21, 24 | 3 | 15 |
| Мини-кейс | 1, 6, 10, 13, 17, 19, 22 | 2.5 | 17.5 |
| "Найди ошибку" | 14 | 1.5 | 1.5 |
| **Итого ручная** | 13 вопросов | | **34** |

### Критерии оценки открытых вопросов

#### Короткий открытый ответ (3 балла)

| Баллы | Критерий |
|-------|----------|
| 3 | Полный, точный ответ. Использует правильную терминологию. Все ключевые элементы присутствуют. |
| 2 | В целом верно, но упущен 1 элемент или есть неточность в терминологии. |
| 1 | Частично верно. Понимание есть, но существенные пробелы. |
| 0 | Неверно или не отвечает на вопрос. |

#### Мини-кейс (2.5 балла)

| Баллы | Критерий |
|-------|----------|
| 2.5 | Правильно применяет фреймворк к ситуации. Логика ясна. Конкретные, обоснованные решения. |
| 1.5–2 | Фреймворк применён верно, но решения недостаточно конкретны или обоснованы. |
| 0.5–1 | Есть понимание фреймворка, но применение к ситуации ошибочное или поверхностное. |
| 0 | Фреймворк не применён или применён неверно. |

#### "Найди ошибку" (1.5 балла)

| Баллы | Критерий |
|-------|----------|
| 1.5 | Ошибка найдена и правильно объяснена. Предложено корректное исправление. |
| 1 | Ошибка найдена, но объяснение неполное или исправление неточное. |
| 0.5 | Замечена часть ошибки, но не главная суть. |
| 0 | Ошибка не найдена или указана несуществующая ошибка. |

## Часть 2: Практика — AI-кейс (50 баллов)

### Проверка: AI первичная + ручная преподавателем

Порядок: AI-модель оценивает по рубрике → преподаватель проверяет и корректирует.

### Блок 1: Граф работ ключевого сегмента (12 баллов)

| Критерий | Баллы | Описание |
|----------|-------|----------|
| Выбор сегмента обоснован | 2 | Сегмент выбран логично для задачи роста чека, обоснование связано с данными кейса |
| Core Job сформулирован верно | 3 | Начинается с глагола, конкретный, соответствует сегменту, не абстракция |
| Big Job определён | 2 | Есть связь «чтобы что?», логичный уровень выше |
| Small Jobs (3–5 штук) | 3 | Конкретные шаги/подзадачи, входят в критическую цепочку |
| Связи между уровнями | 2 | Граф логически связан, нет разрывов, видна цепочка |

**Типичные ошибки (снижение баллов):**
- Демографическое описание вместо контекста (-1)
- Абстрактная формулировка работы: «быть счастливым», «жить хорошо» (-2)
- Перепутаны уровни: Big Job на месте Core Job (-1)
- Small Jobs не связаны с Core Job (-1)

### Блок 2: ABCDX-сегментация (10 баллов)

| Критерий | Баллы | Описание |
|----------|-------|----------|
| A/B определены верно | 3 | Соответствуют данным кейса (маржа + удовлетворённость) |
| C/D определены верно | 2 | Обоснование, почему они не целевые |
| Реальные критерии | 3 | Использованы причинные, а не критерии-следствия |
| Связь с ростом чека | 2 | Объяснено, как фокус на A/B помогает задаче |

**Типичные ошибки:**
- Используются критерии-следствия («чек > 2000», «5+ заказов») вместо причинных (-2)
- Нет обоснования, почему C/D не целевые (-1)
- Не связано с бизнес-задачей роста чека (-1)

### Блок 3: Механики создания ценности (12 баллов)

| Критерий | Баллы | Описание |
|----------|-------|----------|
| Количество (3–5 механик) | 2 | Минимум 3 механики, каждая названа |
| Механики из AJTBD | 3 | Используются реальные механики из методологии |
| Конкретное решение | 4 | Для каждой механики — конкретное, реализуемое решение |
| Связь с ростом чека | 3 | Объяснено, почему каждая механика увеличит средний чек |

**Типичные ошибки:**
- Общие фразы без конкретики: «улучшить UX», «добавить фичи» (-2)
- Механики не из AJTBD, а из общего маркетинга (-1 за каждую)
- Нет связи с ростом чека (-1)
- Решения нереализуемы или не учитывают контекст кейса (-1)

### Блок 4: RAT — гипотеза для проверки (8 баллов)

| Критерий | Баллы | Описание |
|----------|-------|----------|
| 3 рискованных предположения | 2 | Конкретные, а не абстрактные |
| Обоснование выбора первого | 2 | Использована формула RAT (риск × стоимость / проверка) |
| Минимальный эксперимент | 3 | Конкретный, быстрый, дешёвый способ проверки |
| Критерий успеха/провала | 1 | Понятно, что считается подтверждением, а что опровержением |

**Типичные ошибки:**
- Абстрактные предположения: «рынок растёт», «клиенты купят» (-1)
- Эксперимент = «запустить продукт» (это не MVP, это продукт) (-2)
- Нет критерия успеха/провала (-1)

### Блок 5: Коммуникация — one-liner (8 баллов)

| Критерий | Баллы | Описание |
|----------|-------|----------|
| One-liner по формуле | 3 | [Что] + [Core Job] + [Ценность] — все три элемента присутствуют |
| Критика текущего лендинга | 2 | Конкретно указано, что не работает и почему (через призму AJTBD) |
| Предложения по лендингу | 3 | 2–3 конкретных изменения, связанных с целевым сегментом |

**Типичные ошибки:**
- One-liner без работы (Job): «Лучший сервис доставки» (-2)
- One-liner с абстрактной ценностью: «быстро и вкусно» (-1)
- Критика лендинга без привязки к AJTBD: «дизайн плохой» (-1)
- Предложения не связаны с целевым сегментом (-1)
