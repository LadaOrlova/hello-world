# Teen Boost Curriculum

> **Vibe Coding Intensive for Teens (13-17)**
> Build real apps with AI. No coding experience needed. Ship your own project in one week.

**Format:** 7 sessions over one week (1 self-paced setup + 5 workshops + demo day)
**Total contact hours:** ~14 hours (Setup: self-paced | Workshops 1-5: 2.5 hours each | Demo Day: 1.5 hours)
**Group size:** Up to 30 teens per cohort, 1 mentor per 6 teens
**Tools:** VS Code, Claude Code (Sonnet for daily work, Opus for complex tasks), Chrome Extension, Wispr Flow
**Outcome:** Every teen ships a working app to their own server and presents it on Demo Day

---

## Workshop 0: Setup & Dream

**Type:** Self-paced pre-work (complete before Workshop 1)
**Duration:** 45-90 minutes at your own pace
**Model:** Sonnet (for testing that everything works)

### Goal

Get your tools installed and pick a project that makes you excited to wake up early.

### Part 1: Install Your Dev Kit

Follow the setup guides (links provided in the student channel). Install everything in this order:

1. **VS Code** -- your coding headquarters
   - Download from https://code.visualstudio.com
   - Install it, open it, make sure it runs

2. **Claude Code** -- your AI coding partner
   - Open the VS Code terminal (Ctrl+` or Cmd+`)
   - Follow the Claude Code installation guide
   - Test it: type a question and see Claude respond

3. **Chrome Extension** -- lets Claude see and interact with websites
   - Install from Chrome Web Store (link in guide)
   - Pin it to your toolbar

4. **Wispr Flow** -- talk to your computer instead of typing
   - Download and install
   - Test: press the hotkey, say something, watch it type

5. **VPN** (if needed based on your region)
   - Follow the VPN guide if required

**Checkpoint:** Take a screenshot of VS Code with Claude Code running and post it in the group chat. If anything fails, ask for help in the chat -- mentors are there to help.

### Part 2: Pick Your Dream Project

This is the most important step. You are going to build a REAL app that works on the internet. Not a homework assignment. Not a toy. A real thing that real people can use.

**The rules:**
- It must be something YOU are excited about
- It should be a web app (runs in a browser)
- Keep it focused: one core feature done well beats ten half-baked features
- No prior coding knowledge needed -- Claude Code will write the code for you

### Phase 1: "Claude Code Sees Your Computer" (40 min)

**Instructor introduction (5 min):**
Welcome everyone. Today you're going to talk to an AI and it's going to build things on your computer. Not in some sandbox. Not in a browser. On YOUR actual machine. Let's start with something that will blow your mind.

---

#### Assignment 1: Claude Lists Your Files (5 min)

**The wow moment:** Claude Code can see the files on YOUR computer.

Open Claude Code in VS Code terminal and type:

```
List all files on my Desktop
```

Watch it run a real terminal command and show you your actual files. That's not a chatbot. That's an AI with access to your computer.

**What just happened:** Claude Code doesn't just chat -- it runs real commands on your machine. It can see your files, create new ones, edit code, and more. You're not using a toy. You're using a real developer tool.

> **Mentor note:** Walk around and make sure everyone sees their own files listed. This is the first "whoa" moment -- let it land. If a teen has a messy desktop, that's funny and relatable. Use it.

---

#### Assignment 2: Claude Creates Files (10 min)

**Your turn to command the machine.**

Try these one at a time. After each one, check that the file actually appeared:

**Command 1:** "Create a file called `hello.txt` on my Desktop with the text: I built this with AI"

**Command 2:** "Create a file called `my-ideas.md` and write a list of 5 app ideas that would be cool for teenagers"

**Command 3:** "Create a folder called `teen-boost-project` on my Desktop and inside it create a file called `README.md` with a description of my project: [describe your project]"

**Check:** Open your Desktop/file explorer. Are the files there? Open them. Read what Claude wrote.

> **Mentor note:** Make sure teens verify the files exist outside of the terminal. The point is to connect "I typed words" with "a real file appeared on my computer."

---

#### Assignment 3: Your Own Command (10 min)

Now do something Claude hasn't been told to do. Come up with your own request. Examples:

- "Write a poem about my favorite game and save it to a file"
- "Create a JSON file with stats about my top 5 favorite movies"
- "Make a to-do list for this week in a markdown file"

**The skill you're practicing:** Formulating clear requests. The better you describe what you want, the better the result. This is the #1 skill of vibe coding.

**Share:** Post your most creative command and Claude's response in the group chat.

> **Mentor note:** Celebrate creative commands. If someone asks Claude to write a rap about calculus and it does it, that's a win. Encourage experimentation.

---

#### Assignment 4: Multi-Step Commands (10 min)

Now give Claude a complex task that requires multiple steps:

**Try this:** "Create a project folder called `my-app`. Inside it, create an `index.html` file with a simple webpage that says 'Hello, I'm [your name] and I'm building [your project name]'. Make the background dark, the text neon green, and add some CSS animations."

**Then:** "Open this file in my browser"

**WOW MOMENT:** You just built a webpage by talking. No coding knowledge. No tutorials. You described what you wanted and it appeared.

> **Mentor note:** This is the second big wow moment. Help anyone whose browser didn't open the file. Make sure everyone sees their personalized webpage in a browser before moving on.

---

### Phase 2: Your Project Plan (30 min)

**Instructor transition (3 min):**
OK so now you've seen the power. Claude can create files, write code, build web pages. But before we build your real project, we need a plan. In the real world, developers don't just start coding -- they write down what they're building first. We call this a Product Plan. You're going to create yours by having a conversation with Claude.

---

#### Assignment 5: Claude Interviews You About Your Project (15 min)

**Model:** Sonnet

Copy and paste this prompt into Claude Code (replace the bracketed parts):

```
I'm a teenager and I want to build [YOUR PROJECT IDEA].

Act as a friendly product advisor. Ask me 5-7 questions one at a time to understand exactly what I want to build. After I answer all questions, write a Product Plan document that includes:

1. Project name and one-line description
2. Who is this for (target audience)
3. Core features (the 3-5 main things it does)
4. Nice-to-have features (things to add later)
5. What the main screen looks like (describe the layout)
6. Tech notes (what technologies to use)

Save the plan as `product-plan.md` in my project folder.

Start by asking me the first question.
```

**Have a conversation.** Answer Claude's questions honestly. If you don't know an answer, say "I'm not sure, what do you suggest?" Claude will help you figure it out.

> **Mentor note:** Walk around during this phase. Help teens who are stuck on questions. The most common issue is scope -- help them narrow down to 3-5 core features. If someone wants to build "an app that does everything," help them pick the ONE thing that matters most.

---

#### Assignment 6: Refine Your Plan by Voice (12 min)

**Unlock the superpower: Wispr Flow voice input.**

Instead of typing, press the Wispr Flow hotkey and TALK to Claude:

- "Actually, I think the main color scheme should be..."
- "Can you add a feature where users can..."
- "I changed my mind about the name, call it..."
- "Make the product plan shorter and more focused"

Keep refining until your product plan feels right. Read it out loud. Does it sound like YOUR project? If yes, you're ready.

**Checkpoint:** Your `product-plan.md` should be saved in your project folder. Read through it. Show it to a mentor if you want feedback.

> **Mentor note:** Encourage voice input from this point forward. Many teens will be shy about talking out loud. Normalize it: "Everyone's talking to their computer, it's not weird here, it's the skill."

### Phase 3: Your Landing Page (60 min)

**Instructor transition (2 min):**
Now for the fun part. Before we build the actual app, we're going to build a landing page -- that's a marketing page that tells the world about your project. Think of it like a movie trailer for your app. Why? Because even the best app in the world is useless if nobody knows about it. Plus, this will be the first real webpage you've ever built.

---

#### Assignment 7: Generate Marketing Text (10 min)

**Model:** Sonnet

```
Read my product-plan.md file. Based on this plan, write marketing copy for a landing page. Include:

1. A catchy headline (max 8 words)
2. A subheadline that explains what the app does (1 sentence)
3. 3 feature blocks with emoji icons and short descriptions
4. A "How It Works" section (3 simple steps)
5. A call-to-action button text
6. A short "About the Creator" section (I'm [your name], [your age], I built this because...)

Make the tone fun and energetic. This is made by a teenager, for teenagers. No corporate language.

Save it as `landing-copy.md` in my project folder.
```

**Read the result.** Does it sound like you? Does it make your project sound exciting? If not, iterate.

---

#### Assignment 8: Refine the Text by Voice (10 min)

Use Wispr Flow to refine your marketing copy:

- "Make the headline punchier"
- "The third feature description is boring, make it sound more exciting"
- "Add a section about why I built this -- I was frustrated that..."
- "Change the tone to be more funny / more serious / more hype"

**Keep going until you'd actually want to visit this page.**

---

#### Assignment 9: Generate the Landing Page (20 min)

**Model:** Sonnet

```
Read my landing-copy.md file. Now create a complete, beautiful HTML landing page based on this copy.

Requirements:
- Single HTML file with all CSS inline (no external files needed)
- Modern, clean design that looks professional
- Dark mode with accent colors
- Smooth scroll animations
- Mobile-responsive (looks good on phones too)
- Hero section with the headline and a gradient background

- A footer with "Built with AI at Teen Boost"

Save it as `index.html` in my project folder. Then open it in my browser.
```

**WOW MOMENT:** You just built a real, professional-looking website by describing what you wanted. Open it in your browser. Resize the window to see it work on different screen sizes. Show it to the person next to you.

> **Mentor note:** This is the biggest wow of the day. Give teens 2-3 minutes to just admire their landing pages. Encourage them to show each other. Some pages will look amazing. Celebrate that.

---

#### Assignment 10: Customize by Voice (20 min)

Your landing page is live but it's generic. Make it YOURS. Use Wispr Flow to give Claude instructions:

**Try these (pick what matters to you):**
- "Change the color scheme to [your favorite colors]"
- "Add a section with screenshots (for now use placeholder images)"
- "Make the animations more dramatic"
- "Add a countdown timer that says 'Launching Soon'"
- "Add a dark/light mode toggle"
- "Change the font to something that feels more [playful/techy/elegant]"
- "Add particle effects to the background"
- "Add a section with fake testimonials from beta testers"

**Keep iterating.** The more specific your instructions, the better the result. This is the core loop of vibe coding: describe what you want, see the result, refine, repeat.

> **Mentor note:** Give teens freedom here. If someone wants to add confetti animations, let them. If someone wants a minimalist design, great. The point is ownership -- this should feel like THEIR page, not a template.

### Phase 4: Build Your App (90 min)

#### Assignment 11: Claude Builds Your MVP (30 min)

**Model:** Sonnet

This is the big one. Give Claude your product plan and let it build.

```
Read my product-plan.md. Now build the actual application.

Rules:
- Build it as a web app (HTML/CSS/JavaScript or Python with a web framework)
- Focus ONLY on the core features (the 3-5 main ones from the plan)
- Make it work locally first
- Keep the code in my project folder
- After building, tell me how to run it locally to test
```

**What happens next:** Claude will start writing code. This might take a few minutes. Watch the magic -- Claude is writing files, creating folders, building your app in real time.

When it's done, follow Claude's instructions to run the app locally. Open it in your browser.

**WOW MOMENT:** Your app WORKS. It's not perfect, but it does the thing you described. You just went from idea to working software in minutes.

> **Mentor note:** Some apps will be more complex than others. If Claude gets stuck or produces errors, have the teen type: "There's an error. Fix it." Claude is great at debugging its own code. If it's truly stuck, switch to Opus model for that specific fix. Help teens understand that errors are normal -- even professional developers deal with errors constantly.

---

#### Assignment 12: Test and Iterate (20 min)

Your app works but it's probably rough. Time to test and improve.

**Test like a real user:**
1. Try every feature. Click every button. Type weird stuff in every input field.
2. Write down what's broken or ugly.
3. Tell Claude to fix each issue one by one.

**Try these commands:**
- "The [feature] doesn't work when I [do this]. Fix it."
- "The design looks ugly on mobile. Make it responsive."
- "Add a loading spinner when [action] is happening."
- "The colors don't match my landing page. Use the same color scheme."

**The key insight:** Building software is not "write code once and it's done." It's a loop: build, test, fix, improve, repeat. You're doing the same thing professional developers do, except you're using AI instead of writing code by hand.

> **Mentor note:** Encourage teens to be picky. "Good enough" is not the goal -- they should keep iterating until they're proud of it. But also watch the clock. They need at least 20 min for deployment.

---

#### Assignment 13: Combine Landing + App (15 min)

**Model:** Sonnet

Now merge your marketing landing page with your actual app.

```
I have two things:
1. My landing page (index.html) - the marketing page
2. My app - the actual application

Combine them so that:
- The landing page is the homepage (what you see first)
- There's a button on the landing page that takes you to the app
- The app has a "back to home" link
- Both share the same design style (colors, fonts)
- Everything is in one project folder, organized cleanly
```

**Test:** Make sure both pages work and the navigation between them is smooth.

---

#### Assignment 14: Deploy to Your Server (25 min)

**Model:** Sonnet

Here it comes. You're about to put your app on the internet.

Deploy my entire project to my server.

Server details:

1. Copy all project files to the server
2. Install any needed dependencies on the server
3. Set up a web server (nginx) to serve my app

**THE MOMENT:** Open your browser. Type in your server IP. See your app. On the internet. Accessible by anyone with the link.

**WOW MOMENT:** You are now a developer. You have a live app on the internet. Share the URL in the group chat. Visit each other's apps.

> **Mentor note:** This is the emotional peak of the entire intensive so far. When a teen sees their app live, celebrate it. Have everyone in the chat share their URLs. This is the developer workflow. You just experienced it for the first time. From now on, you can change anything about your app and push it live in seconds.

### Context Setting (10 min)

**Instructor:**
You built an app. It's live. That's awesome. But here's a question: how do you know if your app is actually any good? How do you know what features to add next? How do you know if anyone else has built something similar?

The answer is DATA. Today you're going to learn the difference between "asking AI to guess" and "using scripts to calculate the real answer." This is a superpower that most adults don't have.

---

### Phase 6: AI Guessing vs. Real Data (40 min)

#### The Setup: Gaming Community Dataset

**Instructor provides a dataset:** A CSV file with data from a fictional teen gaming community. It includes:
- 500 members with usernames, ages, favorite games, hours played per week
- Monthly activity scores, messages sent, achievements unlocked
- Which gaming platform they use (PC, console, mobile)
- Their content creation stats (streams, videos, posts)

**Mentor distributes the file:** Post the CSV link in chat. Teens download it to their project folder.

---

#### Assignment: Ask AI, Then Verify with Code (15 min)

**Step 1 -- Ask Claude to GUESS (without the data):**

```
Without looking at any data file, guess: what percentage of teen gamers prefer mobile gaming over PC gaming?
```

Write down Claude's guess.

**Step 2 -- Now calculate the REAL answer:**

```
Read the file gaming-community.csv. Write a Python script that calculates the exact percentage of members who play on mobile vs. PC vs. console. Show me the real numbers.
```

**Key learning moment:** Compare Claude's guess vs. the actual data. Are they the same? Almost never. This is why data matters. AI can hallucinate statistics. Scripts calculate the truth.

> **Mentor note:** This is the most important conceptual lesson of the entire intensive. Make sure every teen understands: AI GUESSES, scripts CALCULATE. When you need real answers, you write a script. When you need creative ideas, you use AI. Different tools for different jobs.

---

#### Assignment: Find the Interesting Story (15 min)

```
Analyze the gaming-community.csv file. Write Python scripts to answer these questions:

1. What's the most popular game in the community?
2. Is there a correlation between hours played and number of achievements?
3. Which age group is most active in content creation?
4. What's the average hours played per week, broken down by platform?

For each answer, create a simple chart (bar chart or scatter plot) and save it as a PNG file. Use matplotlib.

After the analysis, write a 3-sentence summary of the most surprising finding.
```

**Show your charts.** Which finding surprised you the most? Share in the group chat.

> **Mentor note:** Help teens interpret the charts. "What does this bar chart tell you? Why is that interesting?" Push them beyond "it made a chart" to "I understand what the chart means."

---

### Phase 7: Analyze a Real Dataset (40 min)

#### Assignment: Deep Dive Analysis (20 min)

**Model:** Sonnet (switch to Opus if the analysis needs complex reasoning)

Now go deeper. Pick ONE interesting question from the dataset and investigate it:

```
I noticed that [interesting finding from before]. Dig deeper into this.

Write Python scripts to:
1. Break down the data by multiple variables
2. Find patterns or outliers
3. Create at least 2 visualizations that tell a story
4. Write a short "Data Report" (5-10 sentences) explaining what you found

Save the report as `data-report.md` and the charts as PNG files in my project folder.
```

---

#### Assignment: Bring It Back to Your Project (20 min)

Now connect this to YOUR app:

```
I'm building [your project]. Based on what we learned from the gaming community data, what metrics should MY app track?

Create a simple analytics dashboard concept for my app that tracks:
- How many people visit
- What features they use most
- Where they drop off

Write a Python script that generates fake sample data for my app and creates charts showing what the analytics dashboard would look like.

Save everything in a folder called `analytics/` in my project.
```

> **Mentor note:** This bridges the abstract data lesson with their personal project. Help teens think about what "success metrics" mean for their specific app.

---

### Phase 8: Research for YOUR Project (40 min)

#### Assignment: Competitive Research (20 min)

**Model:** Opus (complex research reasoning)

```
I'm building [your project - describe it in 2 sentences].

Act as a market researcher. Do the following:

1. List 5 similar apps or websites that already exist
2. For each competitor, describe: what they do well, what they do badly, and how my app is different
3. Identify 3 features my competitors have that I should consider adding
4. Identify 1 thing NONE of them do that could be my unique advantage
5. Write a "Competitive Analysis" document

Save it as `competitive-analysis.md` in my project folder.
```

**Read the analysis.** Does it change how you think about your project? Highlight the most useful insight.

---

#### Assignment: User Research (20 min)

```
Based on my competitive analysis and product plan, create:

1. A user survey (10 questions) I could send to potential users of my app
2. A list of 5 places online where my target users hang out (subreddits, Discord servers, TikTok communities)
3. A "User Persona" - a fictional description of my ideal user (name, age, habits, what frustrates them, how my app helps)

Save it as `user-research.md` in my project folder.
```

> **Mentor note:** This is where teens start thinking like product people, not just coders. Encourage them to actually think about who would use their app.

---

### Wrap-up (10 min)

**Instructor:**
Three things to remember from today:
1. **AI guesses. Scripts calculate.** When you need real answers, write a script.
2. **Data tells stories.** A chart can reveal something nobody expected.
3. **Know your competition.** Before you build, know what already exists.

Tomorrow is going to be wild. You're going to build your own AI army -- custom AI commands that do exactly what you tell them. See you then.

### Introduction (10 min)

**Instructor:**
OK so you've built an app, deployed it, and analyzed data. That's what a developer does. Today you're going to do something developers JUST started doing this year: building AI armies.

Here's the difference:
- **Skill** = a reusable command you teach Claude. Like a shortcut, but way more powerful.
- **Agent** = Claude with a specific role and personality. It doesn't just follow instructions -- it thinks like a specific type of person.
- **Pipeline** = a chain of steps that runs automatically. Input goes in one end, output comes out the other, no human needed.

By the end of today, you'll have all three.

### Phase 9: CLAUDE.md & Skills (40 min)

#### Assignment: Create Your CLAUDE.md (15 min)

**What is CLAUDE.md?** It's an instruction manual for Claude. When you put a `CLAUDE.md` file in your project folder, Claude reads it every time it starts and follows those instructions. It's like programming Claude's personality and knowledge.

```
Create a CLAUDE.md file in my project folder with the following sections:

# Project Context
- This project is [your app name] - [one-line description]
- It's built for [target audience]
- The tech stack is [whatever Claude used to build it]

# Coding Style
- Always use clear variable names (no single letters)
- Add comments explaining what the code does
- Keep functions short and focused
- Use dark mode color schemes by default

# My Preferences
- I'm [your name], [your age]
- I prefer [casual/professional] language
- When I ask for help, explain things like I'm a teenager, not a senior developer
- Always suggest the simplest solution first

# Rules
- Never delete existing files without asking me first
- Always test code before saying it's done
- If something breaks, fix it instead of starting over
```

**Customize it.** Add your own rules. What do you want Claude to always remember about you and your project?

> **Mentor note:** CLAUDE.md is powerful because it persists between sessions. Help teens understand that this is how you "train" Claude to work the way YOU want.

## Appendix A: Instructor Cheat Sheet

### Model Selection Guide

| Situation | Model | Why |
|-----------|-------|-----|
| Daily coding, file creation, quick tasks | **Sonnet** | Fast, cheap, good enough for 90% of tasks |
| Debugging complex issues | **Opus** | Better at reasoning through multi-step problems |
| Agent teams and orchestration | **Opus** | Needs to manage multiple perspectives |
| Competitive analysis and research | **Opus** | Better at synthesis and nuanced reasoning |
| Landing page generation | **Sonnet** | Straightforward code generation |
| Data analysis scripts | **Sonnet** | Python scripts are well within Sonnet's ability |
| Expert Arena debates | **Opus** | Needs to simulate multiple distinct viewpoints |

### Common Problems & Solutions

| Problem | Solution |
|---------|----------|
| Claude Code won't install | Check Node.js version, try reinstalling npm |
| SSH connection fails | Check IP, password copy-paste, VPN interference |
| App doesn't work after deploy | Check nginx config, port conflicts, file permissions |
| API key not working | Verify the key hasn't expired, check rate limits |
| Teen is stuck on project choice | Help them narrow down: "What app do YOU use every day? What annoys you about it? Build a better version." |
| Teen is overwhelmed | Focus on ONE feature. "Forget the rest. What's the ONE thing your app absolutely must do? Let's build that." |
| Teen finishes early | Challenge them: add animations, dark/light mode toggle, user accounts, share feature |
| Teen is behind | Pair them with a mentor for 1-on-1 help. Simplify their project scope. |

### Mentors: Do's and Don'ts

| Do | Don't |
|----|-------|
| Celebrate small wins loudly | Dismiss anyone's project as "too simple" |
| Let teens struggle for 2-3 min before helping | Jump in immediately when something breaks |
| Ask "what have you tried?" before giving the answer | Just fix it for them |
| Encourage voice input (Wispr Flow) | Let them only type (voice is a key skill) |
| Share your own "this is cool" reactions | Be robotic or overly formal |
| Help them debug by asking Claude for help | Write code for them manually |
| Reference real-world developer practices | Make it feel like school |
| Use their names | Call them "students" |
