# Метрики оценки качества AI pipeline (AURA product)

## Контекст

Мы строим AI-продукт с pipeline агентов, который генерирует сегменты клиентов, графы работ, критические цепочки и бизнес-модели. Чтобы улучшать качество и прикрутить autoresearch-подход (Карпати), нужны чёткие метрики на каждом этапе.

## Архитектура оценки

### Три слоя оценки

| Слой | Что проверяет | Стоимость | Скорость |
|------|--------------|-----------|----------|
| **Структурные проверки** | Формат, связность, полнота | Бесплатно | Мгновенно |
| **LLM-as-Judge** | Смысловая корректность по рубрике | ~$0.01-0.10 за оценку | Секунды |
| **Экспертная оценка** | Реальная бизнес-применимость | Дорого (время Ивана) | Минуты-часы |

Нужны все три:
- **Структурные** — для быстрой фильтрации мусора
- **LLM-as-Judge** — для autoresearch-цикла (автоматическая оптимизация промптов)
- **Экспертная** — для калибровки LLM-as-Judge

### Необходимость Gold Standard

Взять 3-5 реальных продуктов, которые Иван хорошо знает. Для каждого вручную создать эталонные:
- Core jobs
- Сегменты
- Граф работ

Это "test set". Без него LLM-as-Judge оценивает в вакууме.

## Этап 1: Генерация Core Jobs

### Вход
Описание продукта, рынка, ролей пользователей

### Выход
Набор core jobs для каждой роли

### Метрики (рубрика для LLM-as-Judge)

| Метрика | Вес | Что оценивает |
|---------|-----|---------------|
| **Формулировка** | 0.15 | Каждая job начинается с глагола в инфинитиве, описывает результат, не процесс |
| **Валидность core** | 0.25 | Это действительно то, что продукт ПОЛНОСТЬЮ выполняет, а не related/adjacent job |
| **Полнота (recall)** | 0.20 | Все основные core jobs покрыты, ничего важного не упущено |
| **Не-дублирование (precision)** | 0.10 | Jobs не пересекаются, каждая уникальна |
| **Специфичность** | 0.10 | Достаточно конкретны для принятия решений, не "абстрактные хотелки" |
| **Ролевое покрытие** | 0.10 | (B2B) Покрыты все роли: LPR, LBR, end user, бизнес |
| **Реальность** | 0.10 | Не "ложные jobs" — люди реально тратят деньги/время/усилия |

**Композитная метрика:** `core_jobs_score = Σ(вес_i × метрика_i)` → число 0-1

### Структурные проверки (автоматические, бесплатные)

- Каждая job содержит глагол в инфинитиве
- Количество core jobs в разумном диапазоне (3-15 на роль)
- Нет двух jobs с cosine similarity > 0.9 (проверка на дубли)
- Каждая job имеет описание контекста и критерии успеха
- Нет jobs-фантомов без конкретного результата

## Этап 2: Формирование сегментов

### Вход
Core jobs из этапа 1 + описание рынка

### Выход
Набор сегментов: характеристики людей/компаний + набор core jobs + success criteria

### Метрики (рубрика для LLM-as-Judge)

| Метрика | Вес | Что оценивает |
|---------|-----|---------------|
| **Job-based определение** | 0.25 | Сегмент определён через набор jobs, а не демографию |
| **Различимость** | 0.20 | Сегменты явно отличаются друг от друга (разные job graphs) |
| **Идентифицируемость** | 0.15 | Можно найти этих людей в реальном мире (описаны наблюдаемые характеристики) |
| **Действенность** | 0.15 | Можно построить продукт/маркетинг конкретно под этот сегмент |
| **Полнота** | 0.10 | Сегменты покрывают рынок без крупных белых пятен |
| **Внутренняя однородность** | 0.15 | Участники сегмента действительно похожи по job graphs |

**Композитная метрика:** `segments_score = Σ(вес_i × метрика_i)` → число 0-1

### Структурные проверки

- Каждый сегмент содержит: характеристики людей/компаний + набор core jobs + success criteria
- Количество сегментов 3-8 (меньше — слишком грубо, больше — неуправляемо)
- Каждый сегмент привязан к core jobs из этапа 1
- Нет сегмента "все остальные" (lazy catch-all)

---

## Этап 3: Генерация графов работ

### Вход
Сегмент из этапа 2 + роль внутри сегмента

### Выход
Иерархический граф работ: Big Job → Core Job → Small Jobs + критические цепочки

### Метрики (рубрика для LLM-as-Judge)

| Метрика | Вес | Что оценивает |
|---------|-----|---------------|
| **Иерархическая корректность** | 0.20 | Big Job → Core Job → Small Jobs, каждый уровень отвечает на "чтобы что?" |
| **Полнота** | 0.20 | Все важные jobs включены, нет пропущенных шагов |
| **Критические цепочки** | 0.20 | Правильно определены must-execute последовательности |
| **Последовательность** | 0.15 | Зависимости и порядок выполнения корректны |
| **Баланс гранулярности** | 0.10 | Не слишком крупно (нет деталей), не слишком мелко (noise) |
| **Действенность** | 0.15 | Граф показывает, где создавать ценность (bottlenecks, rupture points) |

**Композитная метрика:** `graph_score = Σ(вес_i × метрика_i)` → число 0-1

### Структурные проверки

- Граф — связный DAG (directed acyclic graph)
- Каждый узел имеет: название job + уровень + parent
- Нет "сирот" (узлов без связей)
- Есть хотя бы одна отмеченная critical chain
- Глубина графа 3-5 уровней

## Как прикрутить autoresearch-цикл

### Архитектура

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ program.md  │     │ prompts/*.md │     │  eval/*.py  │
│ (правила    │     │ (промпты     │     │ (оценка:    │
│  для агента)│     │  каждого     │     │  структура  │
│             │     │  этапа —     │     │  + LLM-judge│
│ НЕ меняется │     │  МЕНЯЕТСЯ    │     │  + rubrics) │
│ человеком   │     │  АГЕНТОМ)    │     │             │
└─────────────┘     └──────────────┘     │ НЕ меняется │
                                         └─────────────┘
```

### Цикл (для каждого этапа отдельно)

1. Агент читает program.md (правила оптимизации)
2. Модифицирует prompt этапа
3. Прогоняет на 3-5 тестовых продуктах (gold standard)
4. Eval считает score (структурные проверки + LLM-as-Judge)
5. Score улучшился → keep, иначе → discard
6. Логирует в results.tsv
7. Повтор бесконечно

### Борьба с шумом LLM-as-Judge

В отличие от val_bpb у Карпати, LLM-as-Judge вносит шум. Компенсация:
- Прогонять на нескольких test cases и усреднять
- Запускать LLM-as-Judge 3 раза и брать медиану
- Использовать структурные проверки как "быстрый фильтр" до дорогого LLM-judge
- Периодически калибровать LLM-judge экспертной оценкой Ивана

## Что критично знать про AI pipeline продукты

1. **Evals > всё.** Команда, которая пишет лучшие evals, выигрывает. Не промпты. Не архитектура. Evals.

2. **Начните с end-to-end, потом декомпозируйте.** Сначала один гигантский промпт "сгенерируй всё". Посмотрите что выходит. Потом разбейте на этапы и оптимизируйте каждый.

3. **LLM-as-Judge работает.** При хороших рубриках корреляция с экспертной оценкой 80%+. Достаточно для autoresearch-цикла.

4. **Gold standard — инвестиция.** Иван должен лично создать 5 эталонных разборов продуктов. Это 2-3 дня работы. Без этого всё остальное не работает.

5. **Prompt engineering — первый рычаг.** До сложных multi-agent архитектур просто оптимизируйте промпты. 80% результата.

6. **Наблюдаемость (observability).** Логируйте каждый вызов: вход, выход, метрики, версия промпта. Без этого нельзя понять что улучшилось.

7. **Decomposition matters.** Разбиение pipeline на этапы с чёткими входами/выходами делает каждый этап независимо оптимизируемым.

8. **Autoresearch-цикл масштабируется.** Как только есть eval → modify → evaluate → keep/discard, можно запускать автономно и бесконечно.
