### Thesis 1: Ты — оператор конвейера проверки гипотез (ЦЕНТРАЛЬНЫЙ ТЕЗИС)

**Core idea:**
Фундаментальный сдвиг 2025-2026: ты больше не "делаешь продукт" — ты оперируешь конвейер проверки гипотез. AI и vibe coding (Cursor, Replit Agent, v0.dev, Bolt) сделали каждый цикл "идея→прототип→тест" сверхдёшевым. Раньше один цикл стоил $50,000 и занимал месяцы. Сейчас — $500 и дни (Index.dev: 55% ускорение проектов, 3 дня vs недели на прототип). YC W25: 25% codebases AI-generated.

**Ключевое усиление:** "Vibe coding makes throwaway experiments nearly free to BUILD. But the real cost was never the code — it's the LEARNING." Код перестал быть активом — стал расходным материалом (Disposable Code Revolution, 2025: marginal cost of code generation dropped to near-zero). Но дешёвые эксперименты — это не дешёвое обучение. Spotify (520 экспериментов, 58 команд за год): learning rate ~64%, win rate ~12%. Метрика конвейера — не experiments-per-week, а **validated-learnings-per-week** (Learning Velocity = (Experiments × Insights) / Time, Ash Maurya).

Каждая гипотеза проходит через pipeline: Persona → Problem/Job → Value → Demand → Unit Economics (Alexander Cowan). На каждой стадии — минимальный эксперимент: от JTBD-интервью (30 минут) до fake door landing page (1 день) до vibe-coded MVP (1 неделя). Ты не любящий родитель одной идеи — ты оператор производственной линии, где идеи загружаются, проверяются и безжалостно отбраковываются. Dropbox получил 75,000 подписок 3-минутным видео ДО написания кода.

Роль PM/founder необратимо сместилась: от координации разработчиков к формулированию гипотез и операционному управлению конвейером. ACM (2025): "vibe coding imperative for PMs" — через 12-24 месяца AI-native IDEs станут стандартным инструментом продакт-менеджера. Рынок PM переживает K-shaped split: зарплата AI-focused PMs +35%, generalists теряют (Agents Today).

**Status:** 🔄 Усилен в v3 — добавлен build/learn distinction, Disposable Code Revolution, Learning Velocity

### Thesis 3: RAT — операционная система конвейера + Where Are You on the PMF Gradient?

**Core idea:**
Every product decision is a stack of assumptions. RAT (Risk Assumption Test) is the operating system of the hypothesis pipeline. The algorithm: (1) list ALL assumptions, (2) rank by (Probability × Consequences) / Cost of Testing, (3) test the scariest, cheapest-to-test ones FIRST. AI changes the formula: it drops the denominator (cost of testing) dramatically, which means MORE hypotheses bubble up and pipeline throughput explodes.

The Traction Map is the dashboard — sequential cells (market → segment → size → leads → JTBD interviews → unit economics → MVP → first sale → value confirmation → channels → scale), with red markers that stop the line when a defect is found (Jidoka principle). Each green cell on the Traction Map moves you one step on the PMF gradient:

| Traction Map Stage | PMF Gradient | Focus |
|-------------------|--------------|-------|
| Market → Segment → JTBD interviews | **No PMF** — не попали в сегмент | Нахождение сегмента и создание ценности |
| Unit economics → MVP → First sale | **Weak PMF** — редкие немасштабируемые продажи | Создание спроса, тест юнит-экономики |
| Value confirmation → Channels → Scale | **Strong PMF** — повторяемые продажи | Масштабирование, эффективность |
| Demand exceeds capacity | **Being ripped off shelves** | Don't break it |

**Dashboard metric:** Rahul Vohra's PMF Engine — measure PMF weekly using Sean Ellis survey ("very disappointed" ≥ 40%). For AI products: measure AFTER novelty wears off (a16z "retention rebasing": post-novelty retention drops 40-60%).

### Thesis 4: Risks Multiply, Not Add — Why Simpler Products Win

**Core idea:**
The single most powerful mathematical argument for the hypothesis pipeline. Even in a "magical universe" where each assumption has a 40% chance of being confirmed (reality: 10-20%), stacking 7 assumptions gives you: 0.4^7 = 0.16%. In reality, 0.1^7 = 0.0000001. Every additional assumption MULTIPLIES failure probability. Conclusion: strip away assumptions, kill features, remove complexity. This is WHY the pipeline matters — it's a machine for reducing unverified assumptions before you commit.

**Depth:** 🔬 Deep
**Freshness:** 📅 Evergreen (mathematical principle)
**Status:** ✅ Confirmed

**Segment value:** Founders 0→1 (directly actionable), VC/Investors (investment evaluation lens)

**From Knowledge Base (AURA/AJTBD):**
> "Волшебная вселенная: 0,6 в 7 степени = 3%. В самой волшебной вселенной у вас с 7 рисками вероятность успеха 3%. Лучшее, что вы можете сделать для вашего продукта — избавлять его от рисков."
> — Source: Knowledge/RAT/risk-assumption-test.md (lines 158-161)

### Thesis 5: Segment Primacy — Find the WHO Before Building the WHAT

**Core idea:**
The segment (WHO has the job) is more important than the solution (WHAT you build). If you find the right segment with the right job, you can iterate the product indefinitely until it works. If you pick the wrong segment, no amount of iteration will save you. This is the first and most critical input into the hypothesis pipeline — get it wrong, and the whole pipeline processes garbage.

Точное знание сегмента позволяет 10 вещей, которые без него невозможны:
1. **Итерировать всё остальное** — продукт, цены, бизнес-модель, коммуникацию — до успеха (RAT, line 143)
2. **Побеждать универсалов** — конкурировать за ВСЕ работы узкого сегмента (graph-of-jobs.md, line 219)
3. **Создавать точную ценность** — не фичи, а решение конкретных работ (segmentation.md, line 192)
4. **Видеть то, что не видят конкуренты** — полный граф работ = competitive advantage (main-algorithm.md, line 65)
5. **Открывать механику роста** через подсегменты (segmentation.md, lines 63-66)
6. **Выбирать по 10 факторам** — размер, бюджет, частота, боль, конкуренция... (segmentation.md, line 332)
7. **Получать готовую коммуникацию** — AJTBD-знание = тексты лендингов и скрипты продаж (product-strategy.md, lines 216-217)
8. **Оптимизировать скорость конвейера** — выбирать сегменты с быстрым фидбэком (launch-product.md, lines 289-290)
9. **Видеть каналы привлечения** — знание работ → знание выходов на аудиторию (create-acquisition-channel.md)
10. **Создавать устойчивое конкурентное преимущество** — ты знаешь клиента глубже любого конкурента

AI делает segment research драматически проще. Reddit mining, review analysis, competitor scraping с AI-агентами, автоматизированный синтез JTBD-интервью — что занимало недели ручной работы, теперь занимает часы. Но: AI makes research easy; it doesn't make research optional.

### Thesis 6: Дистрибуция — кислород стартапа, без которого задохнёшься

**Core idea:**
В эпоху vibe coding создание продукта перестало быть конкурентным преимуществом. Точка конкуренции сместилась в дистрибуцию. Но НАЗЫВАТЬ дистрибуцию "единственным моатом" — ошибка (v2 → v3 correction).

Дистрибуция — это **кислород**, а не крепость (Simon Wu / Cathay Innovation, $3B+ AUM): без неё ты задохнёшься, но наличие кислорода не делает тебя непобедимым. Distribution — это Bailey (NFX): быстроразвёртываемая позиция для захвата рынка, но НЕ Motte (крепость для удержания). Пример: Groupon имел great distribution but lacked network effects — ultimately failed. Настоящий моат строится из слоёв:

1. **Distribution** (захват рынка) — быстрый старт, но без остальных = Groupon
2. **Workflow Ownership** (Insight Partners, $90B AUM) — "removing you breaks the workflow"
3. **Proprietary Data** (Insight Partners) — уникальные данные, compounding с каждым клиентом
4. **Network Effects** (NFX) — каждый новый пользователь увеличивает ценность
5. **Switching Costs** — персонализация, accumulated context ("I'm a Claude person")

Для стартапа 0→1: дистрибуция — **первый фильтр** при входе идеи в конвейер. Есть ли у тебя distribution advantage? Доступ к аудитории? Экспертиза в канале? Если нет — либо cofounders с distribution advantage, либо другая идея. Thiel: "If you can get just one distribution channel to work, you have a great business."

Практический вывод: тест спроса должен быть частью MVP, а не afterthought. Прежде чем писать код, запусти landing page, outreach — измерь реальный CAC и конверсию.

### Thesis 7: Что дальше — от ручного станка к конвейеру Форда (NEW)

**Core idea:**
Hypothesis pipeline — это не "Lean Startup 2.0". Lean Startup говорил: "учись быстро". Pipeline говорит: "учись В ПРОМЫШЛЕННОМ МАСШТАБЕ". Разница — как между ручным ткацким станком и конвейером Форда. Принцип один (ткань/обучение), но масштаб, скорость и операционная модель — принципиально другие.

|--------|--------------------|-----------------------------|

**Практический next step — как построить конвейер:**

**Шаг 1. Инвентаризация рисков (1 час).** Выпиши ВСЕ рискованные предположения (Knowledge/RAT, line 92: "Вряд ли потратите сильно больше одного часа"). Увидеть все риски — первая и самая важная ценность. Ты бессознательно избегаешь болезненных рисков и прыгаешь в приятное (сделать MVP, написать дизайнеру).

**Шаг 2. Приоритизация.** Отранжируй по RAT: (Probability × Consequences) / Cost of Testing. Alternative: ICE Framework (Impact × Confidence / Effort).

**Шаг 3. Формулирование гипотезы.** Шаблон (Contentsquare): "We believe [solution] will [outcome] for [audience] because [reasoning]."

**Шаг 4. Минимальный эксперимент.** Выбери по стоимости: JTBD-интервью (30 мин) → fake door (1 день) → vibe-coded MVP (1 неделя). Определи success criteria ДО начала (Thoughtworks HDD).

**Шаг 5. Измерь Learning Velocity.** Формула Ash Maurya: LV = (Experiments × Insights) / Time. Velocity must be directed — learning must increase traction, иначе впустую.

**Шаг 6. Decide: persist, pivot, or kill.** Итерации по 6 решенческих интервью (KB: launch-product.md, line 197). Нет продаж за 5 итераций → вернись на предыдущую стадию.

**Шаг 7. Повторяй.** 90-day validation cycles. Не больше 2 сегментов, не больше 3 гипотез канала одновременно (KB: RAT, line 110).

### Тезис 9: Critical Chain — где именно рвётся PMF 🔴

**Суть:** PMF рвётся не потому что продукт плохой, а потому что **цепочка работ клиента рвётся** на каком-то звене. Клиент не доходит до Big Job. Кейс WooHoo (фертильность): 100% → 25% → 6% → 1.5% — конверсии мультиплицируются (та же математика, что T4, но для user journey). Самое ценное место для продукта — там, где больше всего людей «застревают». Стратегии: предыдущая работа = лидогенерация, следующая работа = retention, убить всю цепочку = уничтожение конкурентов (Tilda/Wix убили cPanel).

**Из Knowledge Base:**
> «Критическая цепочка не должна рваться. Если хотя бы одна работа в цепочке не выполнена, весь результат Big Job не достигается.» — `Knowledge/AJTBD/critical-chain.md` (строки 10-11)

**Синтез с T1-T7:** T1 (конвейер) → проверяй не «нравится ли продукт», а «проходит ли клиент всю цепочку». T3 (RAT) → каждое звено = рискованное предположение. T4 → конверсии звеньев мультиплицируются.

**Actionable:** Нарисуй критическую цепочку от потребности до Big Job. Посчитай конверсию между звеньями. Найди звено с максимальным оттоком — это самое ценное место для продукта.

### Тезис 10: Когнитивные ловушки + Decision Hygiene оператора 🔴

**Суть:** Конвейер мощен ровно настолько, насколько честен оператор. 7 когнитивных искажений ломают конвейер: confirmation bias, sunk-cost, optimism bias, planning fallacy. 83% фаундеров переоценивают идеи эмоционально. METR Study (уже в T1): разработчики с AI считали себя на 20% быстрее, но были на 19% медленнее. Decision fatigue — клинически доказанное явление: пик когнитивной функции 90-120 мин после пробуждения. Rand Fishkin (Moz) документировал, как decision fatigue разрушила его ability to make decisions.

**Экспертные мнения:**
- McKinsey (⭐⭐⭐, 2025): Pre-mortems +30% в нахождении проблем — [mckinsey.com](https://www.mckinsey.com/capabilities/strategy-and-corporate-finance/our-insights/bias-busters-premortems-being-smart-at-the-start)
- Brookings (⭐⭐⭐, 2025): Pre-mortem reduces overconfidence + confirmation bias — [brookings.edu](https://www.brookings.edu/articles/the-art-and-science-of-pre-mortems/)
- SAGE Journals (⭐⭐⭐, 2025): Debiasing tools +16% strategic outcomes — [sagepub.com](https://journals.sagepub.com/doi/10.1177/01492063241287188)
- Frontiers in Cognition (⭐⭐⭐, 2025): Decision fatigue = measurable decline — [frontiersin.org](https://www.frontiersin.org/journals/cognition/articles/10.3389/fcogn.2025.1719312/full)
- Psychology Today (⭐⭐, 2025): Peak cognition 90-120 min after waking — [psychologytoday.com](https://www.psychologytoday.com/us/blog/urban-survival/202503/maximizing-decisions-how-high-performers-overcome-decision-fatigue)

**Синтез с T1-T7:** T1 → больше экспериментов = больше возможностей обмануть себя. T2 → "cold-blooded selector" = навык debiasing, не характер. T4 → biased experiment НЕ снижает риск, а УВЕЛИЧИВАЕТ false confidence.

**Actionable:** Pre-mortem на каждую гипотезу ДО теста. Kill/continue решения ТОЛЬКО утром. Batch decisions раз в неделю. Success criteria ДО эксперимента. Anti-confirmation hypothesis параллельно.

**⚠️ Контраргумент:** Over-debiasing → analysis paralysis. Founder intuition (Shreyas Doshi "product sense") — это pattern recognition, не bias.

### Тезис 11: Retention & Churn + AI Retention Illusion 🔴

**Суть:** PMF — не первое «Да», а продолжающееся «Да». Три механики: (1) Aha-момент = точка включения retention — мозг переобучается выполнять работу с продуктом, (2) Hedonic adaptation = тихий убийца — «вчерашний wow = сегодняшний baseline», (3) Next Job = retention через рост ценности. AI-специфичная проблема: «AI tourists» надувают метрики adoption, создавая иллюзию PMF. a16z: retention мерь с M3, не M0. Jason Lemkin: AI-native GRR 40%, NRR 48%. 84% AI churn voluntary. Формула: PMF = M12/M3.

**Из Knowledge Base:**

> «То, что вчера "вау", сегодня базовое качество.» — `Knowledge/Algorithms/decrease-churn.md`

**Экспертные мнения:**

### Тезис 12: Four Forces — навигатор выбора сегмента 🟡

**Суть:** Четыре силы прогресса (добавочная ценность, проблемы с текущим решением, страхи к новому, привычка к старому) = инструмент ВЫБОРА сегмента, не описания клиента. Люди переоценивают свою способность менять привычки в 3 раза. Привычка = физическое изменение мозга. Кейс PitchMe: скопировали интерфейс конкурента + добавили данные → быстрый рост (не боролись с привычкой). Перед проверкой гипотезы ценности оцени четыре силы: если привычка непреодолимая — **смени сегмент**.

**Из Knowledge Base:**
> «Четыре силы = инструмент выбора работ сегмента. Может быть, нафиг этот сегмент?» — `Knowledge/AJTBD/main-idea.md` (строки 80, 177)

**Синтез с T1-T7:** T5 (сегмент) → четыре силы = механика выбора. T4 → привычка = ещё один множитель 0.4^N. T3 → страхи и привычка часто не попадают в RAT (assumption blindness).

**Actionable:** 4 вопроса перед каждой гипотезой: (1) Можем создать ценность? (2) Проблемы с текущим решением? (3) Сила привычки? (4) Страхи снижаемы? Если (3) непреодолима — смени сегмент.

### Тезис 13: ABCDX — навигация после первых продаж 🟡

**Суть:** Что делать, когда первые деньги пришли? Не все деньги одинаково полезны. A-клиенты (маржинальные + удовлетворённые) = PMF signal. C/D-клиенты = PMF noise. Часто 80% времени уходит на C/D, а 80% прибыли приносят A/B. Ключевой вопрос: не «кто платит?», а **«что в человеке такого, что он маржинальный и удовлетворённый?»** — причина, не следствие. Кейс дизайн-студии: A = инвесторы из других городов, высокий бюджет, готовы делегировать.

**Из Knowledge Base:**
> «Проверочный вопрос: "Что в человеке такого, что он удовлетворённый и маржинальный?"» — `Knowledge/ABCDX/abcdx-segmentation.md` (строка 47)

**Синтез с T1-T7:** T1 → конвейер не заканчивается на первой продаже. T3 (PMF Gradient) → ABCDX = точный диагноз «где мы на градиенте». T5 → вторая итерация сегментации.

**Actionable:** После 50-100 продаж: раздели на A/B/C/D, спроси про A «что в нём такого?», перестань слушать C/D, перенастрой маркетинг на A/B.

### Тезис 14: Активирующее знание — продавать непривычное 🟡

**Суть:** Для инновационных/AI продуктов клиент НЕ ЗНАЕТ, что решение возможно. Его Big Job спит. Активирующее знание = механика пробуждения спроса: (1) загрузи знание про лучший способ выполнить Big Job, (2) сними страхи, (3) уволь конкурента. Бьёт через Big Job, не Core Job. Кейс Yango: партнёрство с банками для загрузки знания → конверсия выросла, продукт куплен Совкомбанком. Найди 10-40 «уникалов» (уже наняли непривычное решение) и масштабируй их знание.

**Из Knowledge Base:**
> «Активирующее знание — знание, которое привело к решению выполнять именно такую работу. Бьёт через Big Job, а не Core Job.» — `Knowledge/HowTos/load-activating-knowledge.md` (строки 4, 118)

**Синтез с T1-T7:** T1 → добавь шаг ДО ценности: загрузка знания. T2 → идея мертва не потому что плоха, а потому что нет механики загрузки. T6 → активирующее знание = содержание дистрибуции.

**Actionable:** Найди «уникалов», спроси «что ты узнал, после чего захотел это?» Встрой загрузку в дистрибуцию: landing page, партнёрства, контент.

### Тезис 15: Pricing + Product-Market-PRICE Fit 🟡

**Суть:** Команды тестируют сегмент, проблему, решение — но почти никогда не тестируют цену. ### Тезис 34: Luck Surface Area — удача проектируется 🟢

**Суть:** Pluchino et al. (500+ цитат): удача > таланта для предсказания успеха. Но это не фатализм — это design challenge. Структурированный процесс максимизирует случайные находки.

**Экспертные мнения:**
- Pluchino et al. (⭐⭐⭐, 2018, Nature, 500+ cit.): "Talent vs Luck" — [arXiv](https://arxiv.org/pdf/1802.07068)

**Синтез с T1-T7:** T1 → pipeline = luck amplifier, не prediction machine. T4 → 100 гипотез по 0.16% = ~15% кумулятивный шанс.
