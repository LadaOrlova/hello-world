# Генерация продуктовых требований (PRD)

Скилл превращает идею продукта в полноценные продуктовые требования через последовательный процесс:
1. Сбор идеи и бизнес-контекста
2. Генерация гипотез сегментов и core jobs
3. Генерация и ранжирование рискованных предположений
4. Запуск команды агентов: конкурентный анализ → проектирование PRD → edge-кейсы

---

## ЭТАП 4: Генерация и ранжирование рискованных предположений

Сгенерируй рискованные предположения, используя методологию из `6-Prompts-and-Scripts/Prompts/riskiest-assumption-test.md`.

### Входные данные для генерации
- Идея продукта (ЭТАП 0)
- Бизнес-контекст (ЭТАП 1)
- Выбранные сегменты и jobs (ЭТАП 3)

### Категории риска
- Спрос на рынке
- Экономически привлекательный сегмент
- Готовность сегмента купить (ценностная гипотеза)
- Юнит-экономика (маржинальность)
- Масштабирование спроса
- Операционные / регуляторные / технологические

### Формула ранжирования

```
Score = (P × I) / C
```

Где:
- **P** (1-5) — вероятность наступления риска (5 = чистая гипотеза, 1 = есть эмпирика)
- **I** (1-5) — стоимость последствий, если риск срабатывает (5 = убьёт бизнес, 1 = локальный сбой)
- **C** (1-5) — стоимость проверки (5 = очень дорого, 1 = проверить почти бесплатно)

Чем выше Score — тем приоритетнее проверка.

### Формат карточки риска

```markdown
### Риск N: [Название]

**Предположение:** [конкретная проверяемая гипотеза с цифрами/условиями]

**Риск:** [почему это критично для бизнеса]

**Категория:** [одна из 6 категорий]

**P (вероятность):** N/5 — [обоснование]
**I (последствия):** N/5 — [обоснование, ₽ при наличии]
**C (стоимость проверки):** N/5 — [обоснование]

**Score = P×I/C = [значение]**

**Методы проверки:** [2-5 быстрых экспериментов]
```

Отсортируй по Score по убыванию. Выведи все риски (не ограничивай количество — столько, сколько релевантно).

**Сохрани** результат в `04-risks.md`. Обнови `_spring-context.md`.

### Агент 2: Проектировщик PRD

**Критерий приёмки:** написать продуктовые требования, покрывающие 90% функциональности для создания ценности на выбранных jobs. Уровень функциональности — не хуже конкурентов.

Запусти через `Task` с `subagent_type: "general-purpose"` ПОСЛЕ завершения Агента 1:

```
Ты — продуктовый проектировщик. Твоя задача — написать продуктовые требования (PRD).

## Контекст
Идея продукта: {вставь из 00-product-idea.md}
Бизнес-контекст: {вставь из 01-business-context.md}
Выбранные сегменты и jobs: {вставь из 03-chosen-segments.md}
Риски для учёта: {вставь из 05-chosen-risks.md}
Анализ конкурентов: {вставь из 06-competitors.md}

## Задача
Напиши продуктовые требования (PRD), которые:
1. Покрывают 90% функциональности, необходимой для создания ценности на выбранных core jobs
2. Реализуют функциональность на уровне не хуже конкурентов (используй анализ конкурентов)
3. Учитывают выбранные риски (если есть)

## Структура PRD

### 1. Обзор продукта
- Название
- Цель продукта (какую ценность создаёт)
- Целевые сегменты и jobs

### 2. Целевые пользователи
- Для каждого сегмента: кто они, какие jobs, какие критерии успеха

### 3. Функциональные требования
Для каждой core job из выбранных сегментов:
- Что продукт должен делать, чтобы выполнить эту job
- Критическая последовательность работ (КПР): шаги, которые пользователь не может не выполнить
- Критерии приёмки для каждого требования

### 4. Конкурентный паритет
- Функциональность, которая должна быть не хуже конкурентов (на основе анализа)
- Функциональность, которой нет у конкурентов (наше преимущество)

### 5. Нефункциональные требования
- Производительность, безопасность, масштабируемость (если релевантно)

### 6. Учёт рисков
- Для каждого выбранного риска: как PRD его минимизирует или учитывает

### 7. Метрики успеха
- Как измерить, что product создаёт ценность на выбранных jobs

### 8. Что НЕ входит в scope
- Явные ограничения и исключения

## Правила
- Пиши конкретно, с критериями приёмки
- Формулируй требования от лица пользователя: "Пользователь может..."
- Каждое требование должно быть связано с конкретной job
- Не добавляй функциональность, которая не создаёт ценность на выбранных jobs
```

### Агент 3: Edge-case аналитик

**Критерий приёмки:** сгенерировать 90% наиболее вероятных edge-кейсов для выбранных сегментов и jobs.

Запусти через `Task` с `subagent_type: "general-purpose"` ПОСЛЕ завершения Агентов 1 и 2:

```
Ты — аналитик edge-кейсов. Твоя задача — найти все важные граничные случаи.

## Контекст
Выбранные сегменты и jobs: {вставь из 03-chosen-segments.md}
PRD: {вставь из 07-prd.md}

## Задача
Сгенерируй edge-кейсы — ситуации, которые могут сломать продукт или создать негативный пользовательский опыт. Цель: покрыть 90% наиболее вероятных edge-кейсов.

## Категории edge-кейсов
Для каждого функционального требования из PRD проверь:

1. **Ввод данных:** Что если пустой ввод? Слишком длинный? Спецсимволы? Другой язык? Нулевое значение? Отрицательное число?
2. **Сеть и инфраструктура:** Что если нет интернета? Медленный интернет? Таймаут сервера? Потеря соединения в процессе?
3. **Состояние пользователя:** Что если новый пользователь? Вернувшийся? Заблокированный? Несколько сессий? Другой часовой пояс?
4. **Бизнес-логика:** Что если 0 элементов? 1 элемент? 10000 элементов? Параллельные действия? Конфликтующие операции?
5. **Безопасность:** Prompt injection? XSS? SQL injection? Несанкционированный доступ? Превышение лимитов?
6. **Платежи и экономика (если применимо):** Отмена платежа? Частичный возврат? Просроченная подписка? Двойная оплата?
7. **Мультипользовательские сценарии:** Что если два пользователя одновременно? Конфликт прав? Общий ресурс?
8. **Контекст сегмента:** Специфичные для выбранных сегментов ситуации, которые могут вызвать проблемы

## Формат результата

Для каждого edge-кейса:

| # | Требование из PRD | Edge-кейс | Что может пойти не так | Рекомендация | Приоритет (Critical/High/Medium/Low) |

Отсортируй по приоритету: Critical → High → Medium → Low.

Сохрани результат через Write в файл: {путь к temp}/08-edge-cases.md
```
