# Скилл: Создание черновика главы (Double Diamond — СТРУКТУРИРОВАНИЕ)

Этот скилл превращает тезисы в связный черновик главы через **команду из 4 агентов**. Это этап СТРУКТУРИРОВАНИЯ — создаём читаемый текст, следуем структуре книги.

⚠️ **КЛЮЧЕВОЙ ПРИНЦИП: СУТЬ ТЕЗИСА = КАРКАС СЕКЦИИ. ПОЛИРОВКА, НЕ ГЕНЕРАЦИЯ.**

Ты НЕ "пишешь текст по теме тезиса". Ты БЕРЁШЬ текст "Сути" каждого тезиса как ГОТОВЫЙ КАРКАС секции и ПОЛИРУЕШЬ его: обогащаешь точными цитатами, вплетаешь данные, выстраиваешь переходы между секциями. "Суть" — это уже черновик. Твоя работа — РЕДАКТОР, а не АВТОР.

**АНТИПАТТЕРН (старый подход):** Извлечь атомарные элементы (цифры, цитаты) → собрать из них новый текст → потерять 60-80% логики и аргументации тезисов.

**ПРАВИЛЬНЫЙ ПОДХОД:** Взять текст "Сути" каждого тезиса как есть → каждый тезис = своя секция → writer полирует язык и вплетает цитаты/данные → нарратив тезисов сохраняется.

## ЭТАП 0: Извлечение MUST-INCLUDE элементов (ДЕЛАЕТ ЛИД)

⚠️ **КРИТИЧЕСКИ ВАЖНО:** Этот этап определяет качество всего черновика. Ты НЕ пересказываешь тезисы, ты ИЗВЛЕКАЕШЬ из них конкретные элементы, которые ОБЯЗАНЫ появиться в тексте.

### Шаг 0.1: Прочитай файл тезисов ЦЕЛИКОМ

Прочитай весь файл тезисов от начала до конца. Обрати внимание на:
- **Секцию "Инструкция для draft-этапа"** — авторские указания для черновика
- **Секцию "Главная тема"** — центральное сообщение главы
- **Карту экспертов** — кого цитировать
- **КАЖДЫЙ тезис** — со всеми подсекциями (Суть, KB, Эксперты, Контраргументы)
- **Секцию "Дебаты команды"** — ключевые споры для драматургии
- **Секцию "Примеры и кейсы"** — материал для историй
- **Секцию "Рекомендации по усилению"** — идеи для подачи

### Шаг 0.2: Извлеки MUST-INCLUDE элементы из КАЖДОГО тезиса

Для КАЖДОГО тезиса пройдись по его содержимому и извлеки ВСЕ конкретные элементы шести типов. Копируй ДОСЛОВНО — не пересказывай, не сокращай.

**6 типов MUST-INCLUDE элементов:**

| Тип | Что извлекать | Как копировать |
|-----|--------------|----------------|
| **СУТЬ** | ⚠️ ПЕРВЫМ ДЕЛОМ: ПОЛНЫЙ текст секции "Суть" тезиса | Дословно, целиком, без сокращений. Это КАРКАС будущей секции — самый важный элемент! |
| **DATA** | Каждая конкретная цифра, статистика, факт | Дословно: "4% всех GitHub коммитов от Claude Code" |
| **QUOTE** | Каждая прямая цитата эксперта | Дословно с именем и URL: Boris Cherny: "Coding is largely solved" — [URL] |
| **CONCEPT** | Каждый названный фреймворк, концепция, термин | Дословно: "T-shaped → E/F-shaped навыки", "Mexican standoff" |
| **KB** | Каждая цитата из Knowledge Base | Дословно с путём к файлу |
| **COUNTER** | Каждый контраргумент с источником | Дословно с URL: "METR: 19% МЕДЛЕННЕЕ с AI" — [URL] |

⚠️ **СУТЬ — САМЫЙ ВАЖНЫЙ ТИП.** Текст "Сути" — это уже ГОТОВЫЙ аргумент, логическая цепочка, которая станет основой секции черновика. DATA, QUOTE и остальные типы — это элементы для ОБОГАЩЕНИЯ Сути, а не замена ей.

⚠️ **ПРАВИЛО ИЗВЛЕЧЕНИЯ:** Если элемент есть в тезисе — он попадает в must-include. НЕ ФИЛЬТРУЙ на этом этапе. НЕ РЕШАЙ "это не важно". Извлеки ВСЁ. Фильтрация будет на ЭТАПЕ 3 (Section Briefs).

- [ ] {спор 1}
- [ ] {спор 2}

## Из секции "Примеры и кейсы":
- [ ] {кейс 1}
- [ ] {кейс 2}

## ИТОГО ЭЛЕМЕНТОВ ПО ВСЕМ ТЕЗИСАМ: {N}

### Правила извлечения:

1. **ДОСЛОВНО** — копируй текст элемента как есть, не перефразируй
2. **КАЖДЫЙ URL** — должен быть скопирован рядом с элементом
3. **ЧЕКБОКСЫ `- [ ]`** — каждый элемент начинается с чекбокса (будет отмечен `- [x]` в финальной проверке)
4. **СЧИТАЙ элементы** — итого по каждому тезису и общее количество
5. **НЕ ОБЪЕДИНЯЙ** — каждый отдельный факт, цитата, цифра = отдельный элемент. Ориентир: 8-20 элементов на тезис. Если < 5 — ты что-то пропустил.

## Принцип работы

**СБОРКА через КОМАНДУ АГЕНТОВ:**

```
ЭТАП 0 (Лид): Извлечь MUST-INCLUDE элементы из тезисов
         ↓
ЭТАП 1 (Лид): Изучить контекст книги и издания
         ↓
ЭТАП 2 (Параллельно):
    ├── style-researcher → Style Guide
    ├── case-hunter → Кейсы для каждого тезиса
    └── narrative-architect → Нарративная арка + порядок + word budget
         ↓
ЭТАП 3 (Лид): Собрать Section Briefs = must-include + кейсы + нарратив + style
         ↓
ЭТАП 4 (writer): Двухпроходная сборка → черновик
         ↓
ЭТАП 5 (Лид): Поэлементная верификация → исправление пропусков
         ↓
ЭТАП 6 (Лид): Финальный файл
```

**Почему так:**
- **Провальный подход:** "извлечь атомы из тезисов → собрать из них новый текст" → потеря 60-80% логики и аргументации
- **Правильный подход:** "Суть" каждого тезиса = каркас секции → writer ПОЛИРУЕТ и ОБОГАЩАЕТ, а не генерирует заново
- **Каждый тезис = своя секция** — не объединяем, не теряем аргументы
- **style-researcher** даёт конкретные правила оформления
- **case-hunter** находит яркие примеры для обогащения
- **narrative-architect** определяет ПОРЯДОК тезисов-секций и ПЕРЕХОДЫ между ними (не перестраивает контент!)
- **writer** берёт текст Сути как стартовую точку, вплетает цитаты/данные, полирует язык
- **Двухпроходная полировка:** Проход 1 = обогатить Суть элементами, Проход 2 = отполировать стиль
- **Верификация:** сначала проверяем сохранение СУТИ (главное!), потом — наличие элементов
