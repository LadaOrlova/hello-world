# Требование к ЗИС: Контроль доступа к персональным данным клиентов

> **Автор:** Иван Замесин
> **Дата:** 2026-03-18
> **Статус:** Черновик требования

---

## Суть

По умолчанию новые члены команды НЕ должны иметь доступ ко всем контактным данным всех пользователей. Доступ к персональным данным клиентов (email, телефон, Telegram) выдаётся только определённым ролям и только в рамках их рабочих задач.

---

## Проблема

Когда в команду приходят новые сотрудники (менеджеры по продажам, кураторы, сотрудники поддержки), они потенциально могут получить доступ ко всей базе клиентов. Это создаёт риски:

- **Утечка базы клиентов** — сотрудник может выгрузить все контакты и унести с собой при увольнении
- **Злоупотребление данными** — массовые рассылки, передача данных третьим лицам
- **Нет аудита** — непонятно, кто и когда получал доступ к каким данным

---

## Принцип: минимальный доступ по умолчанию

Доступ к персональным данным клиентов строится по принципу **need-to-know**: сотрудник видит только те данные, которые нужны ему для выполнения текущей задачи.

### Уровни доступа к данным клиентов

| Уровень | Что видит | Кому выдаётся |
|---------|-----------|---------------|
| **Полный доступ** | Все контактные данные всех пользователей. Может выгружать базу в CSV/Excel | Владелец (Иван), администраторы с явным разрешением |
| **Рабочий доступ** | Контактные данные только тех клиентов, с которыми сотрудник сейчас работает. По одному клиенту за раз | Менеджеры по продажам, сотрудники поддержки |
| **Ограниченный доступ** | Видит список студентов потока (имя, прогресс), но НЕ видит контактные данные (email, телефон, Telegram) | Кураторы |
| **Без доступа** | Не видит данных клиентов вообще | Новые сотрудники по умолчанию, контент-менеджеры |

---

## Что значит "рабочий доступ"

Это ключевая механика. Менеджер по продажам или сотрудник поддержки:

1. **Видит список клиентов**, с которыми работает — но без контактных данных (только имя и контекст: продукт, поток, статус)
2. **Чтобы увидеть контакты конкретного клиента** — открывает карточку клиента. Система логирует этот доступ
3. **Не может выгрузить контактные данные** — в интерфейсе нет кнопки экспорта для этой роли
4. **Не может массово просмотреть контакты** — нет списочного отображения с контактными данными, только поштучный просмотр через карточку

### Rate limiting на просмотр контактов

Чтобы сотрудник не мог написать скрипт, который обходит карточки одну за другой и выгружает базу:

- **Ограничение на количество просмотров карточек** — например, не более 50 уникальных карточек клиентов в день для роли "менеджер по продажам"
- **Алерт при аномальной активности** — если сотрудник просматривает карточки клиентов значительно чаще обычного, система уведомляет администратора
- **API rate limiting** — ограничение на количество запросов к endpoint'у с контактными данными. Усложняет написание скриптов для массовой выгрузки
- **Нет bulk-endpoint'а** — для ролей с рабочим доступом нет API, возвращающего контакты нескольких клиентов за один запрос

## Сценарии использования

### Сценарий 1: Менеджер по продажам работает с лидом

1. Менеджер заходит в CRM-раздел, видит список своих лидов (имя, продукт, статус сделки)
2. Открывает карточку конкретного лида — видит его email и телефон
3. Связывается с клиентом, фиксирует результат в карточке
4. Не может открыть 200 карточек подряд — после 50 просмотров за день система блокирует доступ и уведомляет администратора

### Сценарий 2: Сотрудник поддержки помогает студенту

1. Студент пишет в поддержку
2. Сотрудник поддержки ищет студента по имени или email (который студент сам сообщил)
3. Открывает карточку студента — видит контактные данные и историю обучения
4. Решает вопрос

### Сценарий 3: Куратор ведёт поток

1. Куратор видит список студентов своего потока: имя, прогресс, статус
2. Для связи со студентом — использует чат внутри платформы или групповой Telegram-чат (куда студент сам вступил)
3. НЕ видит email и телефон студентов

### Сценарий 4: Новый сотрудник выходит на работу

1. Получает аккаунт в ЗИС с ролью по умолчанию (без доступа к данным клиентов)
2. Руководитель назначает ему роль (менеджер, поддержка, куратор)
3. Система выдаёт доступ в соответствии с ролью

### Сценарий 5: Попытка выгрузить базу

1. Менеджер по продажам пытается через DevTools или скрипт обойти карточки по очереди
2. После 50 запросов за день — API перестаёт возвращать контактные данные
3. Администратор получает уведомление об аномальной активности
4. Администратор разбирается в ситуации и при необходимости отзывает доступ

## Технические требования

### 1. Row-level security (RLS)

Контроль доступа реализуется **на уровне базы данных**, а не только на уровне UI:

- Запросы к таблице `User` автоматически фильтруют контактные поля (email, phone, telegram_username) в зависимости от роли запрашивающего
- Даже при прямом SQL-запросе через ORM — данные не вернутся без соответствующих прав

### 2. API-уровень

- Endpoint получения контактных данных клиента — отдельный от основного профиля
- Для ролей с рабочим доступом: только single-record endpoint (`GET /api/users/:id/contacts`), без bulk-операций
- Rate limiting: настраиваемый лимит запросов в день на пользователя (по умолчанию 50)
- Все запросы к контактным данным логируются (кто, когда, чьи данные запросил)

### 3. Аудит

| Что логируется | Поля |
|----------------|------|
| Просмотр контакта клиента | who (сотрудник), whom (клиент), when, ip_address |
| Выгрузка базы (для полного доступа) | who, when, filters, row_count |
| Изменение роли сотрудника | who_changed, whom, old_role, new_role, when |
| Блокировка по rate limit | who, when, request_count |

### 4. Управление ролями

- В админке — раздел управления ролями и правами доступа
- Только владелец (Иван) и назначенные администраторы могут менять роли
- При изменении роли — мгновенное изменение доступа (не нужен перелогин)

---

## Что НЕ входит в первую версию

- Гранулярные разрешения (доступ к конкретным полям, например "видит email, но не телефон") — возможно в будущем
- Временный доступ (выдать доступ на 24 часа) — заложить в архитектуру
- Двухфакторная подтверждение для просмотра контактов — пока избыточно

---

## Приоритет

Это фундаментальное требование безопасности. Должно быть заложено в архитектуру с самого начала — добавить row-level security после запуска значительно сложнее, чем сразу.

---

## Комментарий для Жени Васева

Жень, это важное требование по безопасности данных. Ключевое: нельзя, чтобы любой сотрудник мог выгрузить всю базу клиентов. Доступ к контактным данным — только поштучно, через карточку, с rate limiting и аудитом.

В архитектуре нужно заложить:
1. **Row-level security** на уровне PostgreSQL (не только проверка в коде)
2. **Отдельный API endpoint** для контактных данных (не вшивать в общий запрос профиля)
3. **Таблица аудита** — логировать каждый доступ к контактным данным
4. **Rate limiting** — настраиваемый лимит на просмотр контактов по роли
