# Требование к ЗИС: Выгрузка базы пользователей по всем потокам всех продуктов

> **Автор:** Иван Замесин
> **Дата:** 2026-03-13
> **Статус:** Черновик требования
> **Связанные бизнес-кейсы:** ADMIN-03 (выгрузка данных), AUTH-03 (полная история пользователя), DRAFT-039 (экспорт CSV/Excel)

---

## Суть

В админке ЗИС нужна возможность выгрузить полную базу пользователей со всей информацией о их участии во всех потоках всех продуктов. Одна таблица, в которой видно: кто, в каком продукте, в каком потоке, по какому тарифу учился, с каким статусом.

---

## Проблема сейчас

Данные о пользователях разбросаны по разным системам (AirTable, отдельные админки Boost и КДП). Нет единого места, где можно увидеть и выгрузить полный список всех пользователей по всем продуктам и потокам. Чтобы собрать сводную информацию — нужно вручную собирать данные из разных источников.

---

## Что нужно

### 1. Страница в админке: "База пользователей"

Раздел в админке (`/admin/users/database` или аналогичный), где отображается сводная таблица всех пользователей с их зачислениями.

**Колонки таблицы:**

| Колонка | Источник (доменная модель) | Описание |
|---------|---------------------------|----------|
| ФИО | `User.name` | Имя пользователя |
| Email | `User.email` | Email (уникальный идентификатор) |
| Телефон | `User.phone` | Телефон |
| Telegram | `User.telegram_username` | Username в Telegram |
| Роль | `User.role` | admin / manager / curator / student |
| Продукт | `Product.title` | Название продукта (Boost, КДП, и т.д.) |
| Поток | `Cohort.title` + `Cohort.number` | Номер и название потока |
| Тариф | `Tier.title` | Название тарифа (Стандарт+, Практика, Максимум, и т.д.) |
| Статус зачисления | `Enrollment.status` | pending / active / completed / refunded / suspended |
| Дата зачисления | `Enrollment.enrolled_at` | Когда зачислен |
| Дата завершения | `Enrollment.completed_at` | Когда завершил (если завершил) |
| Цена на момент покупки | `Enrollment.price_snapshot` | Снапшот цены |
| Валюта | `Enrollment.price_snapshot_currency` | RUB / USD |
| Компания (B2B) | `Company.name` | Название компании (если B2B) |
| Дата регистрации | `User.created_at` | Когда зарегистрировался в системе |

**Важно:** Один пользователь может иметь несколько строк — по одной на каждое зачисление (enrollment). Если пользователь учился в 3 потоках 2 продуктов, у него будет 3 строки.

### 2. Фильтры

Обязательные фильтры на странице:

| Фильтр | Описание |
|--------|----------|
| Продукт | Выбор одного или нескольких продуктов (Boost, КДП, и др.) |
| Поток | Выбор конкретного потока (зависит от выбранного продукта) |
| Тариф | Фильтр по тарифу |
| Статус зачисления | active / completed / refunded / suspended / все |
| Период зачисления | Диапазон дат: "зачислен с ... по ..." |
| Поиск | Текстовый поиск по ФИО, email, телефону, Telegram |
| Компания | Фильтр по компании (для B2B) |

### 3. Выгрузка (экспорт)

- Кнопка **"Выгрузить в CSV"** — скачивает таблицу с учётом текущих фильтров
- Кнопка **"Выгрузить в Excel"** — скачивает .xlsx с учётом текущих фильтров
- Выгружаются **все строки**, не только видимые на текущей странице пагинации
- В выгрузке те же колонки, что отображаются в таблице на экране

### 4. Режимы отображения

Два режима таблицы (переключатель):

1. **По зачислениям** (по умолчанию) — каждая строка = одно зачисление. Один пользователь может занимать несколько строк.
2. **По пользователям** — каждая строка = один пользователь. Продукты/потоки/тарифы агрегируются в одну ячейку (например: "КДП (Поток 58, Практика); Boost (Поток 3, Интенсив)").

## Сценарии использования

### Сценарий 1: Менеджер хочет понять, кто учится в текущем потоке

1. Заходит в "Базу пользователей"
2. Выбирает фильтр: Продукт = "КДП", Поток = "Поток 60"
3. Видит список всех студентов этого потока с тарифами и статусами
4. Нажимает "Выгрузить в Excel" — получает файл со списком

### Сценарий 2: Иван хочет видеть полную картину по всем пользователям

1. Заходит в "Базу пользователей"
2. Не ставит никаких фильтров
3. Видит полный список всех пользователей по всем продуктам и потокам
4. Нажимает "Выгрузить в CSV" — получает полную выгрузку

### Сценарий 3: Настя хочет найти всех B2B-студентов определённой компании

1. Заходит в "Базу пользователей"
2. Ставит фильтр: Компания = "ООО Ромашка"
3. Видит всех сотрудников этой компании, которые учились в любых продуктах
4. Выгружает список для отчёта

### Сценарий 4: Маркетинг хочет найти alumni для upsell

1. Заходит в "Базу пользователей"
2. Ставит фильтры: Продукт = "КДП", Статус = "completed"
3. Видит всех выпускников КДП
4. Выгружает список для email-рассылки с предложением Boost

## Связь с доменной моделью

Основной запрос строится через JOIN:

```
User
  LEFT JOIN Enrollment ON Enrollment.user_id = User.id
  LEFT JOIN Cohort ON Enrollment.cohort_id = Cohort.id
  LEFT JOIN Product ON Cohort.product_id = Product.id
  LEFT JOIN Tier ON Enrollment.tier_id = Tier.id
  LEFT JOIN Order ON Enrollment.order_id = Order.id
  LEFT JOIN Company ON Enrollment.company_id = Company.id
```

Пользователи без зачислений (например, менеджеры, кураторы) тоже должны быть в выгрузке — с пустыми полями продукта/потока.

---

## Требования к производительности

- Таблица должна работать с пагинацией (50-100 записей на странице)
- Выгрузка должна работать для десятков тысяч записей (12 000+ выпускников только КДП)
- Выгрузка больших объёмов — через фоновую задачу (pg-boss job), чтобы не блокировать браузер. Результат — ссылка на скачивание файла

## Комментарий для Жени Васева

Жень, это то, что нужно заложить в ЗИС — полноценная выгрузка базы пользователей. Ключевое: это не просто список User, а **сводная таблица User × Enrollment × Product × Cohort × Tier**, с фильтрами и экспортом в CSV/Excel.

В прототип добавлять не нужно, но в архитектуре это уже заложено — все нужные связи в доменной модели есть (User → Enrollment → Cohort → Product, Enrollment → Tier, Enrollment → Company). Нужен только UI и API endpoint для этого запроса.
