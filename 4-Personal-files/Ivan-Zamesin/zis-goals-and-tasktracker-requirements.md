- Систему целеполагания, в которой повышение и премии сможет видеть **только тот сотрудник**, для кого это повышение и премии
- Другие сотрудники не смогут каким-либо образом получить эти данные (ни через API, ни через прямой доступ к БД, ни через UI)

Это значит, что в архитектуре нужно заложить:
1. **Row-level security** — данные о бонусах фильтруются на уровне БД, а не только на уровне UI
2. **Разделение прав доступа** — роли: владелец (видит всё), сотрудник (видит только своё), менеджер (видит своих подчинённых, если потребуется)
3. **Аудит доступа** — логирование, кто и когда запрашивал данные о бонусах

## Часть 1: Система целеполагания (OKR)

### 1.1. Суть

В админке ЗИС — раздел с целями компании и сотрудников по системе OKR (Objective — Key Results). Цели компании каскадируются на цели сотрудников. Выполнение целей привязано к бонусам.

### 1.2. Иерархия целей

```
Цели компании (на период)
  └── Objective 1
        ├── Key Result 1.1
        ├── Key Result 1.2
        └── Key Result 1.3
  └── Objective 2
        ├── Key Result 2.1
        └── Key Result 2.2

Цели сотрудника (на период)
  └── Objective A (связан с Objective 1 компании)
        ├── Key Result A.1
        └── Key Result A.2
  └── Objective B (личная цель развития)
        ├── Key Result B.1
        └── Key Result B.2
```

### 1.3. Периоды целеполагания

- Базовые периоды: **год**, **полгода**, **2 месяца**
- Можно задать произвольный период (например, 3 месяца, квартал)
- У каждого периода: дата начала, дата окончания, название
- Периоды вложенные: годовые цели → полугодовые → двухмесячные
- Можно создавать новые периоды и менять их длительность

### 1.4. Атрибуты Objective

| Поле | Тип | Описание |
|------|-----|----------|
| Название | Текст (300 символов) | Формулировка цели |
| Период | Ссылка на период | К какому периоду относится |
| Уровень | Enum | Компания / Сотрудник |
| Владелец | Ссылка на сотрудника | Кто отвечает за цель |
| Родительская цель | Ссылка на Objective | Связь с целью верхнего уровня (компании) |
| Статус | Enum | Черновик / Активная / Завершена / Отменена |
| Прогресс | % (0-100) | Рассчитывается из Key Results |

### 1.5. Атрибуты Key Result

| Поле | Тип | Описание |
|------|-----|----------|
| Название | Текст (300 символов) | Формулировка результата |
| Тип метрики | Enum | Число / Процент / Да-Нет / Валюта |
| Начальное значение | Число | Откуда стартуем |
| Целевое значение | Число | Куда хотим прийти |
| Текущее значение | Число | Где сейчас (обновляется вручную, в будущем — автоматически) |
| Прогресс | % (0-100) | Рассчитывается: (текущее - начальное) / (целевое - начальное) × 100 |
| Владелец | Ссылка на сотрудника | Кто отвечает |
| Статус | Enum | На ходу / По плану / Отстаём / Перевыполняем / Завершён |

### 1.6. Связь целей сотрудника с целями компании

- Каждый Objective сотрудника **может** быть привязан к Objective компании (поле "Родительская цель")
- Может быть и без привязки — личная цель развития
- На странице цели компании видно, какие цели сотрудников к ней привязаны и их прогресс
- На странице сотрудника видно все его цели и к каким целям компании они относятся

### 1.7. Бонусы и повышения (приватная часть)

> **Конфиденциально:** видит только сам сотрудник и владелец системы (Иван)

| Поле | Тип | Описание |
|------|-----|----------|
| Тип бонуса | Enum | Повышение зарплаты / Разовая премия / Процент от прибыли |
| Условие | Текст | При каких условиях начисляется (например, "KR 1.1 ≥ 80%") |
| Привязка к KR | Ссылка на Key Result(s) | К каким результатам привязан бонус |
| Размер | Текст | Сумма или формула (например, "5% от прибыли курса КДП") |
| Статус | Enum | Ожидает / Начислен / Отменён |

**Что видит сотрудник, заходя в свой профиль:**
- Свои цели и прогресс
- Какие бонусы привязаны к каким целям
- Какое повышение зарплаты / премия ждёт при достижении целей
- Процент от прибыли (какой процент, от какого курса/продукта)

**Что НЕ видит сотрудник:**
- Бонусы других сотрудников
- Зарплаты других сотрудников

### 1.8. Отображение и навигация

**Страница "Цели компании":**
- Выбор периода (год / полгода / 2 месяца / произвольный)
- Список Objectives с прогресс-барами
- Раскрытие каждого Objective → список Key Results с прогрессом
- Под каждым Objective компании — связанные цели сотрудников

**Страница "Мои цели" (у сотрудника):**
- Список своих Objectives на текущий период
- Прогресс по каждому KR
- Блок "Мои бонусы" — приватный, видит только этот сотрудник

**Индикация прогресса:**
- По плану (зелёный) — прогресс соответствует времени периода
- Отстаём (жёлтый/красный) — прогресс отстаёт от ожидаемого
- Перевыполняем (синий) — опережаем план

### 1.9. Будущие возможности (заложить в архитектуру)

- Автоматическая синхронизация KR с метриками из аналитики (Amplitude, PostHog, внутренние данные ЗИС)
- Уведомления о дедлайнах и отставании
- Исторические данные: сравнение периодов
- Экспорт целей в PDF / Markdown

## Часть 2: Таск-трекер (Kanban-доска)

### 2.1. Суть

Kanban-доска в админке ЗИС — замена Notion. Доска с колонками, которые можно создавать, скрывать, менять порядок. Внутри колонок — секции (проекты), внутри секций — задачи.

### 2.2. Структура доски

```
Доска (например, "ZIS")
  └── Колонка "Sprint 13"
        ├── Секция "Разработка"
        │     ├── Задача ZIS-142
        │     └── Задача ZIS-143
        ├── Секция "Продажи"
        │     └── Задача ZIS-150
        └── Секция "Маркетинг"
              └── Задача ZIS-155
  └── Колонка "Sprint 13 Done"
        ├── Секция "Разработка"
        │     └── Задача ZIS-140
        └── Секция "Продажи"
              └── Задача ZIS-148
  └── Колонка "Sprint 12" (скрыта)
  └── Колонка "Sprint 12 Done" (скрыта)
```

### 2.3. Колонки

- Создаются вручную: нажимаешь "Создать колонку", вводишь название
- **Кнопка "Новый спринт"**: вводишь номер спринта (например, 13) → создаются две колонки: "Sprint 13" и "Sprint 13 Done"
- Порядок колонок меняется drag-and-drop (по горизонтали)
- Горизонтальный скролл, если колонок больше, чем помещается на экран
- Колонки можно **скрыть** (кнопка "скрыть") — они не удаляются, а прячутся
- Есть кнопка "Показать скрытые колонки" — подгружает все скрытые колонки
- Предыдущие спринты не закрываются автоматически — только вручную (скрыть)

### 2.4. Секции внутри колонок

- Секции = горизонтальные разделители внутри колонки (как в Notion Kanban)
- Примеры секций: "Разработка", "Продажи", "Маркетинг", "Контент"
- Секции **единые для всех колонок** — если создал секцию "Разработка", она появляется во всех колонках
- Можно создавать новые секции, удалять, менять порядок (сверху вниз)
- Задачи внутри секции сортируются drag-and-drop

### 2.5. Атрибуты задачи

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|----------|
| ID | Авто | Да | Уникальный идентификатор, формат: `ZIS-{число}` (auto-increment) |
| Название | Текст (300 символов) | Да | Краткое описание задачи |
| Описание | Markdown | Нет | Подробное описание. Два режима: редактирование (markdown-редактор) и отображение (рендеренный markdown). Кнопка "Сохранить" |
| Ответственный | Ссылка на сотрудника | Нет | Кто делает задачу |
| Помощники | Список ссылок на сотрудников | Нет | Кто помогает |
| Приоритет | Enum | Нет | Высокий / Средний / Низкий |
| Теги | Список строк | Нет | Произвольные метки для фильтрации и группировки |
| Секция | Ссылка на секцию | Да | К какому проекту/направлению относится |
| Колонка | Ссылка на колонку | Да | В каком спринте/статусе |

### 2.6. Перемещение задач

- Drag-and-drop между колонками (например, из "Sprint 13" в "Sprint 13 Done")
- Drag-and-drop внутри колонки (изменение порядка, перемещение между секциями)
- При перетаскивании между колонками секция задачи сохраняется (задача попадает в ту же секцию в новой колонке)

### 2.7. Фильтрация "Мои задачи"

- Кнопка "Мои задачи" — фильтрует доску, показывая только задачи, где текущий пользователь — ответственный или помощник
- Работает по всем видимым колонкам

### 2.8. Идентификаторы и проекты

- Базовый формат ID: `ZIS-{число}` (например, ZIS-1, ZIS-2, ..., ZIS-142)
- Auto-increment, глобально уникальный в рамках доски
- **Будущее:** возможность создавать отдельные проекты (доски) с собственным префиксом ID (например, `KDP-1`, `BOOST-1`)

### 2.9. Будущие возможности (заложить в архитектуру)

- Комментарии к задачам (тред комментариев с историей)
- Связь задачи с Key Result из OKR (задача привязана к цели)
- Подзадачи / чек-листы внутри задачи
- Дедлайны и уведомления
- Фильтры и поиск по задачам (по ответственному, приоритету, секции)
- Множественные доски (проекты) с отдельными Kanban

## Решения по открытым вопросам

> Ответы зафиксированы 2026-03-11

### По OKR:

1. **Кто ставит цели сотрудникам?** — Совместно на 1-on-1. Сотрудник приносит черновик целей, на 1-on-1 обсуждаем и заносим. Занести может и сотрудник сам. Иван может отредактировать и сохранить, может отменить цель.
2. **Как часто обновляется прогресс KR?** — В первой версии сотрудник обновляет сам. В будущем — автоматически из данных (заложить в архитектуру).
3. **Нужна ли история изменений KR?** — Да, обязательно. Логировать: кто, когда, какое значение было → стало.
4. **Формулы бонусов:** — Калькулятор не нужен. Достаточно текстового описания условий бонуса.
5. **Период по умолчанию:** — При входе показывать текущий **двухмесячный** период.

### По таск-трекеру:

1. **Лимит задач в колонке?** — Нет, WIP-лимит не нужен.
2. **Архив:** — Скрытые колонки хранятся **вечно**.
3. **Метки/теги:** — Да, добавить теги для задач (помимо секций и приоритета).
4. **Уведомления:** — Нет, не нужны.
5. **Мобильная версия:** — Да, нужна адаптивная версия для телефона.
