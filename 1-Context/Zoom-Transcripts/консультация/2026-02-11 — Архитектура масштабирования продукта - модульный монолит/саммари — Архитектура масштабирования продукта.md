# [Продуктовый-созвон] Архитектура масштабирования продукта
**Дата**: 11 февраля 2026

---

## Контекст
Обсуждение архитектуры масштабирования существующего продукта, направленное на автоматизацию процессов, повышение эффективности и интеграцию с AI-агентами, при сохранении возможности органического роста. Текущая система характеризуется разрозненностью и ручным обслуживанием.

## Ключевые решения

| Решение | Обоснование | Кто принял |
|---|---|---|
| **Принять архитектуру модульного монолита** | Позволит быстро получать результаты, не упереться в проблемы масштабирования и изменений, обеспечит органический и эволюционный рост. Представляет собой промежуточный этап, избегая годовой разработки "космолёта". | Спикер 1 (Женя), поддержано участниками |
| **Отказаться от внешних форм (Typeform, Google Forms)** | Высокая стоимость (2000-5000 долларов в год за Typeform), периодические сбои и возможность легко реализовать собственные формы, используя существующую дизайн-систему. | Спикер 3 (Ваня), Спикер 5 (Виталий), Спикер 1 (Женя) |
| **Исключить роль "Компания", заменив её связью** | Даёт гибкость: пользователь может быть представителем компании и одновременно студентом, обучаться на разных продуктах/потоках, а его знания остаются с ним независимо от компании. | Спикер 1 (Женя) |
| **Переписать LMS (систему управления обучением)** | Текущая LMS не вписывается в новую архитектуру и требует полного переосмысления или значительной переработки для обеспечения масштабирования и автоматизации. | Спикер 3 (Ваня), Спикер 5 (Виталий) |
| **Полностью отказаться от RTTable** | RTTable является разрозненной системой и источником ручной работы. Переход на единую базу данных и собственную админку позволит централизовать данные и процессы, обеспечит удобный и комфортный интерфейс для бэк-офиса. | Спикер 3 (Ваня), Спикер 4 (Максим) |

## Обсуждаемые гипотезы и идеи
-   **B2B личный кабинет**: Предоставить руководителям и HR-специалистам доступ к информации о прогрессе обучения их сотрудников (посещаемость, домашние задания). **Статус**: Принята, будет реализовано как часть системы.
-   **Автоматическое назначение менеджеров и кураторов**: Использовать алгоритм Round Robin для равномерного распределения новых лидов/студентов между свободными сотрудниками. **Статус**: Принята, интегрирована в предлагаемые сценарии.
-   **Использование Event Bus для синхронных внутренних событий**: Внутренние события, такие как запись в БД или обновление статусов, будут публиковаться через Event Bus для синхронизации поведения модулей. **Статус**: Ключевой элемент архитектуры.
-   **Асинхронное выполнение тяжёлых задач через воркер**: Отправка email, уведомлений в Telegram, вызов внешних API (Юкасса), генерация сертификатов/счетов, AI-суммаризации будут выполняться асинхронно через специальный воркер (PG Boss). **Статус**: Ключевой элемент архитектуры.
-   **Интеграция с Zoom API**: Автоматическое создание комнат для воркшопов, управление брейкаут-румами (требует уточнения возможностей API), отправка уведомлений и отслеживание посещаемости. **Статус**: Принята, но детали реализации брейкаут-румов и точного отслеживания посещаемости требуют проработки.
-   **Стандартизация API для AI-агентов (LLM.txt)**: Создание описания API-методов в формате, понятном LLM, для упрощения взаимодействия агентов с системой. **Статус**: Принята как подход к интеграции с AI.

## Технические и продуктовые требования
-   **Архитектура**: Модульный монолит с внутренней коммуникацией через шину данных (Event Bus) для синхронных процессов.
-   **Асинхронная обработка**: Все тяжелые задачи (отправка уведомлений, вызов внешних API, генерация документов, AI-суммаризации) должны выполняться асинхронно через воркер (используется PG Boss).
-   **Единая база данных**: Вся система должна использовать единую базу данных; миграция всех разрозненных данных (включая RTTable) обязательна.
-   **Ролевая модель**: Должна быть гибкой и включать роли: админ, модератор-менеджер, куратор, пользователь. Управление возможностями каждой роли.
-   **Дедупликация лидов**: Система должна автоматически проверять, существует ли новый лид уже в базе данных.
-   **Автоматическое назначение менеджеров/кураторов**: Использовать алгоритмы (например, Round Robin) для распределения нагрузки между сотрудниками.
-   **Собственная админ-панель**: Для менеджеров, кураторов и других сотрудников, заменяющая функционал RTTable.
-   **Жизненный цикл студента**: Полностью автоматизирован от заполнения формы до онбординга, обучения и оффбординга.
    -   **Онбординг**: Автоматическая выдача доступа к LMS, создание персональной онбординг-страницы, генерация персонального календаря, автоматическое назначение куратора, отправка Welcome-сообщений (Telegram с фоллбэком на email).
    -   **Отслеживание прогресса**: Система должна отслеживать открытие уроков, выполнение домашних заданий, посещаемость воркшопов.
    -   **Сбор обратной связи**: Автоматический сбор фидбека от студентов после уроков/курса.
    -   **Сертификация**: Автоматическая генерация PDF-сертификатов на основе шаблона при выполнении критериев (например, посещаемость > 70%, все ДЗ сданы).
-   **B2B личный кабинет**: Предоставление доступа к дашборду с прогрессом студентов для представителей компаний.
-   **API для внешних сервисов**: Предоставление API доступа для ботов и AI-агентов с возможностью генерации ключей для разных ролей и строгим контролем прав доступа на бэкэнде.
-   **Миграция данных с RTTable**: Проведение "Dry Run" миграции, импорт данных, верификация, и консервация RTTable на 2-4 недели как бэкапа перед полным отказом.

## Приоритеты
*   **Заложить фундамент системы**: В первую очередь необходимо реализовать базовые компоненты – систему пользователей, Event Bus и PG Boss. Это критически важно, так как без этой основы невозможно эффективно выстраивать масштабирование и автоматизацию.
