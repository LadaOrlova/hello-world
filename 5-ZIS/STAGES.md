## 1) Критерии приёмки Прототипа (этап 0)

- для каждого бизнес-кейса из `docs/bc/*.md` оформлен и согласован чек-лист приёмки (что именно проверяем и по какому критерию считаем кейс принятым).
- есть граница что входит в прототип что не входит 
- Виталий и Максим подтверждают, что это рабочий скелет будущего сервиса;

Если этих подтверждений нет, этап считается не закрытым.

## 3) Как теперь работаем (чтобы был контроль)

Перед стартом любого этапа фиксируем 4 вещи:

1. Определяем scope (`что входит / что не входит`).
2. Критерии приёмки (чеклист).
3. Срок.
4. Бюджет.

Правило оплаты: этап не стартует и не оплачивается без согласованных критериев приёмки.

Для задач (бизнес-кейсов `docs/bc/*.md`) зафиксирован тот же принцип:

- у каждого кейса есть `Критерии приёмки`, `Срок`, `Бюджет`;
- для кейсов в scope прототипа в поле бюджета: `входит в прототип`;
- для вынесенных/деприоритезированных кейсов: `вне прототипа (оценивается отдельно)`;
- срок по кейсам на текущем этапе: `TBD`.

## 5) Этапы 1-4 (обязательные для выхода в полноценный Production)

| Этап       | Цель                                                                    | Критерий приёмки этапа                                                                                                                  |
| ---------- | ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| **Этап 1** | Привести весь GUI к дизайн-системе и целевому UX/UI                     | Ключевые пользовательские потоки выглядят и работают консистентно (admin/manager/curator/student/company), итог согласован c Заказчиком |
| **Этап 2** | Подготовка инфраструктуры и миграция данных из AirTable и других систем | Есть реестр источников, данные мигрированы в согласованном объёме, сверка данных подтверждена                                           |
| **Этап 3** | Запуск следующего потока одного из продуктов на ZIS                     | Боевой поток проходит сквозной сценарий: лид → оплата → зачисление → обучение → операционная работа команды                             |
| **Этап 4** | Полный переход на платформу ZIS                                         | Согласованные продукты/потоки ведутся через ZIS как через основной рабочий контур                                                       |

Срок и бюджет фиксируются отдельно перед стартом каждого этапа.

---

## 6) Этапы 5+ (опции на выбор Заказчика)

После Этапов 1-4 можно выбрать любой трек развития:

1. **Bot & Communication** (`BOT-08`, `NOTIF-01`, развитие `BOT-01/02/04`).
2. **Billing & Finance** (`PAY-06/07/08`, развитие `INV-04/08`).
3. **Extensibility** (`ADMIN-04`, `LMS-06`, `LEAD-01/03`).
4. **Data/Audit/Ops** (`AUDIT-02`, `OPS-01/02`, `DEDUP-01/02`).

## 7) Ответы на главные вопросы Заказчика

- **«Как принимаем?»** Через заранее согласованный чеклист этапа.
- **«Как контролируем бюджет?»** Каждый этап имеет отдельный согласованный бюджет до старта.
- **«Как контролируем сроки?»** Каждый этап имеет отдельный согласованный срок до старта.
- **«Что не делаем в Прототипе?»** Явно вынесено списком (см. раздел 3).
- **«Что дальше после Прототипа?»** Есть 4 понятных трека этапов 5+ на выбор.
