## MKT-01: UTM-трекинг и атрибуция

### Важно: что такое «лид» для маркетинга

Лид — это **не только тот, кто купил**. Лид — это любой человек, чьи контакты мы получили:
- Записался на вебинар
- Подписался на бота
- Заполнил форму на лендинге
- Оставил заявку на разбор продукта
- Пришёл по реферальной ссылке

Он ещё не купил, но он **уже в нашей экосистеме**. Маркетинг должен видеть каждого такого человека и работать с ним: прогревать, сегментировать, вести к покупке.

### Проблема сейчас

Мы запускаем посевы в TG-каналах, делаем посты в Instagram, рассылки в боте, email — но не можем отследить, что именно привело к покупке. Нет единой системы, которая связывает источник трафика с конверсией. ZIS должен это решить.

### Что нужно: два слоя определения источника

#### Слой 1 — Автоматическое определение канала (по рефереру)

Система сама определяет **верхний уровень** — откуда пришёл человек, без участия маркетолога. Даже если забыли поставить UTM — ZIS всё равно поймёт канал.

| Referer содержит | ZIS определяет как |
|---|---|
| `google.com`, `yandex.ru` | Поисковый трафик |
| `youtube.com` | YouTube |
| `t.me`, `web.telegram.org` | Telegram |
| `instagram.com` | Instagram |
| `vk.com` | ВКонтакте |
| `mail.google.com`, `mail.yandex.ru` | Email |
| (пусто) | Прямой трафик |

Сохраняется в поле `channel` (авто) + `referrer_url` (полный URL).

#### Слой 2 — Ручная разметка (UTM-метки)

Маркетолог сама размечает ссылки для глубины: не просто "Telegram", а "Telegram → посев в канале ProductStar → пост про сарафан".

| Поле | Пример | Обязательность |
|---|---|---|
| `utm_source` | tg-channel-productstar, instagram, email | обязательно |
| `utm_medium` | post, story, bot_broadcast, email_campaign | обязательно |
| `utm_campaign` | conf-march-2026, boost-presale, webinar-120326 | обязательно |
| `utm_content` | hook-sarafan, hook-million-views | опционально |
| `utm_term` | (ключевое слово, если есть) | опционально |

#### Итого: что хранится по каждому лиду

```
Лид: Иванов Иван
├── Канал (авто):     Telegram          ← определён по рефереру
├── utm_source:       tg-channel-productstar  ← из UTM-метки
├── utm_medium:       post
├── utm_campaign:     conf-march-2026
├── utm_content:      hook-sarafan
├── referrer_url:     https://t.me/productstar/1234
└── Первое касание:   записался на вебинар 12.03
```

### Где это должно работать

1. **Лендинги** (zamesin.ru) — все формы и кнопки на лендинге передают UTM + реферер. Включая промежуточные страницы: если на лендинге кнопка «Забрать гайд в боте» ведёт на страницу с кнопкой «Перейти в Telegram» — UTM должен пробрасываться через всю цепочку и попасть в бот вместе с пользователем
2. **Telegram-бот** — при старте бота по ссылке `t.me/bot?start=...` параметры парсятся и сохраняются. Мы видим полный путь: откуда человек попал на лендинг → как перешёл в бот
3. **Публичный API** — внешние формы передают UTM как часть запроса

### Фильтрация в админке `/admin/leads`

Маркетолог должна мочь:

- **Фильтровать по каналу** (авто): «покажи всех, кто пришёл из Telegram» / «из YouTube» / «из email»
- **Фильтровать по кампании**: «покажи всех лидов с конференции марта 2026»
- **Фильтровать по источнику**: «покажи всех, кто пришёл из канала ProductStar»
- **Комбинировать фильтры**: «Telegram + конференция + март 2026»
- **Видеть в списке** колонки: Имя, Email, Канал, Кампания, Дата, Статус
- **В карточке лида** — полная информация: все UTM, реферер, первое касание, история действий

## MKT-02: Сегментация базы для рассылок

### Что нужно

**Готовые сегменты (предустановленные):**

| Сегмент | Логика | Для чего |
|---|---|---|
| Все лиды (не купили) | status ≠ converted, нет заказов | Прогревающие рассылки |
| Студенты КДП (активные) | enrollment.product = КДП, status = active | Контент по курсу, upsell BOOST |
| Выпускники КДП | enrollment.product = КДП, status = completed | Upsell BOOST, конфа, Makers |
| Студенты BOOST (активные) | enrollment.product = BOOST, status = active | Контент по курсу |
| Выпускники BOOST | enrollment.product = BOOST, status = completed | Upsell следующие продукты |
| Alumni (все выпускники) | любой enrollment.status = completed | Конфы, новые продукты, рефералка |
| Подписчики бота (без покупки) | есть в боте, нет заказов | Вовлечение, прогрев |
| Пришли с вебинара | utm_campaign = webinar-* | Follow-up цепочки |
| Пришли по рефералке | referred_by ≠ null | Специальные офферы |
| Остывшая база | нет активности 30+ дней (не читает, не кликает, не заходит) | Реактивация или исключение из рассылок (экономия ресурсов) |
| Были на диагностике | action = "диагностика", не купили | Follow-up, спец. оффер |
| Активные рефереры | привели ≥ 1 человека по реферальной ссылке | Благодарность, доп. мотивация, изучение профиля |

**Кастомные сегменты (конструктор):**

Маркетолог (я) должна мочь собрать сегмент из условий:
- Продукт: КДП / BOOST / AURA / любой
- Статус: лид / студент / выпускник
- Источник: utm_source, utm_campaign
- Время: зарегистрировался после/до даты
- Поведение в боте: прошёл контент-цепочку X, кликнул кнопку Y, не дочитал цепочку, сходил на вебинар, записался на диагностику
- Канал коммуникации: в боте / только email / оба / нет канала (потерянный контакт)
- Рефералка: пришёл по рефералке / сам реферер / привёл ≥ N человек
- Данные из анкет: ответы из входящих анкет и форм (роль, компания, цели, опыт) — чтобы можно было изучить, кто эти люди, и фильтровать по тому, что они сами о себе рассказали

## MKT-03: Массовые рассылки с аналитикой

### Что нужно

**Создание рассылки:**

1. Создать новую рассылку или скопировать предыдущую (состоящую из текста, фото, кнопок)
2. Выбрать сегмент (из MKT-02) — гибкая фильтрация: по продукту, статусу, активности, должности, данным из анкет и т.д. Работает одинаково и для бот-рассылок, и для email
3. Выбрать канал: Telegram-бот / email / оба
4. Написать сообщение — с переменными персонализации (`{имя}`, `{продукт}` и др.) и в боте, и в email
5. Запланировать время отправки или отправить сразу
6. Подтвердить → отправить

**Аналитика рассылки (после отправки):**

| Метрика | Описание |
|---|---|
| Отправлено | Количество получателей |
| Доставлено | Сколько дошло (для email — не bounced) |
| Открыто | Для email — open rate. Для бота — прочитано |
| Не открыло | Кто именно не открыл — для повторной рассылки |
| Кликнуло по ссылке | Click rate, с разбивкой по каждой ссылке в сообщении |
| Отписалось | Сколько отписалось после этой рассылки (сигнал качества контента/сегмента) |
| Конвертировалось | Купили после рассылки (в течение 7 дней) |

**Повторная рассылка «не открывшим»:**

- Одна кнопка: копирует рассылку, подставляет сегмент "не открывшие"
- Маркетолог меняет только тему/заголовок и отправляет
- Настраиваемая задержка (по умолчанию 2–3 дня после первой)

**Персонализация (переменные в сообщениях):**

В текст рассылки можно вставлять данные о получателе:
- `{имя}` — имя контакта
- `{продукт}` — название продукта (текущий или последний)
- `{промокод}` — персональный или общий промокод
- `{реферальная_ссылка}` — индивидуальная реферальная ссылка
- Любые другие поля из профиля и анкет — система знает про человека много, и всё это можно подставлять

**Email: поддержка HTML-вёрстки:**

- Возможность вставить готовый HTML-код письма (свёрстанного в нейросетях или дизайн-инструментах)
- Предпросмотр HTML перед отправкой

## MKT-04: Воронка конверсии с визуализацией

### Проблема сейчас

Данные разбросаны: лиды в одном месте, оплаты в другом, зачисления в третьем. Чтобы понять конверсию "от первого касания до покупки", нужно вручную сводить таблицы. Непонятно, сколько людей на каком этапе, сколько времени они там находятся, и где мы теряем больше всего.

### Что нужно: единая воронка (маркетинг + продажи)

**Полная воронка от первого касания до повторной покупки:**

```
МАРКЕТИНГОВАЯ ЧАСТЬ (до покупки):

Этап 1: Подписался / оставил контакт ──── [15 000] ──── 100%
         │
         ▼
Этап 2: Забрал контент / гайд ──── [4 200] ──── 28%
         │
         ▼
Этап 3: Был на вебинаре ──── [1 800] ──── 12%
         │
         ▼
Этап 4: Прошёл диагностику (Руслан) ──── [320] ──── 2.1%
         │
         ▼
─────────────────────────────────────────────────────
ПРОДАЖНАЯ ЧАСТЬ (после покупки):

Этап 5: Купил / отправил заявку ──── [180] ──── 1.2%
         │
         ▼
Этап 6: Зачислен, проходит курс ──── [170] ──── 1.1%
         │
         ▼
Этап 7: Завершил курс ──── [140] ──── 0.9%
         │
         ▼
Этап 8: Купил следующий продукт ──── [35] ──── 0.2%
```

### История контактов (карточка лида)

По каждому человеку нужно видеть полную историю взаимодействий:

```
Лид: Иванова Мария
├── 05.02 — Подписалась на бота (из поста в TG-канале ProductStar)
├── 12.02 — Забрала гайд по AJTBD
├── 01.03 — Зарегистрировалась на вебинар «Разбор продуктов»
├── 12.03 — Была на вебинаре
├── 14.03 — Открыла follow-up рассылку, кликнула на лендинг КДП
├── 18.03 — Записалась на диагностику к Руслану
├── 22.03 — Прошла диагностику
├── 25.03 — Купила КДП (поток 53)
├── 15.05 — Завершила КДП
└── 20.05 — Купила BOOST

Время от первого касания до покупки: 48 дней
Количество касаний до покупки: 7
```

### Ключевые метрики

| Метрика | Что показывает |
|---|---|
| **Время до покупки** | Сколько дней от первого касания до оплаты — в среднем и по каналам |
| **Количество касаний до покупки** | Сколько взаимодействий нужно, чтобы человек купил |
| **Конверсия по этапам** | Где теряем больше всего: подписался → вебинар? вебинар → диагностика? |
| **Объём на каждом этапе** | Сколько людей прямо сейчас на каждом этапе воронки |
| **Конверсия лид → оплата** | По каналу, кампании, типу входящего |
| **Repeat purchase rate** | Сколько выпускников покупают следующий продукт |
| Активные follow-up цепочки | Сколько лидов сейчас в прогреве |

### Мультитач-атрибуция

У нас путь до покупки — это **всегда несколько касаний**. Человек мог:
1. Увидеть пост в TG-канале → подписался на бота
2. Через неделю — получил рассылку → забрал гайд
3. Через 2 недели — увидел YouTube-ролик → зарегистрировался на вебинар
4. После вебинара — получил follow-up email → купил

Какой канал "привёл" эту покупку?

Нужно **записывать все касания** (вся цепочка), а не только первый или последний источник. Тогда можно анализировать:

- **Первый контакт** (first touch): откуда человек узнал о нас
- **Последний контакт** (last touch): что подтолкнуло к покупке
- **Вся цепочка**: какие каналы участвовали в конверсии

Это важно, потому что без мультитача каналы, которые приводят людей (например, TG-посевы), выглядят бесполезными — хотя на самом деле они критичны для верха воронки.

### LTV — сколько приносит один клиент за всё время

Человек может купить несколько продуктов: КДП → BOOST → Makers → следующий курс. Важно видеть:

| Метрика | Что показывает |
|---|---|
| **LTV по каналу** | Лиды из канала X приносят в среднем 150К за всё время, а из Y — только 80К |
| **LTV по сегменту** | Выпускники BOOST покупают в среднем ещё 1.5 продукта, а КДП — 0.8 |
| **LTV по реферальности** | Люди, пришедшие по рефералке, покупают больше / приводят больше друзей |
| **Топ-рефереры** | Кто из студентов привёл больше всего покупателей (и сколько денег это принесло) |

Это помогает понять: **куда вкладывать бюджет** — в канал с дешёвыми лидами или в канал с высоким LTV?

### Фильтры и выгрузка

- По продукту: КДП / BOOST / все
- По каналу: utm_source
- По кампании: utm_campaign
- По типу входящего: запрос на продукт / общее касание / вебинарный / реферал
- По периоду: неделя / месяц / квартал / произвольный
- **Выгрузка в CSV/Excel** — для анализа вне системы

## MKT-05: Реферальная программа в ZIS

### Флоу: как это должно работать

#### 1. Автогенерация реферального инструмента

Каждый студент, который начал обучение, автоматически получает **уникальный реферальный инструмент** — ссылку или промокод (или оба):

| Вариант | Формат | Плюсы |
|---|---|---|
| **Реферальная ссылка** | `zamesin.ru/ref/IVAN123` или `t.me/bot?start=ref_IVAN123` | Кликнул → попал в систему, всё автоматом |
| **Промокод** | `IVAN123` | Можно использовать где угодно — в разговоре, в посте, на лекции |

Оба генерируются **автоматически** в личном кабинете студента. Без участия менеджера.

#### 2. Студент делится

Студент видит в личном кабинете (и/или в боте):

```
Твоя реферальная ссылка: zamesin.ru/ref/IVAN123
Твой промокод: IVAN123

Что получит друг:
→ Скидка [X]% на любой курс

Что получишь ты за каждого купившего друга:
→ 1 месяц доступа к Makers бесплатно

Количество использований не ограничено.

Привёл друзей: 3
Из них купили: 1
Месяцев Makers накоплено: 1

[Скопировать ссылку] [Скопировать промокод]
```

#### 3. Друг покупает

```
Друг переходит по ссылке / вводит промокод при покупке
       │
       ▼
  Система автоматически:
  ├── Применяет скидку [X]% к заказу
  ├── Фиксирует связь: referred_by = ID реферера
  ├── После оплаты: referral_count у реферера +1
  ├── Начисляет рефереру 1 месяц Makers
  └── Отправляет уведомление рефереру: «Твой друг купил! Тебе +1 месяц Makers»
```

### Что нужно от ZIS

| # | Требование | Описание |
|---|---|---|
| 1 | **Автогенерация** | Уникальная ссылка + промокод генерируются автоматически для каждого студента |
| 2 | **В личном кабинете** | Студент видит свою ссылку, промокод, статистику рефералов |
| 3 | **В боте** | Кнопка «Пригласи друга» — показывает ссылку и прогресс |
| 4 | **Хранить `referred_by`** | У каждого пользователя — кто его привёл (ID реферера) |
| 5 | **Автоматический подсчёт** | Когда друг покупает → у реферера `referral_count` +1 |
| 6 | **Автоматическое начисление** | 1 месяц Makers за каждого купившего друга — без участия менеджера |
| 7 | **Автоматическая скидка** | Друг вводит промокод или переходит по ссылке → скидка применяется автоматически |
| 8 | **Уведомления** | Бот/email рефереру: «Твой друг купил, тебе начислен бонус» |
| 9 | **Без ограничений** | Количество рефералов не ограничено — чем больше, тем лучше |

### Бонусная структура

| Кто | Что получает | Значение |
|---|---|---|
| **Друг** (новый покупатель) | Скидка на первый курс | **???%** — согласовать с Ваней |
| **Реферер** (студент) | Доступ к Makers за каждого купившего друга | **1 месяц** (предложение Вани) |

> Открытый вопрос: конкретный % скидки, потолок месяцев Makers, на какие продукты распространяется.

### В AdminUI (для менеджера/маркетолога)

| Что видно | Зачем |
|---|---|
| Список всех рефереров | Кто активно делится |
| Топ-рефереры | Кого поощрить, кого вовлечь ещё больше |
| Конверсия рефералов | Сколько перешли по ссылке → сколько купили |
| ROI программы | Сколько потратили на бонусы (месяцы Makers) vs сколько заработали |
| Связи реферер → друзья | Кто кого привёл |

### Настраиваемые условия

Маркетолог/менеджер должен мочь **настраивать условия** реферальной программы без разработчика:

- Размер скидки для друга (5%, 10%, 15%)
- Бонус рефереру (1 месяц Makers, 2 месяца, скидка на курс)
- Пороги: за 1 друга — X, за 5 друзей — Y
- На какие продукты распространяется (все / только КДП+BOOST / конкретный)
- Включить/выключить программу

## MKT-07: Раздел «Маркетинг» в AdminUI

### Проблема сейчас

### Главный дашборд: рост аудитории и источники

Это первое, что маркетолог видит при входе. Стратегический фокус — **awareness**: сколько новых людей маркетинг привёл в экосистему.

**Блок 1: Новые люди за период (неделя / месяц / квартал)**

| Метрика | Что показывает |
|---|---|
| **Новые лиды** | Сколько людей впервые стали частью экосистемы (подписались на бот, оставили контакт, зарегистрировались на вебинар) — не купили, а просто появились |
| **Новые клиенты** | Сколько из новых (или старых) лидов впервые купили |
| **Конверсия лид → клиент** | % тех, кто из "познакомились" перешёл в "купили" |
| **Динамика** | График: новые лиды и клиенты по неделям/месяцам — растём или нет? |

**Блок 2: Источники — откуда приходят (таблица с сортировкой)**

| Канал | Новые лиды | Новые клиенты | Конверсия лид→клиент | Выручка |
|---|---|---|---|---|
| Telegram (посевы) | — | — | — | — |
| YouTube | — | — | — | — |
| Instagram | — | — | — | — |
| Бот (органика) | — | — | — | — |
| Email-рассылка | — | — | — | — |
| Рефералы | — | — | — | — |
| Другое / неизвестно | — | — | — | — |

Сортировка по любому столбцу. Ключевой вопрос: какой канал реально работает — тот, что приводит много людей, или тот, что приводит покупающих?

**Блок 3: Тренды аудитории**

Ключевая потребность: **видеть, куда движется аудитория** — растёт ядро или уходит.

Примеры трендов, которые нужно отслеживать:
- **Рост ядра** — горячие лиды, рефереры, повторные покупатели: их становится больше или меньше?
- **Отток** — сколько людей уходят в "остывшие" за период, из каких каналов уходят больше всего
- **Приток по каналам** — какой канал растёт, какой падает (Telegram набирает, YouTube упал)
- **Активность** — среднее количество взаимодействий на человека: люди вовлекаются глубже или поверхностнее?
- **Реферальный рост** — ядро рефереров растёт? Сколько новых рефереров за период?

Формат: графики с линиями по неделям/месяцам, чтобы схватывать динамику визуально.

> **Важно: весь маркетинговый раздел проектируется для вайб-кодинга.** Маркетолог (или помогающий разработчик) должна мочь быстро добавлять новые виджеты, графики, тренды и срезы данных через вайб-кодинг — без полноценной разработки. Данные уже в системе, нужна возможность гибко их визуализировать по запросу.

## MKT-08: Маркетинговый бот (Telegram)

### Проблема сейчас

В ZIS бот описан **только для студентов** (BOT-01..09): уведомления, прогресс, онбординг. Но у нас 15 000 подписчиков в боте, и большинство — **не студенты**, а люди на разных стадиях прогрева. Бот — это главный маркетинговый инструмент для работы с этой аудиторией. Сейчас это делается в BotHelp, но ему не хватает гибкости (нельзя отправить галерею картинок + текст + кнопки, нет удобной сегментации, нельзя вайб-кодить новые фичи).

### Бот как точка входа — UTM-трекинг

Бот — одна из главных точек входа новых лидов. При подписке на бот нужно:

- Считывать UTM из start-ссылки (`t.me/bot?start=...`)
- Определять, откуда пришёл человек (посев, пост, реклама, рефералка)
- Сохранять эту информацию в профиле лида в ZIS

Пример: `t.me/bot?start=utm_source=tg-productstar_campaign=conf-march` → в ZIS лид с пометкой «пришёл из канала ProductStar, кампания конференция».

### Действия в боте, которые нужно трекать

Каждое действие пользователя в боте добавляет информацию к его профилю и автоматически включает его в соответствующий список/сегмент:

| Действие в боте | Что сохраняется | В какой список попадает |
|---|---|---|
| Подписался на бота | дата, источник (UTM) | Все подписчики |
| Зарегистрировался на вебинар | какой вебинар, дата | Регистранты вебинара X |
| Забрал гайд | какой гайд | Получили гайд Y |
| Подписался на waiting list | на что, дата | Waiting list BOOST / КДП / ... |
| Записался на консультацию к Руслану | дата | Записались на диагностику |
| Кликнул «Посмотреть курс» | какой курс | Интересуются продуктом Z |
| Купил (по ссылке из бота) | что, когда | Покупатели |

**Принцип:** человек нажал кнопку → к нему автоматически добавилась информация → он попал в список. Маркетолог потом может фильтровать и рассылать по этим спискам.

### Списки и сегменты в боте

Действия в боте автоматически формируют списки (см. таблицу выше). Эти списки интегрируются с общей сегментацией (MKT-02) — можно комбинировать бот-списки с любыми другими фильтрами и делать рассылки по результату.

Референс: как это сделано в **SaleBot** — удобная работа со списками и сегментами.

### Контентные цепочки (drip-кампании)

Автоматические серии сообщений по триггеру:

**Пример: регистрация на вебинар**
```
Человек зарегистрировался на вебинар
       │
       ▼  сразу
  Сообщение 1: «Ты зарегистрирован! Вот ссылка, вот дата»
       │
       ▼  через 1 час
  Сообщение 2: «Кстати, пока ждёшь — забери гайд по AJTBD»
       │
       ▼  за 1 день до вебинара
  Сообщение 3: «Напоминание: завтра в 19:00!»
       │
       ▼  через 1 день после вебинара
  Сообщение 4: «Запись + саммари + CTA на КДП»
```

**Пример: незавершённое действие**
```
Человек начал регистрацию, но не отправил телефон
       │
       ▼  через 2 часа
  Напоминание: «Кажется, ты не завершил регистрацию. Остался один шаг!»
       │
       ▼  через 24 часа (если всё ещё не завершил)
  Второе напоминание: «Места заканчиваются — завершить регистрацию?»
```

### Запоминание данных пользователя

Если человек уже вводил email и телефон (например, регистрировался на предыдущий вебинар), **не нужно спрашивать повторно**. Система должна подставлять известные данные автоматически.

> Не самая критичная фича, но сильно улучшает опыт для повторных пользователей.

### Форматы сообщений

Что нужно уметь отправлять одним сообщением:

| Формат | Описание |
|---|---|
| Текст + кнопки | Базовый формат — есть сейчас |
| Изображение + текст + кнопки | Ключевой формат для маркетинга |
| **Галерея** изображений + текст + кнопки | Нет в BotHelp, нужно в ZIS |
| Видео + текст + кнопки | Для YouTube Shorts, анонсы |

### Интерфейс для команды

Бот — это основной рабочий инструмент маркетинга. Нужен **удобный интерфейс** (не CLI, не код), в котором маркетолог может:

- Создавать и редактировать сообщения (текст + медиа + кнопки)
- Настраивать цепочки (триггер → сообщение 1 → задержка → сообщение 2)
- Видеть списки подписчиков и управлять ими
- Запускать и планировать рассылки
- Видеть аналитику: доставлено, открыто, кликнуто

> Идеально — интерфейс, который можно дорабатывать через вайб-кодинг (добавлять новые типы сообщений, новые триггеры).

## MKT-09: Маркетинговая платформа — умный контент-план

### Формат разработки

Весь MKT-09 буду вайбкодить сама. Задача ZIS — дать мне данные. Ниже описываю, какие данные нужны, чтобы их можно было запрашивать через API.

### Что нужно

**1. Контент-план внутри платформы (вторая вкладка в маркетинг-кабинете)**

Табличный вид, как привычный контент-план, но внутри ZIS:

| Дата | Тип | Где | Что | Статус | Результат |
|---|---|---|---|---|---|
| 20.03 | Анонс вебинара | TG-канал | Пост + картинка | готово | 2.1K просмотров |
| 20.03 | Анонс вебинара | Бот (рассылка) | Текст + кнопка | готово | 89% доставлено, 34% кликнули |
| 22.03 | Email-рассылка | Email | HTML-письмо | запланировано | — |

**2. Автоподтягивание запусков и событий**

Когда в системе создаётся новый поток курса или вебинар — он **автоматически появляется в контент-плане** с набором задач по продвижению по заданным маркетологом правилам.

Пример: создали вебинар на 26 марта → система автоматически ставит:
- за 14 дней: анонс в TG-канале + рассылка по боту
- за 7 дней: второй анонс + email-рассылка
- за 1 день: напоминание зарегистрированным
- через 1 день: пост с записью + follow-up рассылка

Пример: создали поток КДП-54 на 14 апреля → автоматически:
- за 30 дней: открыть набор + пригласить 5 инфлюенсеров
- за 21 день: рассылка по боту (были на вебинарах, не купили)
- за 14 дней: кейс выпускника
- за 7/3 дня: «осталось N мест»

Правила настраиваются **один раз**, дальше генерируются для каждого нового запуска.

**3. Повторяющиеся задачи, которые не нужно помнить**

- **Пригласить 5 инфлюенсеров** — автоматическое напоминание на каждый поток
- **Выступления Вани** — если запланировано, попадает в контент-план (до: анонс, после: пост с итогами)
- Любые повторяющиеся активности, которые маркетолог задаст

**4. Аналитика прямо в контент-плане**

Результат подтягивается автоматически, потому что рассылки, бот, события — всё внутри платформы:
- Рассылки: доставлено, открыто, кликнуло, конвертировалось
- События: зарегистрировалось → пришло → купило
- TG-канал Вани (если интеграция возможна): просмотры, репосты, какой пост привёл подписчиков

**5. Ежемесячный маркетинговый отчёт**

Автоматическая сводка за месяц: новые лиды, новые клиенты, общий охват, топ-каналы, результаты запусков и рассылок, рефералы. Можно скачать или отправить в чат команды.

> Подробный бриф: [`content-platform-aura-brief.md`](../../4-Personal-files/Vladlena-Orlova/content-platform-aura-brief.md)

> **Вайб-кодинг**: вся платформа проектируется для быстрой доработки — добавить тип активности, изменить правила, подключить источник аналитики, настроить шаблон отчёта.
