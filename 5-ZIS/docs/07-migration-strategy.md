# Стратегия миграции с AirTable

> Часть архитектуры бэк-офиса Zamesin Information System.
> Основной документ: [PLAN.md](../PLAN.md)

---

## Подход: однократный импорт + переход

Вместо двусторонней синхронизации (которая создаёт сложность с конфликтами, soft-lock полей, reconciliation) — делаем чистый переход: импортируем данные один раз и переключаем команду на бэк-офис.

При команде из 15 человек это реально сделать за 1-2 дня.

---

## Шаг 1: Подготовка (до импорта)

- Получить доступ к AirTable API и изучить текущие таблицы
- Сопоставить поля AirTable → схему PostgreSQL (маппинг)
- Написать скрипт импорта (Node.js, использует Prisma)
- Определить "замороженную" дату: после неё данные вводятся только в бэк-офис

## Шаг 2: Dry-run импорт

```
Скрипт в режиме dry-run (без записи в БД):

1) Читает все записи из AirTable API
2) Для каждой записи:
   - Нормализация: email → lowercase + trim
   - Дедупликация: hash(email + phone) → dedup_key
   - Валидация обязательных полей
   - Маппинг на Prisma-модели
3) Отчёт:
   ├── N записей — готовы к импорту
   ├── M записей — пустые обязательные поля → пометка needs_review
   ├── K записей — дубликаты → будут объединены
   └── L записей — невалидный email → сохранить + флаг email_invalid
```

## Шаг 3: Импорт

```
Скрипт в режиме execute:

1) Создать записи в PostgreSQL в правильном порядке (FK):
   Users → Organizations → Products → Tiers → Cohorts → Enrollments → Payments
2) Для записей с needs_review: создать, но пометить
3) Лог: каждая созданная запись + AirTable record ID (для сверки)
4) Итоговый отчёт: сколько создано, сколько ошибок
```

## Шаг 4: Верификация

```
1) Сравнить количество записей: AirTable vs PostgreSQL
2) Spot-check: 10-20 случайных записей — сверить поля вручную
3) Проверить needs_review записи — исправить вручную в бэк-офисе
4) Попросить менеджеров проверить "свои" данные в бэк-офисе
```

## Шаг 5: Переход

```
1) Команда начинает работать в бэк-офисе
2) AirTable остаётся доступной read-only для справки
3) Через 2-4 недели, если нет обращений к AirTable → архивировать
```

---

## Что НЕ делаем

- **Двусторонняя синхронизация** с AirTable — не реализуется
- **Soft-lock полей** в AirTable — не нужен при полном переходе
- **Фаза "переключения"** с read-only AirTable sync — не нужна
- **Reconciliation** между двумя БД — не нужна

Если позже понадобится экспорт данных обратно в AirTable (например, для отчётов) — это одноразовый скрипт, не постоянная синхронизация.

## Миграция AURA Framework

AURA Framework — отдельное веб-приложение с собственной БД подписчиков. Остаётся отдельным приложением (за рамками Zamesin IS), но пользователи и подписки синхронизируются.

### Цель

Связать пользователей AURA с User в Zamesin IS для:

- Единого профиля (студент, купивший КДП, видит свою подписку AURA)
- Кросс-продуктовых бонусов (покупка КДП → бонусная подписка Maker)
- Единой аналитики (LTV с учётом подписок)

### Маппинг данных

```
AURA БД                          →  Zamesin IS
─────────────                        ──────────
aura_user.email                  →  User.email (match по email)
aura_user.name                   →  User.name (если в ZIS пусто)
aura_user.telegram               →  User.telegram_user_id / telegram_username
aura_subscription.plan           →  Subscription.plan (maker | founder)
aura_subscription.status         →  Subscription.status
aura_subscription.started_at     →  Subscription.started_at
aura_subscription.next_payment   →  Subscription.next_payment_date
aura_subscription.expires_at     →  Subscription.expires_at
aura_payment_method              →  Subscription.payment_method
```

### Стратегия

```
Вариант A: Однократный импорт (аналогично AirTable)
  1. Экспорт пользователей и подписок из AURA БД
  2. Match по email → найти/создать User в ZIS
  3. Создать Subscription для каждой активной подписки
  4. AURA продолжает работать автономно
  5. Проблема: подписки рассинхронизируются

Вариант B: Периодическая синхронизация (рекомендуется)
  1. Однократный импорт (как вариант A)
  2. Cron job (pg-boss, ежедневно): запрос к AURA API/БД
     → обновить Subscription.status, next_payment_date
  3. Направление: AURA → ZIS (read-only, ZIS не меняет данные AURA)
  4. При рассогласовании → лог + алерт менеджеру

⚠️ Уточнить у заказчика:
  - Есть ли API у AURA или нужен прямой доступ к БД?
  - Структура таблиц AURA (получить schema dump)
  - Сколько активных подписчиков? (~сотни?)
  - Рекуррентные платежи AURA идут через ЮKassa?
```

### Шаги импорта

```
1) Получить доступ к AURA БД / API
2) Изучить структуру данных (таблицы, поля, связи)
3) Написать маппинг-скрипт (Node.js + Prisma)
4) Dry-run: проверка совпадений по email
5) Импорт: создание User (если нет) + Subscription
6) Верификация: сравнить количество, spot-check
7) Настроить cron-синхронизацию (если вариант B)
```

### Этап реализации

- **Этап 1**: однократный импорт пользователей AURA → User (match по email)
- **Этап 3**: полная синхронизация подписок (Subscription), кросс-продуктовые бонусы
