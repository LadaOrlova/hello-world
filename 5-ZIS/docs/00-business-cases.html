1. Единые пользователи и авторизация AUTH
Что болит: у каждого продукта (Boost, «Как делать продукт», Makers, Aura) отдельная база пользователей и отдельный вход. Один и тот же человек — разные сущности в разных системах.
✓
AUTH-01. Пользователь регистрируется один раз и получает единый аккаунт, действующий во всех продуктах экосистемы.
Реализовано в ZIS-007 (WP-007): better-auth + Email OTP, единая таблица User, 4 роли.
✓
AUTH-02. Пользователь входит в любой продукт через единую точку авторизации — отдельная регистрация для каждого продукта не нужна.
Реализовано в ZIS-007: единый auth endpoint /api/auth/_, OTP-вход по email.\*
●
AUTH-03. Команда видит полную историю пользователя в одном месте: какие продукты покупал, в каких потоках учился, когда и по какой цене.
Частично реализовано: единая БД с Order/Enrollment/Payment связями через User. Admin UI для users, billing, enrollments работает (ZIS-008..ZIS-011). Единой «карточки пользователя» с агрегированной историей пока нет — данные доступны через отдельные разделы админки. → DRAFT-034
●
AUTH-04. Если пользователь ранее был зарегистрирован в одном продукте и приобретает доступ к другому — его аккаунт автоматически объединяется, а не создаётся заново.
Архитектурно заложено (email = единый идентификатор, дедупликация лидов в CRM — ZIS-013), но сценарий merge аккаунтов при миграции из старых систем требует реализации скрипта импорта (DRAFT-008: AirTable Import).
2. Единый личный кабинет пользователя LK
Что болит: студенты пишут, что неудобно отдельно авторизоваться в каждом продукте; они не видят полный список своих покупок.
●
LK-01. Студент заходит в один личный кабинет и видит все свои активные доступы (курсы, подписки, сообщества).
Архитектура поддерживает (единая БД Enrollment/Subscription), но отдельного «личного кабинета студента» с агрегированным списком доступов нет. Сейчас студент видит конкретный курс через /student/lms/[enrollmentId] (ZIS-012). → DRAFT-033
●
LK-02. В личном кабинете отображаются продукты, которые студент ещё не купил, с возможностью сразу перейти к покупке.
Не реализовано. Витрина продуктов входит в scope ЛК студента. → DRAFT-033
✕
LK-03. Покупка нового продукта не требует повторной регистрации или ввода платёжных данных, если они уже сохранены.
Деприоритизировано Иваном (25.02.2026): «Давайте не будем это делать.» Повторная регистрация не нужна (AUTH-01/02 реализованы). Сохранение платёжных данных — избыточная сложность на текущем этапе.
3. Управление продуктами и курсами COURSE
Что болит: запуск каждого нового курса — отдельная кодовая база, отдельная база данных, отдельная админка. Разворачивание нового продукта занимает двое суток вместо нескольких часов.
●
COURSE-01. Любой член команды может запустить новый курс (лендинг + оплата + онбординг) в рамках единой системы, не создавая отдельной инфраструктуры.
Частично реализовано: Products/Tiers/Cohorts CRUD (ZIS-009), Billing (ZIS-010), Enrollments + Onboarding (ZIS-011). Лендинги пока не реализованы — курс создаётся через админку, но публичного лендинга для каждого продукта нет. → DRAFT-035
●
COURSE-02. Новый курс автоматически появляется в единой админке и в личных кабинетах пользователей.
Частично реализовано: в админке — да (ZIS-009: /admin/products). В «личном кабинете» студента — нет (см. LK-01). → DRAFT-033
●
COURSE-03. Команда может управлять структурой курса из админки: добавлять, удалять, редактировать материалы; публиковать и снимать с публикации блоки лендинга.
Частично реализовано: управление структурой курса работает — модули/уроки CRUD, reorder, deactivate/reactivate (ZIS-012: /admin/lms). Управление лендингом не реализовано. → DRAFT-035
✕
COURSE-04. Курс может иметь разные модели монетизации: разовая покупка, поток с тарифами, подписка — без необходимости переписывать инфраструктуру под каждую модель.
Деприоритизировано Иваном (25.02.2026): «Давайте не будем это делать, это довольно сложная штука.» Базовое: Product.type = course | subscription, тарифы и потоки работают (ZIS-009). Подписочная модель отложена на неопределённый срок.
✓
COURSE-05. Команда может добавить к существующему курсу новый тариф или новый поток, не затрагивая доступы уже зачисленных студентов.
Реализовано в ZIS-009: добавление Tier/Cohort через админку. Существующие Enrollment не затрагиваются.
4. Оплата и биллинг PAY
Что болит: способы оплаты (Stripe, Яндекс.Сплит) живут как отдельные интеграции в каждом продукте; апгрейды доступа при связанных покупках выполняются вручную.
✓
PAY-01. При оплате система фиксирует снапшот: цену, валюту и полное описание тарифа на момент покупки. Изменение актуальных цен не влияет на исторические записи.
Реализовано в ZIS-010/ZIS-011: Enrollment.priceSnapshot фиксирует цену, валюту и тариф на момент покупки. Order хранит amount/currency.
●
PAY-02. Если покупка одного продукта даёт право на доступ к другому (например, курс включает в себя членство в сообществе), система выдаёт эти доступы автоматически без ручных действий команды.
Заложено в плане: DRAFT-029 (Cross-product Bonuses, Фаза 3) — автоматическая подписка Maker при покупке КДП/BOOST.
✓
PAY-03. Менеджер видит в единой админке все транзакции пользователя вне зависимости от того, через какой продукт была совершена оплата.
Реализовано в ZIS-010: /admin/billing — список заказов с платежами по всем продуктам.
●
PAY-04. Настройка нового способа оплаты для нового продукта производится через интерфейс без необходимости привлекать разработчика для ручного изменения конфигурации.
Не реализовано. Сейчас ЮKassa — единственный провайдер, настраивается через env-переменные. Stripe запланирован как будущая фаза (п. 45 ROADMAP). → DRAFT-042
✕
PAY-05. Кассовые чеки формируются и отправляются автоматически по всем платёжным провайдерам, включая ручные ссылки на оплату (T-Bank, Яндекс Сплит), а не только по ЮKassa и Stripe.
Не реализовано. Сейчас ЮKassa и Stripe автоматически отправляют чеки. При ручном выставлении ссылок через T-Bank/Split чек приходится выбивать вручную через кассу, что создаёт риск ошибок и необходимость чеков коррекции.
✕
PAY-06. Менеджер может инициировать возврат средств из админки: система фиксирует возврат во внутренних данных (заказ, зачисление, отчётность), независимо от того, через какой канал фактически проводится возврат в платёжной системе.
Не реализовано. Сейчас возврат — полностью ручной процесс: поиск карточки в CRM, поиск платежа в платёжной системе, возврат через интерфейс провайдера (ЮKassa, Stripe, Split) или банковскую платёжку (T-Bank). Данные вносятся вручную.
✕
PAY-07. Студент может апгрейднуть тариф (например, со «Стандарт» на «Премиум»): система автоматически рассчитывает разницу, формирует ссылку на доплату и после оплаты обновляет доступы.
Не реализовано. Сейчас апгрейд — ручной процесс: менеджер формирует отдельную ссылку на оплату с суммой разницы, после доплаты вручную меняет тариф в CRM.
✕
PAY-08. Цены в иностранных валютах (USD и др.) автоматически пересчитываются по актуальному курсу ЦБ РФ, без необходимости ручного обновления.
Не реализовано. Сейчас Настя каждую среду вручную обновляет долларовые цены в админке. Базовая валюта — рубли. Нужен cron-процесс синхронизации курсов.
5. Лендинги и формы сбора лидов LEAD
Что болит: подключение лендинга к системе сбора данных (например, отклики по вакансиям, заявки на курс) занимает много времени из-за ручного маппинга полей.
●
LEAD-01. Любой член команды может создать новый лендинг (курс, вакансия, акция) и подключить его к единой системе сбора данных без длительного технического согласования. Лендинги должны создаваться в промышленных масштабах, а изменения в них — вноситься быстро и легко.
⚡ Приоритет Ивана (25.02.2026): «Очень важно, чтобы мы могли лендинги клепать в промышленных масштабах и вносить изменения в промышленных масштабах.»
Частично реализовано: TypeForm webhook + внутренняя форма /lead + публичный API /api/leads (ZIS-013). Создание самих лендингов вне ZIS, но подключение к системе — через стандартные webhook'и. Самостоятельное подключение нового лендинга командой без разработчика пока невозможно. → DRAFT-035
✓
LEAD-02. Лид, пришедший с любого лендинга, автоматически попадает в единую базу и отображается в соответствующем разделе админки.
Реализовано в ZIS-013: лиды из TypeForm/внутренней формы попадают в единую БД, отображаются в /admin/leads с фильтрами и статусами. Автоматический auto-assign менеджера.
●
LEAD-03. Лендинг вакансий: отклик на вакансию сохраняется в системе и отображается в разделе «Вакансии» единой админки с возможностью управлять статусами кандидатов.
⚡ Уточнение Ивана (25.02.2026): «Нужно, чтобы админка вакансий, которую сделала Алёна, интегрировалась в единую админку.» Алёна Тушминцева уже создала страницу вакансий с отдельной админкой — её нужно встроить в ZIS Admin UI. → DRAFT-043
6. LMS (Learning Management System) LMS
Что болит: текущая LMS неудобна и для студентов, и для команды; запуск нового потока требует значительных усилий; администрирование контента завязано на разработчика.
✓
LMS-01. После покупки курса студент автоматически получает доступ в LMS — отдельная регистрация не нужна.
Реализовано в ZIS-011 + ZIS-012: после оплаты запускается onboarding process, шаг grant_lms выдаёт доступ автоматически. Студент заходит через /student/lms/[enrollmentId].
●
LMS-02. Студент видит свой прогресс по курсу и может добавлять материалы в избранное и делать личные заметки.
Частично реализовано: прогресс по урокам (отметка «Изучено») работает (ZIS-012). Избранное и личные заметки не реализованы. → DRAFT-036
●
LMS-03. Студент видит групповой прогресс (например, статистику своей учебной группы) для элементов соревновательной механики.
Заложено частично: куратор видит матрицу прогресса группы (ZIS-012). Для студента аналогичного представления нет. → DRAFT-037
✓
LMS-04. Куратор видит матрицу прогресса своих студентов: кто что просмотрел, что ещё не начал.
Реализовано в ZIS-012: /curator/lms/[cohortId] — таблица «студент × урок».
✓
LMS-05. Команда может добавлять, редактировать, удалять материалы курса из административного интерфейса без привлечения разработчика.
Реализовано в ZIS-012: /admin/lms — CRUD модулей/уроков, reorder, revision history, deactivate/reactivate.
●
LMS-06. Команда может добавить в LMS новую функциональную страницу (например, страницу группы со статистикой) и опубликовать её самостоятельно — через вайб-кодинг, без необходимости ждать разработчика.
⚡ Уточнение Ивана (25.02.2026): «Нужно, чтобы можно было вайб-кодить условно что угодно, что не противоречит данным и расширяет функциональность. Нужно, чтобы здесь позволялось такое расширение.» Не конструктор страниц, а расширяемая архитектура для быстрой разработки новых страниц. → DRAFT-044
✓
LMS-07. Онбординг студента в новый поток (добавление в чаты, предоставление доступов) происходит автоматически после подтверждения оплаты.
Реализовано в ZIS-011 + ZIS-014: полный onboarding process из 5 шагов (grant_lms, create_onboarding_page, subscribe_calendar, assign_curator, send_welcome), запускается автоматически после payment.success webhook.
📌 Замечание Ивана (25.02.2026): «LMS мы сделаем вообще отдельным большим проектом. Будет много сценариев приёмки конкретно по новой LMS.» Текущая LMS в ZIS — базовая версия; полноценная LMS может стать отдельным проектом с собственным циклом разработки.
7. Единая административная панель ADMIN
Что болит: каждый продукт — своя админка; разным ролям нужны разные срезы данных, но сейчас это невозможно настроить без лишней сложности.
●
ADMIN-01. Команда работает в одной AdminUI, внутри которой видны все модули: пользователи, курсы, оплата, вакансии, LMS, маркетинг. Все существующие отдельные админки (Boost, вакансии Алёны, лендинг КДП) интегрируются в единую систему.
⚡ Топ-приоритет Ивана (25.02.2026): «Нам нужна единая админка на все наши продукты и на все наши внутренние штуки. Админку Boost тоже нужно интегрировать. Админку вакансий Алёны тоже нужно интегрировать.»
Частично реализовано: единая админка /admin/* с sidebar-навигацией: users, companies, products, billing, enrollments, lms, leads, events, dashboard (ZIS-008..ZIS-015). Модули вакансий, маркетинга, Boost-админки не интегрированы.
●
ADMIN-02. Каждая роль (продажи, контент, финансы, кураторы) видит только релевантный для неё срез данных и инструменты.
Частично реализовано: RBAC middleware requireRole() ограничивает доступ по ролям (ZIS-003). Куратор видит только свои данные. Однако гранулярного разделения UI по ролям (разные sidebar-секции для разных ролей) пока нет — все admin/manager видят одинаковый интерфейс. → DRAFT-038
●
ADMIN-03. Сотрудник без технической экспертизы может выполнить типовую операцию (добавить запись, изменить статус, выгрузить данные) без помощи разработчика.
Частично реализовано: через Admin UI можно создавать пользователей, лидов, продукты, заказы, зачисления, управлять контентом LMS. Выгрузка данных (export) не реализована. → DRAFT-039
●
ADMIN-04. Члены команды могут самостоятельно «вайб-кодить» новые разделы и инструменты внутри админки — добавлять новые сущности, лендинги, логику — и всё это быстро и легко интегрируется в единую систему.
⚡ Приоритет Ивана (25.02.2026): «Не no-code. Ребята могут вайб-кодить в этой админке всё, что им душе угодно. Это очень важная штука автоматизации. Нужно, чтобы всё нормально, быстро, легко подключалось.»
Не реализовано. Требуется расширяемая plugin-архитектура или модульная система, позволяющая командным разработчикам быстро добавлять новые разделы в единую админку. → DRAFT-045
8. Автоматизация и боты BOT
Что болит: кураторы вручную отвечают на типовые вопросы студентов в чатах потоков; уведомления о новых лидах и оплатах не централизованы.
●
BOT-01. Команда получает автоматическое уведомление о новом лиде или совершённой оплате независимо от того, с какого продукта пришёл пользователь.
Частично реализовано: при onboarding процессе ставится TG-уведомление в служебный чат (send_team_message — ZIS-014). Уведомления о новых лидах в реальном времени пока нет — требует доработки CRM-модуля. → DRAFT-021
●
BOT-02. Студент может узнать информацию о своих доступах, статусе оплаты и прогрессе через единый бот, не обращаясь к куратору.
Заложено в плане: DRAFT-020 (TG: Student Bot, Фаза 2). Сейчас бот поддерживает только /start для привязки аккаунта (ZIS-014).
✓
BOT-03. Боты потоков и продуктов не множатся пропорционально числу потоков: один бот обслуживает все потоки продукта, адаптируя ответы к контексту.
Архитектурно реализовано в ZIS-014: один бот (grammY) обслуживает все потоки. TelegramChat привязан к Cohort/Product, ChatMembership отслеживается централизованно.
●
BOT-04. Команда может расширить сценарии бота (добавить новые команды, ответы) без переработки всей бот-инфраструктуры.
