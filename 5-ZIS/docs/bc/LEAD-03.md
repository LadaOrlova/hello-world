# LEAD-03: Внешний HR-портал (или лендинг вакансий) может отправлять отклики кандидатов в ZIS как в API-сервис, а команда управляет воронкой найма через единый слой данных и статусов

**Статус в прототипе:** Частично

## Параметры реализации

### Критерии приёмки

- [ ] Реализована авторизация внешнего HR-портала через API-ключ, созданный авторизованным пользователем ZIS (например, менеджером), с действиями от его имени.
- [ ] Подготовлен onboarding-гайд: как внешняя HR-команда подключает свой портал к ZIS за 1 рабочий день.
- [ ] Опционально (если успеваем в прототипе): добавлен source/mapping для кандидатов `externalSystem`, `externalCandidateId`, `vacancyExternalId`.
- [ ] Не требуется переносить внешний HR-портал в кодовую базу ZIS.
- [ ] Не требуется единая БД между ZIS и HR-приложением.
- [ ] Не требуется встраивать весь HR UI внутрь админки ZIS.

## Что работает в прототипе

Базовые строительные блоки уже есть и подходят для внешней HR-интеграции:

1. Публичные точки приёма заявок (API/webhook) уже используются для лидов
2. В админке есть зрелый паттерн обработки входящих заявок (списки, статусы, фильтры)
3. Better-auth и bearer-токены можно использовать для service-to-service доступа
4. Есть инфраструктура асинхронной обработки событий (pg-boss)

## Что нужно доработать в прототипе

Чтобы LEAD-03 соответствовал концепции «вайб-кодинг во внешних приложениях + ZIS как API-ядро», нужно:

- [ ] Реализовать service-to-service авторизацию для HR-порталов и лендингов вакансий
- [ ] Подготовить onboarding-гайд: как внешняя HR-команда подключает свой портал к ZIS за 1 рабочий день
- [ ] Опционально: Добавить source/mapping для кандидатов: `externalSystem`, `externalCandidateId`, `vacancyExternalId`

### Что не требуется по этой концепции

- [ ] Не требуется переносить внешний HR-портал в кодовую базу ZIS
- [ ] Не требуется единая БД между ZIS и HR-приложением
- [ ] Не требуется встраивать весь HR UI внутрь админки ZIS

## Что может быть на следующих этапах

### Интеграции

- HeadHunter/LinkedIn/другие площадки подключаются к внешнему HR-порталу, а в ZIS приходят нормализованные события
- Двусторонняя синхронизация статусов: обновление в HR-портале отражается в ZIS, и наоборот через API
- Унифицированные адаптеры для «вайб-код» мини-приложений в HR-контуре

### Пользовательский опыт

- Единый список кандидатов в ZIS с фильтрами по источнику, вакансии и стадии
- Карточка кандидата с историей изменений статуса и источниками данных
- Уведомления рекрутеру/менеджеру при смене критичных статусов

### Расширенные сценарии

- SLA по воронке найма: время отклика, время до интервью, время закрытия вакансии
- Рейтинг источников кандидатов по конверсии в найм
- Версионирование HR API-контракта для независимых релизов внешних команд
