## Параметры реализации

### MVP: проектирование сущностей и базового пайплайна

> **Решение (2026-03-17):** На запуске BOOST домашек ещё нет, но сущности нужно закладывать на старте, чтобы понимать, как всё будет развиваться и не плодить лишнего. MVP = проектирование модели данных и описание процесса. Реализация UI и логики — в следующих итерациях.

#### Критерии приёмки (MVP)

- [ ] Спроектирована и задокументирована модель данных для домашних заданий (в docs/01-domain-model.md).
- [ ] Описан базовый пайплайн ДЗ: создание задания → отправка решения → проверка → фидбек → принятие/доработка.
- [ ] Определены роли: кто создаёт ДЗ (менеджер/куратор), кто проверяет (куратор/AI), кто видит результаты.
- [ ] Сущности не конфликтуют с существующей моделью (Lesson, Module, Enrollment, LessonProgress).

### Итерация 2: Реализация

#### Критерии приёмки

- [ ] Менеджер/куратор может создать домашнее задание к уроку: описание, формат ответа (текст, файл, ссылка), дедлайн.
- [ ] Студент видит ДЗ на странице урока в LMS и может отправить решение.
- [ ] Куратор видит очередь ДЗ для проверки (свои группы).
- [ ] Куратор оставляет фидбек: текст, оценка (принято / на доработку / отклонено).
- [ ] Студент видит фидбек в LMS и может отправить доработанное решение.
- [ ] В матрице прогресса куратора (LMS-04) видно: у кого ДЗ сдано, принято, на доработке.

## Черновик модели данных

```
Homework (домашнее задание — шаблон)
├── id
├── lessonId → Lesson
├── title
├── description (текст задания)
├── format: text | file | link | mixed
├── dueOffsetDays? (дедлайн = cohort.startDate + offset, или абсолютная дата)
├── isRequired: boolean
├── order (порядок, если несколько ДЗ на урок)
├── createdAt

HomeworkSubmission (отправка решения студентом)
├── id
├── homeworkId → Homework
├── enrollmentId → Enrollment (студент + поток)
├── content (текст ответа или JSON с файлами/ссылками)
├── status: draft | submitted | in_review | accepted | revision_requested | rejected
├── submittedAt
├── createdAt

HomeworkReview (проверка куратором / AI)
├── id
├── submissionId → HomeworkSubmission
├── reviewerId → User (куратор) | null (если AI)
├── reviewType: curator | ai
├── feedback (текст фидбека)
├── verdict: accepted | revision_requested | rejected
├── createdAt
```

## Базовый пайплайн

```
Менеджер создаёт ДЗ к уроку
         ↓
Студент видит ДЗ в LMS → отправляет решение
         ↓
Submission.status = submitted
         ↓
Куратор видит в очереди → проверяет → оставляет фидбек
         ↓
   ┌─ accepted → Submission.status = accepted
   ├─ revision_requested → студент дорабатывает → новая submission
   └─ rejected → Submission.status = rejected
```

## Открытые вопросы

- Один урок — одно ДЗ или несколько?
- ДЗ привязано к уроку или к модулю?
- Дедлайн: фиксированная дата или относительно начала потока?
- AI-ревью: на каком этапе подключать? Pre-review (AI проверяет перед куратором) или отдельный режим?
- Файлы: где хранить (S3, local storage)?
