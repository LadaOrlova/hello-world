# NOTIF-01: Студент выбирает предпочтительный транспорт для уведомлений: Telegram, email или другой мессенджер. Система доставляет уведомления через выбранный канал

**Статус в прототипе:** Не реализовано

## Параметры реализации

### Критерии приёмки

- [ ] В личном кабинете студент может выбрать предпочтительный канал уведомлений (минимум: Telegram или email).
- [ ] Выбор канала сохраняется в профиле пользователя и используется системой для последующих отправок.
- [ ] Уведомления отправляются в выбранный канал для поддержанных типов событий прототипа.
- [ ] Если выбран Telegram, но Telegram-аккаунт не привязан, система применяет fallback на email.
- [ ] Студент может в любой момент изменить предпочтительный канал, и последующие уведомления идут по новому выбору.
- [ ] Факт отправки или неотправки уведомления (канал, статус, время) фиксируется в системе для операционного контроля.
- [ ] В рамках прототипа поддержка других мессенджеров (кроме Telegram) остаётся follow-up и не блокирует приёмку базового сценария.

## Что нужно реализовать в прототипе

Минимальный сценарий выбора канала уведомлений:

- [ ] Студент в личном кабинете выбирает предпочтительный канал: Telegram или email
- [ ] Система отправляет уведомления через выбранный канал
- [ ] Если канал недоступен (Telegram не привязан) — система использует fallback (email)
- [ ] Студент может изменить выбор в любой момент

### Что уже есть в платформе

- Telegram-бот с привязкой аккаунтов — отправка в Telegram технически возможна
- Email пользователя хранится в системе — можно отправлять письма
- Поле preferredChannel в профиле пользователя — предусмотрено в схеме данных
- Очередь pg-boss — можно планировать и ставить в очередь уведомления
