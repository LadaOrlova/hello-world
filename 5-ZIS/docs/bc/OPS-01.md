# OPS-01: При сбое в системе во время активного воркшопа или потока команда знает чёткий порядок действий для восстановления работы

**Статус в прототипе:** Частично

## Параметры реализации

### Критерии приёмки

- [ ] Для команды подготовлен и доступен операционный runbook с пошаговыми действиями при типовых инцидентах во время потока и воркшопа.
- [ ] Runbook покрывает минимум базовые классы проблем прототипа: недоступность приложения, сбой фоновых процессов, зависшие onboarding и jobs.
- [ ] Команда может по runbook быстро проверить состояние системы через `/api/health`.
- [ ] Команда может обнаружить и диагностировать зависшие процессы через dashboard и process ops.
- [ ] Команда может выполнить восстановительные действия (retry и resolve) для зависших шагов через Process Ops.
- [ ] Для инцидента доступна трассировка по логам (структурированные логи), достаточная для определения причины сбоя.
- [ ] Для критичных сбоев настроено уведомление команды в Telegram (или эквивалентный оперативный канал), либо в прототипе зафиксировано как обязательный follow-up с датой внедрения.

## Что работает в прототипе

Базовые инструменты мониторинга и восстановления:

1. Эндпоинт проверки здоровья (`/api/health`) — можно быстро проверить, работает ли система
2. Дашборд менеджера показывает зависшие задачи (dead jobs) — видно, что что-то сломалось
3. Панель Process Ops позволяет перезапустить или вручную завершить зависший шаг процесса
4. Логи системы ведутся через Pino — можно найти причину сбоя

## Что нужно доработать в прототипе

- [ ] Создать операционный runbook: пошаговая инструкция для команды при типовых сбоях
- [ ] Добавить автоматическое уведомление в Telegram при обнаружении проблем

### Что уже есть в платформе

- Health check API — можно подключить внешний мониторинг
- Дашборд с dead jobs — видно зависшие процессы
- Process Ops — retry/resolve для шагов онбординга
- Структурированные логи — быстрый поиск ошибок

## Что может быть на следующих этапах

### Интеграции

- Подключение внешнего мониторинга (UptimeRobot, Datadog)
- Автоматический алерт при падении системы

### Расширенные сценарии

- Автоматический failover: система восстанавливается без ручного вмешательства
- Плановое обслуживание: страница «Техработы» для студентов
- Post-mortem процесс: шаблон разбора инцидента
