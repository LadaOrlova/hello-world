# LEAD-02: Лид, пришедший с любого лендинга, автоматически попадает в единую базу и отображается в соответствующем разделе админки

**Статус в прототипе:** Работает (базовый сценарий)

> **Источник доработок:** Фидбек Руслана (продажи/работа с клиентами), 2026-03-17.

## Параметры реализации

### Критерии приёмки (базовые — работают)

- [x] Лиды из разных источников (TypeForm webhook, внутренняя форма `/lead`, публичный API) автоматически попадают в единую базу ZIS.
- [x] В админке `/admin/leads` отображается единый список лидов без разделения по отдельным системам.
- [x] Для каждого лида в списке доступны минимум: источник, дата создания, контактные данные, текущий статус.
- [x] Дедупликация по email предотвращает создание дублирующих лидов при повторной заявке с тем же адресом.
- [x] Для нового лида автоматически назначается ответственный менеджер по текущему правилу auto-assign.
- [x] Менеджер может переводить лид по статусам воронки.
- [x] Все изменения статуса лида сохраняются через Event Log и доступны для просмотра.

### Критерии приёмки (MVP-доработки)

#### Карточка лида

- [ ] В админке доступна отдельная страница карточки лида (`/admin/leads/[id]`).
- [ ] В карточке отображаются: профиль (имя, email, phone, telegram, компания), статус, назначенный менеджер, продукт/тариф, UTM-данные, даты.
- [ ] В карточке отображается лог заметок (structured notes): массив записей `{date, managerId, text}`.
- [ ] Менеджер может добавлять заметки из карточки (например: «позвонил, обсудили цену», «отправил презентацию»).
- [ ] Если лид конвертирован — в карточке видна ссылка на связанного пользователя и его заказы.
- [ ] Быстрые ссылки для коммуникации: «Написать в Telegram» (deep link `t.me/username`), «Написать email» (mailto:).

#### Операционные статусы воронки

- [ ] Статусы лида отражают реальный процесс продаж: `new` → `in_progress` → `negotiation` → `converted` / `lost`.
- [ ] При переводе лида в `lost` менеджер указывает причину (`lostReason`): не актуально по дате, перестал отвечать, не актуально по бюджету, не целевой лид, другое.
- [ ] Для B2B-лидов сохраняется отдельный статус `reject` (отклонение заявки, не путать с `lost`).

> **Решение (2026-03-17):** Статусы лида описывают воронку **до покупки**. Когда лид переходит в `converted` — создаётся заказ (Order), и дальнейшие этапы (выставление счёта, отправка ссылки на оплату, оплата) отслеживаются через статусы Order. Это разделение ответственности: Lead = продажа, Order = оплата.

| Этап менеджера | Сущность в ZIS | Статус |
|---|---|---|
| Новый лид | Lead | `new` |
| В работе | Lead | `in_progress` |
| Принимает решение | Lead | `negotiation` |
| Выставляем счёт | Order | `pending_payment` |
| Отправили ссылку на оплату | Order | paymentUrl сгенерирован |
| Оплатил | Order | `paid` |

#### Счётчики воронки

- [ ] В списке лидов `/admin/leads` отображаются счётчики количества лидов по каждому статусу (бейджи в шапке или боковая панель).
- [ ] На дашборде менеджера — виджет «Воронка лидов»: количество лидов на каждом этапе.

## Что работает в прототипе

Единая база заявок из всех источников:

1. Заявка приходит из TypeForm — автоматически сохраняется в системе через webhook
2. Заявка приходит с внутренней формы `/lead` — сохраняется в системе
3. Заявка приходит через публичный API — сохраняется в системе
4. Менеджер открывает раздел «Лиды» в админке (`/admin/leads`) и видит все заявки из всех источников в одной таблице
5. Для каждой заявки показан источник, дата, контактные данные и текущий статус
6. Система автоматически дедуплицирует заявки по email — повторная заявка с того же адреса не создаёт дубль
7. Менеджер автоматически назначается на новую заявку
8. Менеджер может менять статус заявки

### Связь лида и пользователя

> **Решение (2026-03-17):** Отдельное поле `userId` на модели Lead **не добавляется** — это избыточно. Связь лида с пользователем вычисляется:
> - Явная конвертация: `convertedToUserId` (уже есть, ставится при конвертации)
> - До конвертации: поиск по email/phone через существующий `dedupKey`
>
> В карточке лида UI показывает «Связанный пользователь» если найден по email. Хранить FK дополнительно — данные будут рассинхронизироваться.
