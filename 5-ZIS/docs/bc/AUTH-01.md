# AUTH-01: Пользователь регистрируется один раз и получает единый аккаунт, действующий во всех продуктах экосистемы

**Статус в прототипе:** Работает

## Параметры реализации

### Критерии приёмки

- [ ] Пользователь может начать вход с email на `/auth` и получить одноразовый код (OTP) на указанный email.
- [ ] Пользователь может подтвердить вход кодом на `/auth/verify`.
- [ ] При первом успешном входе по email автоматически создаётся новый аккаунт без отдельного шага регистрации.
- [ ] При повторном входе с тем же email используется существующий аккаунт (дубликат аккаунта не создаётся).
- [ ] Один и тот же аккаунт пользователя работает в разных контурах системы (минимум: student и admin/curator при назначенной роли) без повторной регистрации.
- [ ] После входа пользователь перенаправляется в раздел, соответствующий его роли.
- [ ] Активная сессия сохраняется: при повторном открытии системы повторный вход не требуется, пока сессия валидна.

## Что работает в прототипе

Полный цикл регистрации и входа с единым аккаунтом:

1. Пользователь заходит на страницу входа (`/auth`) и вводит свой email
2. Система отправляет на указанный email одноразовый код подтверждения (6 цифр, действует 5 минут)
3. Пользователь вводит код на странице подтверждения (`/auth/verify`)
4. Если пользователь входит впервые — система автоматически создаёт ему аккаунт (регистрация и вход — один и тот же процесс, не нужно отдельно «регистрироваться»)
5. После входа пользователь автоматически перенаправляется в свой раздел в зависимости от роли: админ и менеджер — в панель управления, куратор — в кабинет куратора, студент — в личный кабинет
6. Один и тот же аккаунт (email) используется во всех разделах системы: админке, личном кабинете студента, кабинете куратора — без повторной регистрации
7. Пользователь может записаться на любой курс, перейти в другой раздел или получить новую роль — всё в рамках одного аккаунта
8. Сессия сохраняется: при повторном визите не нужно входить заново, пока сессия активна

## Что может быть на следующих этапах

### Интеграции

- Подключение боевой отправки email через Mailgun/Resend (сейчас в dev-режиме код выводится в логи сервера)
- Вход через Telegram (привязка аккаунта Telegram к email-аккаунту)

### Пользовательский опыт

- Запоминание устройства: на доверенных устройствах не запрашивать код повторно
- Более информативная страница входа: логотип, брендинг, подсказки
- Автоматическая подстановка кода из email (autocomplete one-time-code)

### Расширенные сценарии

- Двухфакторная аутентификация для администраторов
- Управление активными сессиями: пользователь видит, где он вошёл, и может завершить сессии
- Журнал входов для безопасности (кто, когда, откуда)
