## Параметры реализации

### Критерии приёмки

- [ ] В системе подключён платёжный провайдер YooKassa через конфигурацию платформы и он используется в рабочем checkout-сценарии.
- [ ] Для YooKassa полный платёжный цикл работает сквозно: создание заказа -> переход к оплате -> получение webhook -> обновление статуса оплаты в ZIS.
- [ ] После успешного webhook заказ и платёж в ZIS переходят в корректные финальные статусы без ручного обновления данных.
- [ ] Ошибочный или неуспешный платёж фиксируется в системе с корректным статусом и без ложного зачисления доступа.
- [ ] Инфраструктура приёма webhook валидирует входящие события и обрабатывает только допустимые события от YooKassa.
- [ ] Все транзакции по YooKassa отображаются в едином биллинге ZIS наравне с остальными заказами.
- [ ] В рамках прототипа допускается отсутствие UI-конфигуратора провайдеров в админке; подключение выполняется через техническую конфигурацию (follow-up: self-service UI).

## Бронирование места на потоке при оплате

> **Фидбек Насти (2026-03-11):** Бронирование мест — критично для Boost (места улетают быстро). Должно работать единообразно для B2C и B2B.

### Критерии приёмки (бронирование)

- [ ] При начале процесса оплаты (B2C: создание pending-заказа) место на потоке бронируется за пользователем.
- [ ] При выставлении счёта компании (B2B) места на потоке бронируются за указанных сотрудников.
- [ ] Забронированные места учитываются в лимите мест потока и не доступны другим покупателям.
- [ ] На дашборде менеджера отображаются бронирования с задерживающейся оплатой (превышение ожидаемого срока).
- [ ] Аннулирование брони **не происходит автоматически** — требуется явное подтверждение менеджера.
- [ ] После подтверждения оплаты бронь переходит в статус «оплачено».
- [ ] После отмены заказа или аннулирования менеджером место освобождается.
