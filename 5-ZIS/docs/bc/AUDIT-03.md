# AUDIT-03: Внутренние изменения системы (новые модули, изменения логики) фиксируются в changelog с описанием «как было / как стало / зачем»

**Статус в прототипе:** Работает

## Параметры реализации

### Критерии приёмки

- [ ] Для изменений runtime и кода проекта ведётся `CHANGELOG.md` в установленном формате.
- [ ] Для изменений документации, процессов и архитектуры ведётся `docs/CHANGELOG.md` в установленном формате.
- [ ] Каждая запись changelog содержит минимум смысловые блоки: что было, что стало, зачем (или эквивалент по принятому шаблону проекта).
- [ ] Записи имеют уникальный идентификатор (формат `CHG-NNN` или текущий принятый эквивалент) для ссылок и трассировки.
- [ ] По changelog можно восстановить историю ключевых изменений по дате и затронутым частям системы.
- [ ] Отсутствуют немые существенные изменения: все значимые изменения логики и модулей отражаются в changelog.

## Что работает в прототипе

Ведение структурированного журнала изменений:

1. Каждое изменение кода фиксируется в `CHANGELOG.md` с описанием: что изменилось и зачем
2. Каждое изменение документации и архитектуры фиксируется в `docs/CHANGELOG.md` по формату: было / стало / почему / затронутые файлы
3. Записи нумеруются сквозной нумерацией (CHG-NNN) для простоты ссылок
4. Изменения группируются по дням
5. Новый сотрудник может открыть changelog и понять историю решений по проекту
