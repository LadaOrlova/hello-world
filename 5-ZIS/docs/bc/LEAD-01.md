# LEAD-01: Любая внешняя точка входа (HR-портал, кастомный лендинг, конструктор сайта, бот) может отправлять лиды в ZIS как в API-сервис, без встраивания в кодовую базу ZIS

**Статус в прототипе:** Частично

## Параметры реализации

### Критерии приёмки

- [ ] Внешний источник (не встроенный в код ZIS) может отправить лид в ZIS через публичный API ingestion.
- [ ] Для интеграции зафиксирован контракт API лидов: версия, обязательные поля, формат ответа и коды ошибок.
- [ ] При повторной отправке одного и того же внешнего лида по idempotency-ключу дубликат в ZIS не создаётся.
- [ ] Для каждого принятого лида сохраняется источник интеграции (`externalSystem`) и внешний идентификатор (`externalLeadId`).
- [ ] Доступ к ingestion API защищён service-to-service авторизацией для внешних приложений.
- [ ] Принятые лиды отображаются в едином разделе лидов админки ZIS независимо от источника.

## Что работает в прототипе

Базовый API-контур для приёма лидов уже есть:

1. TypeForm подключается к ZIS через webhook
2. Внутренняя форма `/lead` может использоваться как готовая точка сбора
3. Публичный API (`/api/leads`) позволяет внешним приложениям отправлять заявки
4. Telegram-боты создают лидов автоматически при контакте (source=telegram_bot) — см. BOT-01
5. Все лиды попадают в единый раздел «Лиды» в админке ZIS
6. Публичные продуктовые лендинги уже могут генерироваться из данных курса

## Что может быть на следующих этапах

### Пользовательский опыт

- Каталог подключённых источников лидов в админке: HR-порталы, лендинги, боты, формы
- Мастер подключения нового источника: ключ API, webhook URL, проверка тестового события

### Интеграции

- Готовые адаптеры: Tilda, Webflow, TypeForm, кастомные Next.js/React-лендинги
- Пилот внешнего HR-портала с собственной БД и отправкой лидов в ZIS по API

### Расширенные сценарии

- Версионирование API и обратная совместимость для внешних команд
- Rate limiting и антиспам-политики для публичных интеграций
- SLA/алертинг по каналу ingestion (ошибки, задержки, деградация)
