## Что работает в прототипе

ZIS уже предоставляет стабильное API-ядро, на которое можно опираться при создании внешних приложений:

1. **tRPC API** — 12 роутеров с типобезопасным контрактом: users, billing, enrollments, lms, crm, products, companies и др.
2. **Auth API** — better-auth с bearer-токенами: внешнее приложение может аутентифицировать запросы к ZIS
3. **Event Bus** — pg-boss публикует бизнес-события (оплата, зачисление, онбординг), на которые можно подписаться
4. **Webhooks** — входящие точки для внешних систем (TypeForm, YooKassa, Telegram)
5. **Стандартизированная архитектура** — каждый раздел админки следует одному паттерну (page.tsx + client-компонент), понятному AI-инструментам

## Что может быть на следующих этапах

### Экосистема внешних приложений

- HR-портал (своя БД, свой деплой) — читает данные о пользователях и зачислениях через ZIS API
- Кастомные лендинги продуктов — отправляют заявки через webhook в ZIS, получают данные о продуктах через API
- Маркетинговые боты — подписываются на события ZIS (новый лид, оплата) и реагируют автономно
- Внутренние инструменты команды — дашборды, отчёты, автоматизации, работающие поверх ZIS API

### Инфраструктура для разработчиков

- Документация для вайб-кодинга: «как за час создать мини-приложение, которое работает с данными ZIS»
- SDK / клиентская библиотека для типизированного доступа к ZIS API
- Песочница: тестовый инстанс ZIS с seed-данными для безопасных экспериментов
- Мониторинг внешних интеграций: какие приложения подключены, сколько запросов, какие ошибки

## Архитектурный принцип

Вайб-кодеры **не коммитят в ZIS-репозиторий** и не меняют схему данных. Они создают отдельные приложения со своими базами данных и взаимодействуют с ZIS исключительно через Public API и подписку на события. Это обеспечивает:

- **Изоляцию** — ошибка во внешнем приложении не ломает ядро системы
- **Свободу** — любой стек, любой фреймворк, любой AI-инструмент
- **Безопасность** — ZIS контролирует, какие данные доступны через API
