## 10. Система агентов

### 10.1 Концепция

Агенты — это автоматизированные процессы, выполняющие операции по расписанию (cron) или по событиям (через pg-boss Jobs, ставящиеся в транзакции сервисов). На первых этапах агенты реализованы как pg-boss Job handlers. В будущем — как отдельные процессы с доступом через MCP/API.

Все действия агентов логируются:

- Event (event_type: "agent_action_completed", source: "agent")
- AuditLog (source: "agent")
- Уведомления в служебный TG-чат команды

### 10.2 Список агентов

| Агент                 | Триггер                            | Действие                                                                                                                                               | Фаза    |
| --------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | ------- |
| **Онбордер**          | Event: payment_success             | Создаёт онбординг-страницу, мониторит вступление в чаты, отправляет инструкции                                                                         | 1 (Job) |
| **Контролёр доступа** | Cron: ежедневно                    | Проверяет ChatMembership ↔ Enrollment, предупреждает / удаляет                                                                                         | 1 (Job) |
| **Заливщик контента** | Event: видео добавлено в Kinescope | Создаёт урок в LMS, привязывает к модулю/потоку. Этап 2: полуавтоматический (webhook → предзаполнение, менеджер подтверждает). Будущее: полный автомат | 2 (Job) |
| **Наблюдатель B2B**   | Cron: каждые 4 часа                | Проверяет поступления по неоплаченным счетам, обновляет статусы                                                                                        | 2 (Job) |
| **Follow-up**         | Cron: ежедневно                    | Отправляет напоминания необработанным лидам и неоплаченным заявкам                                                                                     | 2 (Job) |
| **Аналитик**          | Cron: еженедельно                  | Формирует отчёт по ключевым метрикам, отправляет в TG                                                                                                  | 3 (Job) |
| **NPS**               | Event: cohort_completed            | Отправляет опрос NPS студентам через 1 день после завершения                                                                                           | 3 (Job) |
| **Реферальный**       | Event: promo_code_used             | Начисляет бонус рефереру (подписку Maker)                                                                                                              | 3 (Job) |

### 10.3 Архитектура агентов (этапы 1-3)

```
На этапах 1-3 агенты = pg-boss Job handlers в Worker:

Worker
├── onboarding.*     ← Агент "Онбордер"
├── access.*         ← Агент "Контролёр доступа"
├── billing.b2b.*    ← Агент "Наблюдатель B2B"
├── crm.followup.*   ← Агент "Follow-up"
├── analytics.*      ← Агент "Аналитик"
├── feedback.nps.*   ← Агент "NPS"
└── promo.referral.* ← Агент "Реферальный"

Каждый handler:
  1. Получает данные из pg-boss Job
  2. Выполняет бизнес-логику (через tRPC services)
  3. Записывает Event + AuditLog
  4. Отправляет уведомление в служебный TG-чат
```

### 10.4 Эволюция в MCP-агенты (будущая фаза)

```
В будущем агенты могут стать отдельными процессами:

Агент (отдельный процесс)
├── Получает API Token (роль: agent, ограниченные scopes)
├── Подключается к системе через MCP-сервер
├── Слушает Event'ы через webhook/SSE
├── Выполняет действия через MCP Tools
└── Логирует через AuditLog (source: "agent")

Преимущество: можно запускать агентов на другой машине,
писать на другом языке, использовать OpenClaw/LangChain.
Переход плавный — бизнес-логика та же, меняется только способ вызова.
```

## 11. API для AI-агентов: MCP-сервер и CLI-интерфейс

### 11.1 Мотивация и концепция

Система проектируется так, чтобы менеджер (или любой пользователь с соответствующей ролью) мог подключить внешний AI-агент — свой бот, скрипт или полноценное AI-приложение — и через него выполнять действия в бэк-офисе **в рамках прав своего пользователя**. Это позволяет:

- **Навайбкодить бота** — менеджер без глубоких технических навыков с помощью Claude/GPT создаёт бота, который автоматизирует его рутину (разбор лидов, рассылка напоминаний, формирование отчётов)
- **Интеграция с внешними сервисами** — подключить Zapier, n8n, Make.com и другие автоматизации
- **AI-ассистент менеджера** — чат-бот в Telegram, который понимает естественный язык и выполняет действия в системе ("Зачисли Петрову в поток 14 на продвинутый тариф")
- **Аналитические агенты** — агент, который каждое утро собирает дэшборд и отправляет сводку в Telegram
- **Батчевые операции** — скрипт для массовых операций через CLI без захода в веб-интерфейс

### 11.2 Два интерфейса доступа

### 11.3 MCP-сервер (Model Context Protocol)

MCP-сервер позволяет AI-агентам (Claude Desktop, Cursor, самописным агентам на Claude/OpenAI API) работать с системой через стандартный протокол — вызывать «тулы» и читать «ресурсы».

#### Транспорт

```
Два варианта подключения:

1) stdio (локальный)
   ├── MCP-сервер запускается как subprocess
   ├── Общение через stdin/stdout
   ├── Подходит для Claude Desktop, Cursor, локальных агентов
   └── Конфигурация: в ~/.claude/mcp_servers.json или settings IDE

2) SSE / Streamable HTTP (удалённый)
   ├── MCP-сервер как HTTP-эндпоинт: POST /mcp
   ├── Подходит для удалённых агентов, автоматизаций
   ├── Аутентификация через Bearer token в заголовке
   └── Поддержка streaming для длинных операций
```

#### Аутентификация и авторизация

```
┌──────────────────────────────────────────────────────────────┐
│ Принцип: агент действует ОТ ИМЕНИ конкретного пользователя  │
│                                                              │
│ 1) Пользователь создаёт API Token в бэк-офисе:              │
│    Web App → Настройки → API Tokens → "Создать токен"        │
│    ├── Название: "Мой утренний бот"                          │
│    ├── Срок действия: 90 дней (или бессрочный)               │
│    ├── Scopes (ограничение прав ниже роли): опционально      │
│    │   ├── read:leads — только чтение лидов                  │
│    │   ├── write:leads — чтение + запись лидов                │
│    │   ├── read:enrollments — чтение зачислений               │
│    │   ├── write:enrollments — создание зачислений            │
│    │   ├── read:analytics — чтение аналитики/ОС              │
│    │   ├── write:payments — операции с оплатами               │
│    │   ├── write:schedule — управление расписанием            │
│    │   ├── write:lms — управление контентом LMS               │
│    │   ├── admin:system — мониторинг системы                  │
│    │   └── * — все права роли (по умолчанию)                 │
│    └── Токен: ztk_a1b2c3d4e5f6... (показывается один раз)   │
│                                                              │
│ 2) Агент при каждом запросе передаёт токен                   │
│    → система определяет User по токену                       │
│    → применяет RBAC роли User + scope ограничения токена     │
│    → менеджер через агент видит РОВНО то же, что в Web App   │
│                                                              │
│ 3) Аудит: все действия через токен логируются                │
│    AuditLog.source = "api_token"                             │
│    AuditLog.token_name = "Мой утренний бот"                  │
│    → полная трассировка: кто, через какой агент, когда       │
│                                                              │
│ 4) Отзыв токена: мгновенная блокировка в Web App             │
│    При деактивации пользователя (is_active=false)            │
│    → все его токены автоматически инвалидируются              │
└──────────────────────────────────────────────────────────────┘
```

#### MCP Resources (ресурсы для чтения)

Ресурсы — это данные, которые AI-агент может прочитать для контекста. Read-only, не изменяют состояние.

```
Resources:

dashboard://summary
  └── Персональная сводка (аналог M.1): лиды, оплаты, dead jobs, метрики
      Данные фильтруются по RBAC текущего пользователя

leads://list?status=new&assigned_to=me
  └── Список лидов с фильтрацией

leads://{id}
  └── Детали конкретного лида

cohorts://list?status=active
  └── Список потоков (только доступные по RBAC)

cohorts://{id}
  └── Детали потока: студенты, расписание, метрики

cohorts://{id}/students
  └── Список студентов потока с онбординг-статусами

enrollments://{id}
  └── Детали зачисления: статус, онбординг, посещаемость

schedule://{cohort_id}
  └── Расписание занятий потока

lms://products/{id}/modules
  └── Модули и уроки продукта (контент)

lms://enrollments/{id}/progress
  └── Прогресс студента по урокам

feedback://{cohort_id}/summary
  └── Сводка обратной связи: NPS тренд, AI-инсайты, % ответивших

payments://pending
  └── Ожидающие оплаты (> 30 мин)

invoices://overdue
  └── Просроченные счета

system://health
  └── Здоровье интеграций, состояние очередей

system://dead-jobs
  └── Мёртвые задачи, требующие внимания

analytics://cohort/{id}
  └── Аналитика по потоку: посещаемость, NPS, конверсия

products://list
  └── Справочник продуктов с тарифами
```

#### MCP Tools (инструменты для действий)

Тулы — это действия, которые AI-агент может выполнять. Каждый тул маппится на одну или несколько функций менеджера (M.\*) и, соответственно, на цепочку атомарных операций (раздел 9).

```
Tools:

── Лиды ──────────────────────────────────────────────

update_lead_status
  ├── Параметры: { lead_id, status, notes? }
  ├── Маппинг: M.2 → INT.5 + DB.2 + DB.25 + INT.11
  ├── Пример: "Взять лид #851 в работу"
  └── RBAC: лид должен быть назначен на текущего пользователя
            (или пользователь — admin/ops_director)

── Оплаты ────────────────────────────────────────────

create_payment_link
  ├── Параметры: { lead_id, cohort_id, tier_id, promo_code? }
  ├── Маппинг: M.3 → INT.5 + INT.1 + DB.1 + DB.4 + DB.5
  │            + PAY.1 + DB.25
  ├── Возвращает: { payment_url, enrollment_id, expires_at }
  └── RBAC: scope write:payments

create_invoice
  ├── Параметры: { company_id, product_id, tier_id, seats,
  │               due_date, students[]? }
  ├── Маппинг: M.4 → INT.5 + DB.19 + DB.6 + DOC.2 + S3.1 + EMAIL.1 + DB.25
  ├── Возвращает: { invoice_id, pdf_url }
  └── RBAC: scope write:payments

confirm_invoice_payment
  ├── Параметры: { invoice_id }
  ├── Маппинг: M.5 → INT.5 + DB.6 + DB.4 + DB.25
  └── RBAC: scope write:payments

process_refund
  ├── Параметры: { enrollment_id, reason }
  ├── Маппинг: M.8 → INT.5 + PAY.3 + DB.5 + DB.4 + DB.3 + LMS.2 + TG.3 + DB.25
  ├── Возвращает: { refund_id, process_id }
  ├── ⚠️ Деструктивное действие — требует confirmation: true
  └── RBAC: scope write:payments

── Зачисление ────────────────────────────────────────

enroll_student
  ├── Параметры: { user_id | lead_id, cohort_id, tier_id }
  ├── Маппинг: M.6 → INT.5 + INT.1 + DB.1 + DB.3 + INT.4 + [каскад] + DB.25
  ├── Возвращает: { enrollment_id, process_id }
  └── RBAC: scope write:enrollments

batch_enroll
  ├── Параметры: { company_id, cohort_id, tier_id,
  │               students: [{ name, email, phone?, telegram? }] }
  ├── Маппинг: M.7 → INT.5 + INT.1 (xN) + [для каждого:] DB.1 + DB.3 + INT.4
  ├── Возвращает: { process_id, total: N }
  ├── Асинхронный: возвращает process_id для отслеживания прогресса
  └── RBAC: scope write:enrollments

── Расписание ────────────────────────────────────────

create_schedule
  ├── Параметры: { cohort_id, title, type, scheduled_at,
  │               duration_minutes, available_for_tiers[]? }
  ├── Маппинг: M.10 → INT.5 + INT.8 + DB.11 + INT.4 + [каскад] + DB.25
  ├── Возвращает: { schedule_id, process_id, zoom_url }
  └── RBAC: scope write:schedule

reschedule
  ├── Параметры: { schedule_id, new_scheduled_at }
  ├── Маппинг: M.11 → INT.5 + INT.8 + DB.11 + INT.4 + [каскад] + DB.25
  ├── Возвращает: { process_id }

── Обратная связь ────────────────────────────────────

create_feedback_campaign

  │               send_via, target_lesson_id? }

send_feedback_reminder

  ├── Параметры: { cohort_id, force?: boolean }

send_invoice_reminder
  ├── Параметры: { invoice_id }
  ├── Маппинг: EMAIL.1 + DB.25 + INT.11
  └── RBAC: scope write:payments
```

### 11.4 CLI-интерфейс (zamesin)

CLI — тонкий клиент поверх того же API. Для скриптов, автоматизаций и ручных операций из терминала.

```
Установка и аутентификация:

  npm install -g @zamesin/cli
  zamesin auth login              # OAuth-флоу → открывает браузер
  # или
  zamesin auth token ztk_a1b2...  # API-токен напрямую

Основные команды:

  # Дэшборд
  zamesin dashboard               # Утренняя сводка (как M.1)

  # Лиды
  zamesin leads list --status=new --mine    # Мои новые лиды
  zamesin leads show 851                    # Детали лида
  zamesin leads update 851 --status=contacted --note="Написала в TG"

  # Оплаты
  zamesin pay create-link --lead=849 --cohort=14 --tier=advanced --promo=PARTNER20
  zamesin pay confirm-invoice 78
  zamesin pay refund --enrollment=1198 --reason="Не подошёл формат"

  # Зачисление
  zamesin enroll --lead=849 --cohort=14 --tier=advanced
  zamesin enroll batch --company=5 --cohort=14 --tier=team --file=students.csv

  # Расписание
  zamesin schedule create --cohort=14 --title="Юнит-экономика" \
    --type=workshop --date="2026-02-16T19:00" --duration=120
  zamesin schedule reschedule 81 --date="2026-02-10T20:00"

  # LMS-контент
  zamesin lms modules --product=1
  zamesin lms lessons --module=5
  zamesin lms progress --enrollment=1203

  # Обратная связь
  zamesin feedback campaigns --cohort=14
  zamesin feedback remind 47

  # Потоки
  zamesin cohorts list --status=active
  zamesin cohorts complete 14

  # Мониторинг
  zamesin system health
  zamesin system dead-jobs
  zamesin system retry-job 8921

  # Процессы
  zamesin process status 501        # Статус процесса (с прогрессом)
  zamesin process watch 520         # Live-наблюдение за процессом

  # Вывод
  --format=json                     # JSON (для скриптов/агентов)
  --format=table                    # Таблица (по умолчанию)
  --format=csv                      # CSV (для экспорта)
```

### 11.6 Архитектура MCP-сервера

```
┌─────────────────────────────────────────────────────────────────────┐
│                      MCP Server (Node.js)                           │
│                                                                     │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ Transport Layer                                              │   │
│  │ ├── StdioTransport (для локальных клиентов)                  │   │
│  │ └── SSETransport (для удалённых клиентов)                    │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                            │                                        │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ Auth Layer                                                   │   │
│  │ ├── Извлечение токена из env (stdio) или Header (SSE)        │   │
│  │ ├── Валидация токена → определение User                      │   │
│  │ ├── Загрузка роли + scopes токена                            │   │
│  │ └── Контекст запроса: { user_id, role, scopes, token_name }  │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                            │                                        │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ Handler Layer                                                │   │
│  │                                                              │   │
│  │ Resources:                                                   │   │
│  │ ├── DashboardResource → GET /api/dashboard (с фильтром RBAC)│   │
│  │ ├── LeadsResource → GET /api/leads (с фильтром RBAC)        │   │
│  │ ├── CohortsResource → GET /api/cohorts                      │   │
│  │ └── ...                                                      │   │
│  │                                                              │   │
│  │ Tools:                                                       │   │
│  │ ├── UpdateLeadTool → POST /api/leads/:id                    │   │
│  │ ├── CreatePaymentLinkTool → POST /api/payments/create       │   │
│  │ ├── EnrollStudentTool → POST /api/enrollments               │   │
│  │ └── ...                                                      │   │
│  │                                                              │   │
│  │ Каждый handler:                                              │   │
│  │ 1. Проверяет scope токена                                    │   │
│  │ 2. Вызывает тот же API, что использует Web App              │   │
│  │ 3. Возвращает структурированный результат                   │   │
│  │ 4. Записывает в AuditLog с source="mcp"                     │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                            │                                        │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ API Client (внутренний HTTP-клиент к основному API)          │   │
│  │ ├── Передаёт API Token в Authorization header               │   │
│  │ ├── Retry с backoff при 429/503                              │   │
│  │ └── Timeout: 30s (обычные), 120s (batch-операции)           │   │
│  └──────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────┘
```

### 11.7 Безопасность

```
┌──────────────────────────────────────────────────────────────┐
│ Принципы безопасности API для агентов:                        │
│                                                              │
│ 1) Принцип наименьших привилегий                             │
│    - Токен не может дать больше прав, чем роль пользователя  │
│    - Scopes позволяют ограничить токен ещё сильнее            │
│    - Менеджер НЕ может через токен получить доступ admin'а   │
│                                                              │
│ 2) Rate Limiting для API-токенов                             │
│    - 100 req/min на токен (configurable)                     │
│    - 1000 req/hour на пользователя (сумма всех токенов)      │
│    - Burst: до 20 req/sec                                    │
│    - При превышении: 429 Too Many Requests + Retry-After     │
│                                                              │
│ 3) Аудит и мониторинг                                        │
│    - Все действия через токен → AuditLog                     │
│    - Дэшборд "API Usage" для admin:                          │
│      ├── Какие токены активны                                │
│      ├── Сколько запросов, какие действия                    │
│      ├── Аномалии (всплеск запросов, необычные действия)     │
│      └── Последний раз использован                           │
│    - Алерт при подозрительной активности                     │
│                                                              │
│ 4) Ротация и отзыв токенов                                   │
│    - Токен можно отозвать мгновенно из Web App               │
│    - Рекомендация: ротация каждые 90 дней                     │
│    - При утечке: одна кнопка "Отозвать все токены"           │
│                                                              │
│ 5) Запрет на эскалацию привилегий                             │
│    - Через API нельзя: создавать токены, менять роли,         │
│      управлять пользователями (кроме студентов)              │
│    - Эти действия — только через Web App с 2FA               │
│                                                              │
│ 6) Sandbox-режим (опционально)                               │
│    - Токен с флагом sandbox: true                            │
│    - Все действия выполняются, но не применяются:             │
│      ├── Платежи → в тестовую ЮKassa                        │
│      ├── TG → сообщения не отправляются (dry-run)            │
│      ├── Email → только на email владельца токена             │
│      └── Полезно для разработки и тестирования ботов         │
└──────────────────────────────────────────────────────────────┘
```

### 11.9 Открытые вопросы (MCP/CLI)

1. **SSE vs WebSocket** — какой транспорт для удалённого MCP? SSE проще, WebSocket двусторонний. Предложение: начать с SSE, перейти на Streamable HTTP (новый стандарт MCP).
2. **Webhook'и наружу** — нужно ли уведомлять внешних агентов о событиях в системе (новый лид, оплата, dead job)? Предложение: да, через configurable webhook URL с HMAC-подписью.
3. **MCP Prompts** — стоит ли добавить в MCP-сервер predefined prompts ("Утренняя сводка", "Отчёт по потоку", "Анализ рисков отчисления")? Предложение: да, как дополнительную удобство для AI-клиентов.
4. **Максимальное количество токенов на пользователя** — предложение: 5 активных токенов.
