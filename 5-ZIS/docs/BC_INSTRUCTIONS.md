# Инструкция: анализ бизнес-кейсов ZIS

## Зачем

Для каждого из 62 бизнес-кейсов нужен отдельный документ, который отвечает на два вопроса:

1. **Что уже работает в прототипе?** — можно показать заказчику/инвестору сегодня
2. **Что будет доведено до production-качества?** — на следующих этапах разработки

Документы хранятся в `docs/bc/{CASE-ID}.md` — по одному на кейс.

## Принцип разделения: прототип vs production

### Прототип (базовый уровень)

Прототип = механика работает, можно показать сценарий от начала до конца.

Признаки прототипного уровня:

- Happy path проходит полностью
- Mock-интеграции вместо боевых (например, mock YooKassa вместо реальной)
- UI функциональный, но без полировки
- Данные создаются через seed или админку, результат виден
- Нет обработки ошибок и граничных случаев

### Production (следующие этапы)

Production = система надёжна, удобна, масштабируема.

Признаки production-качества:

- Боевые интеграции (платёжные системы, email, SMS)
- Обработка ошибок, retry, fallback-сценарии
- Полированный UX (responsive, loading states, validations)
- Расширенные сценарии (edge cases, массовые операции)
- Аналитика, отчёты, метрики

## Методология анализа

Для каждого кейса выполни шаги:

### Шаг 1. Прочитать кейс

Открой `docs/00-business-cases.html` и найди кейс по ID (например, `COURSE-01`). Запиши:

- Полный текст кейса (формулировка бизнес-потребности)
- Текущий статус (зелёный / жёлтый / красный)
- Примечание, если есть

### Шаг 2. Проверить backlog

Поищи связанные задачи:

- В `backlog/tasks/` и `backlog/completed/` — по ID кейса или по ключевым словам
- В `backlog/drafts/` — черновики будущих задач
- Запиши, какие задачи относятся к кейсу

### Шаг 3. Проверить код

Проверь реализацию в коде:

- **tRPC роутеры** (`src/server/trpc/routers/`) — какие процедуры связаны с кейсом
- **UI-страницы** (`src/app/`) — какие страницы реализуют функционал
- **Сервисы** (`src/server/services/`) — бизнес-логика
- **Prisma-схема** (`prisma/schema.prisma`) — модели данных

Важно: проверяй не только наличие кода, но и полноту реализации. Например, поле может быть в базе данных, но отсутствовать в UI-форме.

### Шаг 4. Определить уровень реализации

Кейс попадает в одну из трёх категорий:

**A. Реализовано** (зелёный) — сценарий проходит от начала до конца, доработки косметические.

**B. Частично** (жёлтый) — часть шагов работает, часть нет. Описывай и то, и другое.

**C. Не реализовано** (красный) — в коде нет ничего специфичного для этого кейса.

Для категории C действует особое правило — см. ниже «Кейсы без реализации».

### Шаг 5. Заполнить шаблон

Используй шаблон ниже. Пиши на языке бизнеса — так, чтобы понял продакт-менеджер, а не разработчик.

## Шаблон файла `docs/bc/{CASE-ID}.md`

```markdown
# {CASE-ID}: {Текст кейса}

**Статус в прототипе:** {Работает / Частично / Не реализовано}

## Что работает в прототипе

Перечисли конкретные действия, которые можно выполнить прямо сейчас.
Формат — нумерованный список шагов сценария.

Пример:

1. Менеджер создаёт продукт в админке: название, slug, тип
2. Менеджер добавляет тарифы с ценами
3. Менеджер открывает набор в поток
4. Клиент видит продукт в каталоге, выбирает тариф
5. Клиент проходит оплату (тестовую) и попадает в курс

## Что нужно доработать в прототипе

Мелкие доработки, чтобы демо-сценарий был убедительным.
Только то, что можно сделать быстро (часы, не дни).

- [ ] Пример: добавить поле «Описание» в форму создания продукта

## Что может быть на следующих этапах

Перечисли возможности, которые выведут кейс на production-качество.
Группируй по смыслу (UX, интеграции, масштабирование и т.д.).

### Интеграции

- Пример: подключение боевой платёжной системы

### Пользовательский опыт

- Пример: email-уведомление после оплаты

### Расширенные сценарии

- Пример: массовая загрузка студентов

## Связанные задачи

| ID задачи | Название      | Статус |
| --------- | ------------- | ------ |
| ZIS-009   | Products CRUD | Done   |
```

## Кейсы без реализации

Если по кейсу в коде ничего специфичного нет — **всё равно описывай прототип**, но как план минимальной реализации. Логика:

1. **Опиши сценарий прототипа** — что система должна делать для этого кейса на самом базовом уровне. Пиши как шаги, которые пользователь сможет выполнить. Каждый шаг — с чекбоксом `- [ ]`, потому что ничего из этого пока не работает.

2. **Отметь, что есть в платформе уже сейчас.** Даже если кейс не реализован, часть инфраструктуры может уже существовать. Например: система пользователей, роли, компании, продукты, оплата — всё это фундамент, на который опирается будущая реализация. Перечисли то, что пригодится.

3. **Секция «Что может быть на следующих этапах»** заполняется как обычно — production-качество.

Пример для нереализованного кейса:

```markdown
# EXAMPLE-99: Менеджер может выгрузить отчёт по студентам в Excel

**Статус в прототипе:** Не реализовано

## Что нужно реализовать в прототипе

Минимальный сценарий, чтобы кейс заработал:

- [ ] Менеджер открывает раздел «Зачисления» в админке
- [ ] Менеджер нажимает кнопку «Экспорт в Excel»
- [ ] Система формирует файл со списком студентов и их статусами
- [ ] Файл скачивается в браузер

### Что уже есть в платформе

- Раздел «Зачисления» в админке с таблицей студентов — работает
- Данные о студентах, курсах и статусах — хранятся в системе
- Роли и права доступа — менеджер уже видит нужные данные

## Что может быть на следующих этапах

### Пользовательский опыт

- Выбор колонок для экспорта
- Фильтры по дате, курсу, статусу перед экспортом

### Расширенные сценарии

- Экспорт в PDF
- Автоматическая отправка отчёта на email по расписанию

## Связанные задачи

Задач пока нет — кейс предстоит реализовать.
```

Обрати внимание на отличия от шаблона для реализованного кейса:

- Секция называется **«Что нужно реализовать в прототипе»** вместо «Что работает в прототипе»
- Шаги — с `- [ ]` чекбоксами (ничего не сделано)
- Добавлена подсекция **«Что уже есть в платформе»** — перечисляет существующий фундамент
- Нет секции «Что нужно доработать в прототипе» (она сливается с основной)

## Правила написания

1. **Язык бизнеса.** «Клиент оплачивает курс» вместо «billing.checkout.start мутация создаёт Order».
2. **Конкретные действия.** «Менеджер нажимает кнопку Создать продукт» вместо «CRUD реализован».
3. **Честная оценка.** Если что-то работает только через seed — так и пиши. Если UI нет — отмечай.
4. **Без технического жаргона.** Читатель — продакт-менеджер, не разработчик. Исключение: названия систем (YooKassa, Telegram).
5. **Разделяй «работает» и «доделать».** Если механика есть, но кнопки в интерфейсе нет — это «доработать в прототипе», а не «работает».
6. **Не пропускай нереализованные кейсы.** Даже если ничего нет — опиши минимальный прототип как план с чекбоксами. Заказчик должен видеть, что кейс учтён и понятно, что нужно сделать.
