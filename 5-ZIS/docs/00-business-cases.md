# ZIS — Бизнес-кейсы (черновик)

> Документ фиксирует бизнес-сценарии, которые должна поддерживать система ZIS.
> Технические детали реализации намеренно вынесены за рамки этого документа.
> Задача — дать полную картину того, _что_ должно работать, чтобы затем расставить
> приоритеты по версиям релиза.

### Условные обозначения статусов

| Статус             | Значение                                                                   |
| ------------------ | -------------------------------------------------------------------------- |
| 🟢 **Реализовано** | Функционал полностью работает в рамках заявленного бизнес-кейса            |
| 🟡 **В плане**     | Частично реализовано или заложено в ROADMAP/backlog, требует доработки     |
| 🔴 **Не покрыто**  | Для этого бизнес-требования ещё ничего не сделано (нет ни кода, ни задачи) |

## 1. Единые пользователи и авторизация `AUTH`

**Что болит сейчас:** у каждого продукта (Boost, «Как делать продукт», Makers, Aura)
отдельная база пользователей и отдельный вход. Один и тот же человек — разные сущности
в разных системах.

### Бизнес-кейсы

- 🟢 **AUTH-01.** Пользователь регистрируется один раз и получает единый аккаунт, действующий
  во всех продуктах экосистемы.

  > _Реализовано в ZIS-007 (WP-007): better-auth + Email OTP, единая таблица User, 4 роли._

- 🟢 **AUTH-02.** Пользователь входит в любой продукт через единую точку авторизации —
  отдельная регистрация для каждого продукта не нужна.

  > _Реализовано в ZIS-007: единый auth endpoint `/api/auth/_`, OTP-вход по email.\*

- 🟡 **AUTH-03.** Команда видит полную историю пользователя в одном месте: какие продукты
  покупал, в каких потоках учился, когда и по какой цене.

  > _Частично реализовано: единая БД с Order/Enrollment/Payment связями через User. Admin UI для users, billing, enrollments работает (ZIS-008..ZIS-011). Единой «карточки пользователя» с агрегированной историей пока нет — данные доступны через отдельные разделы админки. → [DRAFT-034](../backlog/drafts/draft-034%20-%20Карточка-пользователя-User-Profile-360.md)_

- 🟡 **AUTH-04.** Если пользователь ранее был зарегистрирован в одном продукте
  и приобретает доступ к другому — его аккаунт автоматически объединяется,
  а не создаётся заново.
  > _Архитектурно заложено (email = единый идентификатор, дедупликация лидов в CRM — ZIS-013), но сценарий merge аккаунтов при миграции из старых систем требует реализации скрипта импорта (DRAFT-008: AirTable Import)._

## 2. Единый личный кабинет пользователя `LK`

**Что болит сейчас:** студенты пишут, что неудобно отдельно авторизоваться
в каждом продукте; они не видят полный список своих покупок.

### Бизнес-кейсы

- 🟡 **LK-01.** Студент заходит в один личный кабинет и видит все свои активные доступы
  (курсы, подписки, сообщества).

  > _Архитектура поддерживает (единая БД Enrollment/Subscription), но отдельного «личного кабинета студента» с агрегированным списком доступов нет. Сейчас студент видит конкретный курс через `/student/lms/[enrollmentId]` (ZIS-012). → [DRAFT-033](../backlog/drafts/draft-033%20-%20Личный-кабинет-студента-Student-Dashboard.md)_

- 🟡 **LK-02.** В личном кабинете отображаются продукты, которые студент ещё не купил,
  с возможностью сразу перейти к покупке.

  > _Не реализовано. Витрина продуктов входит в scope ЛК студента. → [DRAFT-033](../backlog/drafts/draft-033%20-%20Личный-кабинет-студента-Student-Dashboard.md)_

- ⛔ **LK-03.** ~~Покупка нового продукта не требует повторной регистрации или ввода
  платёжных данных, если они уже сохранены.~~
  > _**Деприоритизировано Иваном (25.02.2026):** «Давайте не будем это делать.» Повторная регистрация не нужна (AUTH-01/02 реализованы). Сохранение платёжных данных — избыточная сложность на текущем этапе._

- 🔴 **LK-04.** Студент видит историю своих покупок и может найти чеки и платёжные документы.
  > _Не реализовано. Данные заказов и платежей есть в БД, нужно отобразить в ЛК. MVP: список заказов + ссылка на чек от провайдера. Фидбек Леры, 2026-03-17._

- 🔴 **LK-05.** Студент может обратиться в поддержку из любого места в ЛК и LMS.
  > _Не реализовано. MVP: кнопка «Написать в поддержку» (Telegram deeplink / email). v2: внутренняя тикет-система. Фидбек Леры, 2026-03-17._

## 3. Управление продуктами и курсами `COURSE`

**Что болит сейчас:** запуск каждого нового курса — отдельная кодовая база,
отдельная база данных, отдельная админка. Разворачивание нового продукта занимает
двое суток вместо нескольких часов.

### Бизнес-кейсы

- 🟡 **COURSE-01.** Любой член команды может запустить новый курс (лендинг + оплата + онбординг)
  в рамках единой системы, не создавая отдельной инфраструктуры.

  > _Частично реализовано: Products/Tiers/Cohorts CRUD (ZIS-009), Billing (ZIS-010), Enrollments + Onboarding (ZIS-011). Лендинги пока не реализованы — курс создаётся через админку, но публичного лендинга для каждого продукта нет. → [DRAFT-035](../backlog/drafts/draft-035%20-%20Лендинги-продуктов-Product-Landing-Pages.md)_

- 🟡 **COURSE-02.** Новый курс автоматически появляется в единой админке и в личных кабинетах
  пользователей.

  > _Частично реализовано: в админке — да (ZIS-009: `/admin/products`). В «личном кабинете» студента — нет (см. LK-01). → [DRAFT-033](../backlog/drafts/draft-033%20-%20Личный-кабинет-студента-Student-Dashboard.md)_

- 🟡 **COURSE-03.** Команда может управлять структурой курса из админки: добавлять, удалять,
  редактировать материалы; публиковать и снимать с публикации блоки лендинга.

  > _Частично реализовано: управление структурой курса работает — модули/уроки CRUD, reorder, deactivate/reactivate (ZIS-012: `/admin/lms`). Управление лендингом не реализовано. → [DRAFT-035](../backlog/drafts/draft-035%20-%20Лендинги-продуктов-Product-Landing-Pages.md)_

- ⛔ **COURSE-04.** ~~Курс может иметь разные модели монетизации: разовая покупка, поток с тарифами,
  подписка — без необходимости переписывать инфраструктуру под каждую модель.~~

  > _**Деприоритизировано Иваном (25.02.2026):** «Давайте не будем это делать, это довольно сложная штука.» Базовое: `Product.type = course | subscription`, тарифы и потоки работают (ZIS-009). Подписочная модель отложена на неопределённый срок._

- 🟢 **COURSE-05.** Команда может добавить к существующему курсу новый тариф или новый поток,
  не затрагивая доступы уже зачисленных студентов.
  > _Реализовано в ZIS-009: добавление Tier/Cohort через админку. Существующие Enrollment не затрагиваются._

## 4. Оплата и биллинг `PAY`

**Что болит сейчас:** способы оплаты (Stripe, Яндекс.Сплит) живут как отдельные
интеграции в каждом продукте; апгрейды доступа при связанных покупках выполняются
вручную.

### Бизнес-кейсы

- 🟢 **PAY-01.** При оплате система фиксирует снапшот: цену, валюту и полное описание тарифа
  на момент покупки. Изменение актуальных цен не влияет на исторические записи.

  > _Реализовано в ZIS-010/ZIS-011: `Enrollment.priceSnapshot` фиксирует цену, валюту и тариф на момент покупки. `Order` хранит `amount/currency`._

- 🟡 **PAY-02.** Если покупка одного продукта даёт право на доступ к другому (например, курс
  включает в себя членство в сообществе), система выдаёт эти доступы автоматически
  без ручных действий команды.

  > _Заложено в плане: DRAFT-029 (Cross-product Bonuses, Фаза 3) — автоматическая подписка Maker при покупке КДП/BOOST._

- 🟢 **PAY-03.** Менеджер видит в единой админке все транзакции пользователя
  вне зависимости от того, через какой продукт была совершена оплата.

  > _Реализовано в ZIS-010: `/admin/billing` — список заказов с платежами по всем продуктам._

- 🟡 **PAY-04.** Настройка нового способа оплаты для нового продукта производится через
  интерфейс без необходимости привлекать разработчика для ручного изменения конфигурации.
  > _Не реализовано. Сейчас ЮKassa — единственный провайдер, настраивается через env-переменные. Stripe запланирован как будущая фаза (п. 45 ROADMAP). → [DRAFT-042](../backlog/drafts/draft-042%20-%20Billing-Мульти-провайдер-и-UI-конфигурации.md)_

- 🔴 **PAY-05.** Кассовые чеки формируются и отправляются автоматически по всем платёжным
  провайдерам, включая ручные ссылки на оплату (T-Bank, Яндекс Сплит), а не только
  по ЮKassa и Stripe.

  > _Не реализовано. Сейчас ЮKassa и Stripe автоматически отправляют чеки. При ручном выставлении ссылок через T-Bank/Split чек приходится выбивать вручную через кассу, что создаёт риск ошибок и необходимость чеков коррекции._

- 🔴 **PAY-06.** Менеджер может инициировать возврат средств из админки: система фиксирует
  возврат во внутренних данных (заказ, зачисление, отчётность), независимо от того,
  через какой канал фактически проводится возврат в платёжной системе.

  > _Не реализовано. Сейчас возврат — полностью ручной процесс: поиск карточки в CRM, поиск платежа в платёжной системе, возврат через интерфейс провайдера (ЮKassa, Stripe, Split) или банковскую платёжку (T-Bank). Данные вносятся вручную._

- 🔴 **PAY-07.** Студент может апгрейднуть тариф (например, со «Стандарт» на «Премиум»):
  система автоматически рассчитывает разницу, формирует ссылку на доплату
  и после оплаты обновляет доступы.

  > _Не реализовано. Сейчас апгрейд — ручной процесс: менеджер формирует отдельную ссылку на оплату с суммой разницы, после доплаты вручную меняет тариф в CRM._

- 🔴 **PAY-08.** Цены в иностранных валютах (USD и др.) автоматически пересчитываются
  по актуальному курсу ЦБ РФ, без необходимости ручного обновления.

  > _Не реализовано. Сейчас Настя каждую среду вручную обновляет долларовые цены в админке. Базовая валюта — рубли. Нужен cron-процесс синхронизации курсов._

- 🔴 **PAY-09.** Система скидок и наценок: менеджер применяет скидку как отдельную сущность
  (тип, %, причина), система автоматически рассчитывает НДС-наценку для B2B.
  > _Не реализовано. MVP: ручная скидка менеджером + авторасчёт НДС с ручной корректировкой. Фидбек Насти, 2026-03-11._

- 🔴 **PAY-10.** Перенос студента на другой поток с сохранением или перепривязкой оплаты.
  > _Не реализовано. В MVP — ручной перенос менеджером. Автоматизация — v2. Фидбек Насти, 2026-03-11._

- 🔴 **PAY-11.** Интеграция с T-Bank и Яндекс Сплит через API.
  > _Не реализовано. v2 — требует технического исследования. Фидбек Насти, 2026-03-11._

- 🔴 **PAY-12.** Автоматическое формирование чеков возврата.
  > _Не реализовано. Вне MVP — зависит от провайдера. Фидбек Насти, 2026-03-11._

- 🔴 **PAY-13.** Финансовая аналитика: данные для отчётности по каждому PAY-кейсу.
  > _Не реализовано. MVP: хранение данных для аналитики. Фидбек Насти, 2026-03-11._

## 5. Лендинги и формы сбора лидов `LEAD`

**Что болит сейчас:** подключение лендинга к системе сбора данных
(например, отклики по вакансиям, заявки на курс) занимает много времени
из-за ручного маппинга полей.

### Бизнес-кейсы

- 🟡 **LEAD-01.** Любой член команды может создать новый лендинг (курс, вакансия, акция)
  и подключить его к единой системе сбора данных без длительного технического
  согласования. Лендинги должны создаваться **в промышленных масштабах**,
  а изменения в них — вноситься быстро и легко.

  > _⚡ **Приоритет Ивана (25.02.2026):** «Очень важно, чтобы мы могли лендинги клепать в промышленных масштабах и вносить изменения в промышленных масштабах.»_
  >
  > _Частично реализовано: TypeForm webhook + внутренняя форма `/lead` + публичный API `/api/leads` (ZIS-013). Создание самих лендингов вне ZIS, но подключение к системе — через стандартные webhook'и. Самостоятельное подключение нового лендинга командой без разработчика пока невозможно. → [DRAFT-035](../backlog/drafts/draft-035%20-%20Лендинги-продуктов-Product-Landing-Pages.md)_

- 🟡 **LEAD-02.** Лид, пришедший с любого лендинга, автоматически попадает в единую базу
  и отображается в соответствующем разделе админки. Менеджер работает с лидом через
  карточку с операционными статусами воронки, логом заметок и счётчиками по этапам.

  > _Базовый сценарий реализован (ZIS-013). Доработки по фидбеку Руслана (2026-03-17): карточка лида `/admin/leads/[id]`, операционные статусы (new → in_progress → negotiation → converted/lost + lostReason), structured notes, счётчики по статусам, быстрые ссылки Telegram/email._

- 🟡 **LEAD-03.** Лендинг вакансий: отклик на вакансию сохраняется в системе и отображается
  в разделе «Вакансии» единой админки с возможностью управлять статусами кандидатов.
  > _⚡ **Уточнение Ивана (25.02.2026):** «Нужно, чтобы админка вакансий, которую сделала Алёна, интегрировалась в единую админку.» Алёна Тушминцева уже создала страницу вакансий с отдельной админкой — её нужно встроить в ZIS Admin UI. → [DRAFT-043](../backlog/drafts/draft-043%20-%20HR-Вакансии-модуль-управления-откликами.md)_

- 🔴 **LEAD-04.** Полная история коммуникации с лидом: Telegram-переписка, email, заметки
  менеджера — всё в одной карточке. Менеджер может общаться с лидом из интерфейса ZIS.
  > _Не реализовано. MVP-часть (structured notes) входит в LEAD-02. Полная история коммуникации и отправка из ZIS — v2. Фидбек Руслана, 2026-03-17._

- 🔴 **LEAD-05.** Задачи по лидам: менеджер получает задачи с дедлайнами, видит список дел
  на день, автозадача «связаться» для новых лидов.
  > _Не реализовано. v2 — в MVP задачи заменяются фильтрами по статусу и lastContactedAt. Фидбек Руслана, 2026-03-17._

## 6. LMS (Learning Management System) `LMS`

**Что болит сейчас:** текущая LMS неудобна и для студентов, и для команды;
запуск нового потока требует значительных усилий; администрирование контента
завязано на разработчика.

### Бизнес-кейсы

#### Для студентов

- 🟢 **LMS-01.** После покупки курса студент автоматически получает доступ в LMS —
  отдельная регистрация не нужна.

  > _Реализовано в ZIS-011 + ZIS-012: после оплаты запускается onboarding process, шаг `grant_lms` выдаёт доступ автоматически. Студент заходит через `/student/lms/[enrollmentId]`._

- 🟡 **LMS-02.** Студент видит свой прогресс по курсу и может добавлять материалы
  в избранное и делать личные заметки.

  > _Частично реализовано: прогресс по урокам (отметка «Изучено») работает (ZIS-012). Избранное и личные заметки не реализованы. → [DRAFT-036](../backlog/drafts/draft-036%20-%20LMS-Избранное-и-личные-заметки-студента.md)_

- 🟡 **LMS-03.** Студент видит групповой прогресс (например, статистику своей учебной группы)
  для элементов соревновательной механики.

  > _Заложено частично: куратор видит матрицу прогресса группы (ZIS-012). Для студента аналогичного представления нет. → [DRAFT-037](../backlog/drafts/draft-037%20-%20LMS-Групповой-прогресс-для-студента.md)_

- 🟢 **LMS-04.** Куратор видит матрицу прогресса своих студентов: кто что просмотрел,
  что ещё не начал.

  > _Реализовано в ZIS-012: `/curator/lms/[cohortId]` — таблица «студент × урок»._

- 🟢 **LMS-05.** Команда может добавлять, редактировать, удалять материалы курса
  из административного интерфейса без привлечения разработчика.

  > _Реализовано в ZIS-012: `/admin/lms` — CRUD модулей/уроков, reorder, revision history, deactivate/reactivate._

- 🟡 **LMS-06.** Команда может добавить в LMS новую функциональную страницу (например,
  страницу группы со статистикой) и опубликовать её самостоятельно — через вайб-кодинг,
  без необходимости ждать разработчика.

  > _⚡ **Уточнение Ивана (25.02.2026):** «Нужно, чтобы можно было вайб-кодить условно что угодно, что не противоречит данным и расширяет функциональность. Нужно, чтобы здесь позволялось такое расширение.» Не конструктор страниц, а расширяемая архитектура для быстрой разработки новых страниц. → [DRAFT-044](../backlog/drafts/draft-044%20-%20LMS-Конструктор-страниц.md)_

- 🟢 **LMS-07.** Онбординг студента в новый поток (добавление в чаты, предоставление
  доступов) происходит автоматически после подтверждения оплаты.
  > _Реализовано в ZIS-011 + ZIS-014: полный onboarding process из 5 шагов (grant_lms, create_onboarding_page, subscribe_calendar, assign_curator, send_welcome), запускается автоматически после payment.success webhook._

#### Для команды (аналитика и управление)

  > _Не реализовано. Требуется расширяемая plugin-архитектура или модульная система, позволяющая командным разработчикам быстро добавлять новые разделы в единую админку. → [DRAFT-045](../backlog/drafts/draft-045%20-%20Admin-No-code-инструменты.md)_

## 12. B2B-документооборот и инвойсинг `INV`

**Что болит сейчас:** счета и договора для юрлиц формируются через Google-скрипт
с ограничениями (только одна цена, только один продукт одновременно). Для Boost
и интенсивов счета выставляются вручную. Реестр документов разрознен по таблицам
и папкам. При масштабировании продаж Настя становится узким горлышком.

### Бизнес-кейсы

- 🟡 **INV-01.** B2B-клиент может самостоятельно сформировать счёт на оплату
  через публичную форму: указать ИНН (реквизиты подтягиваются из ЕГРЮЛ),
  выбрать продукт, поток и тариф — и получить PDF-счёт.

  > _Реализовано в ZIS: `/b2b/start` → `/company/register` — B2B-клиент заполняет форму (ИНН/КПП, юр. адрес, контактное лицо), выбирает продукт/поток/тариф. После одобрения менеджером создаётся Company + Contract + Invoice. PDF-документы генерируются через pg-boss jobs. Автоподтягивание из ЕГРЮЛ не реализовано — реквизиты вводятся вручную._

- 🟢 **INV-02.** При формировании счёта для B2B клиент может сразу указать данные
  студентов (ФИО, email), чтобы после оплаты онбординг запускался автоматически,
  без дополнительной переписки.

  > _Реализовано в ZIS: компания ведёт реестр сотрудников (`/company` → CompanyEmployee), при создании enrollment request выбирает конкретных сотрудников. После подтверждения оплаты менеджером (`markCompanyRequestPaid`) система автоматически создаёт User, Order, Enrollment для каждого сотрудника и запускает онбординг._

- 🟢 **INV-03.** Все документы (счета, договора, инвойсы) по всем продуктам и клиентам
  хранятся в едином реестре с привязкой к компании, а не в разрозненных таблицах и папках.

  > _Реализовано в ZIS: `/company/documents` — единый реестр документов (invoices, contracts, acts) с привязкой к компании. Менеджер видит документы через `/admin/companies/[id]/invoices` и `/admin/companies/[id]/contracts`. Все документы хранятся в единой БД с привязкой к Company._

- 🟡 **INV-04.** Нумерация счетов и договоров формируется автоматически с учётом
  продукта (префиксы), года и порядкового номера — в настраиваемом формате.

  > _Частично реализовано в ZIS: автонумерация работает по формату `INV-YYYYMMDD-XXXXXX` (счета), `CTR-YYYYMMDD-XXXXXX` (договоры), `ACT-YYYYMMDD-XXXXXX` (акты). Формат фиксирован — настраиваемых префиксов по продуктам пока нет._

- 🔴 **INV-05.** Система поддерживает два типа договоров: шаблонный (типовая оферта,
  не требует согласования) и кастомный (загружается вручную после согласования
  с юристом). Оба типа отображаются в едином реестре с привязкой к компании.

  > _Не реализовано. Сейчас типовой договор — ссылка на оферту на сайте. Кастомные договоры согласуются через юриста Володю и хранятся отдельно._

- 🔴 **INV-06.** Для иностранных компаний формируются инвойсы на английском языке.
  Для отдельных стран (Казахстан, Германия) поддерживаются специфические требования
  к оформлению документов.

  > _Не реализовано. Сейчас Настя формирует международные инвойсы вручную (PDF без подписи). Есть шаблоны на английском и двуязычный для Казахстана. Германия требует дополнительные строки._

## 13. Дедупликация и регистрация при оплате `DEDUP`

**Что болит сейчас:** в CRM появляются дубли лидов: один и тот же человек
создаёт несколько карточек при повторных попытках оплаты (брошенные корзины).
Команда тратит время на разбор дублей.

### Бизнес-кейсы

- 🔴 **DEDUP-01.** При оформлении заказа пользователь автоматически регистрируется
  в системе (или привязывается к существующему аккаунту по email), чтобы все его
  попытки оплаты, брошенные корзины и завершённые покупки были в одной карточке.

  > _Не реализовано. Сейчас каждая попытка оплаты создаёт отдельный лид в CRM. Решение, озвученное на созвоне: регистрировать пользователя в момент создания заказа (до перехода к оплате), а если аккаунт уже существует — просить авторизоваться._

- 🟡 **DEDUP-02.** Повторный студент (ранее проходивший обучение) при возврате
  на новый поток автоматически распознаётся системой, и ему предлагается
  соответствующая скидка без ручного вмешательства менеджера.

  > _Частично заложено: единая БД пользователей и зачислений позволяет определить повторного студента. Автоматическое применение скидки при повторном прохождении не реализовано. Сейчас Настя формирует ссылку с другой суммой вручную._

## 14. Маркетинг и CRM `MKT`

**Что болит сейчас:** маркетинговые данные (UTM-метки, источники трафика) теряются
по дороге от рекламы до CRM; сегментация базы и рассылки делаются вручную через
сторонние инструменты (BotHelp, Google Sheets); нет единого представления воронки
конверсии.

### Бизнес-кейсы

- 🔴 **MKT-01.** UTM-трекинг и атрибуция: система сохраняет источник трафика
  (UTM-метки и referer) при создании лида или заказа для маркетинговой аналитики.
  > _Не реализовано. MVP: хранение UTM-меток в Lead и Order (поле `utmData: JSON`). Аналитические дашборды — вне ZIS (vibe-coding). Фидбек Лады, 2026-03-13._

- 🔴 **MKT-02.** Сегментация базы: маркетолог может фильтровать пользователей и лидов
  по продукту, статусу, источнику, активности для таргетированных коммуникаций.
  > _Не реализовано. v2 — ZIS хранит данные, сегментация строится внешними инструментами или vibe-coding. Фидбек Лады, 2026-03-13._

- 🔴 **MKT-03.** Рассылки: маркетолог может отправлять целевые рассылки по сегментам
  базы через email и Telegram.
  > _Не реализовано. Не MVP — рассылки остаются в BotHelp. ZIS предоставляет данные через API. Фидбек Лады, 2026-03-13._

- 🔴 **MKT-04.** Воронка конверсии: маркетолог видит путь пользователя от первого контакта
  до покупки и может анализировать, где теряются клиенты.
  > _Не реализовано. Не MVP — строится вне ZIS (vibe-coding поверх данных ZIS). ZIS обеспечивает хранение timeline-событий (AUTH-03). Фидбек Лады, 2026-03-13._

- 🔴 **MKT-05.** Реферальная программа: выпускник может пригласить друга и получить бонус
  при его покупке.
  > _Не реализовано. Не MVP — требует решения Вани. Модель: реферальный код → скидка/бонус при конверсии. Фидбек Лады, 2026-03-13._

- 🔴 **MKT-06.** A/B тестирование: маркетолог может тестировать разные варианты
  страниц/предложений и видеть, какой работает лучше.
  > _Не реализовано. Не MVP — реализуется внешними инструментами или vibe-coding. Фидбек Лады, 2026-03-13._

- 🔴 **MKT-07.** Маркетинговая секция в AdminUI: менеджер видит ключевые маркетинговые
  метрики и имеет доступ к инструментам маркетинга.
  > _Не реализовано. MVP: счётчики набора на потоки (виджет на дашборде, часть ADMIN-01). Полноценная маркетинговая секция — v2. Фидбек Лады, 2026-03-13._

- 🟡 **MKT-08.** Маркетинговый бот: Telegram-бот как канал привлечения и прогрева лидов
  с отслеживанием источника.
  > _Частично реализовано: бот работает (ZIS-014). MVP-доработка: парсинг UTM из `?start=` параметра. Фидбек Лады, 2026-03-13._

## Открытые вопросы (требуют отдельного обсуждения)

1. **Флоу студента (B2C)** — Алена готовит детальную схему пути студента от первого касания
   до завершения курса. На её основе появятся дополнительные бизнес-кейсы.
   _(По итогам созвона 24.02: Алёна начала работу, обещала прислать на следующий день.)_
2. **Флоу ботов** — Лера подготовила схему в Figma и описание работ бота прямой
   коммуникации со студентом. Основные сценарии зафиксированы в BOT-05..08.
   Остаётся вопрос: один бот на всё (онбординг + саппорт + upsell + AI-ассистент)
   или несколько специализированных ботов. _(Обсуждено на созвоне 25.02.)_
3. **Подписочная модель (Aura)** — финансовый учёт подписок: Настя видит только
   суммы списаний, без детализации. Подписочную систему делали Ваня и Илья.
   Нужен отдельный разбор, как учитывать подписки в единой системе.
   _(Вынесено из созвона 24.02 — тема не раскрыта, требует отдельного звонка.)_
4. **Интеграция с 1С и Диадок** — 1С Фреш не интегрируется с Диадоком напрямую.
   УПД выгружаются из 1С в XML и загружаются в Диадок вручную.
   Автоматизация ЭДО — отложенная задача.
5. **Крипто-платежи** — существуют, но нельзя афишировать на российском рынке.
   Потребуется гео-фильтрация (geo-IP) для показа этой опции.
6. **Дизайн-система** — вопрос пока открыт; команда должна предоставить Жене
   референсы и гайдлайны.
7. **Разграничение ZIS и Aura** — Aura развивается как отдельный продукт,
   возможно с выходом на западный рынок. Нужно уточнить, какие бизнес-кейсы
   Auры входят в текущий скоп ZIS, а какие — в отдельный контур.
8. **Миграция данных из AirTable** — импорт всех исторических студентов и данных
   в единую БД. Мягкий переход: тестирование на пилотных пользователях,
   затем полный перенос. До потока 40 КДП — один архивный поток,
   с потока 40 — поточные данные доступны.
9. **LMS как отдельный проект** — Иван считает, что полноценная LMS станет
   отдельным большим проектом со своими сценариями приёмки. Текущая LMS в ZIS —
   базовая версия. _(Озвучено 25.02.2026.)_
10. **Полнота списка сценариев** — Иван отметил, что текущий список точно не полный,
    и вся команда должна пройтись по нему и дополнить недостающие сценарии.
