# GLOSSARY.md — глоссарий сущностей ZIS

Документ фиксирует назначение ключевых сущностей системы на уровне моделей Prisma.

## Границы схем БД

- `public` — доменная схема приложения, управляется Prisma (`prisma/schema.prisma`).
- `pgboss` — служебная схема очередей pg-boss (jobs, queues, schedules), не является доменной моделью Prisma.

Правило: бизнес-логика и продуктовые данные живут в `public`; операционные данные очередей — в `pgboss`.

## Auth и доступ

| Модель | Зачем нужна |
| --- | --- |
| `User` | Базовая учетная запись пользователя с ролью (`admin/manager/curator/student`) и каналами связи. |
| `Session` | Активные сессии авторизации пользователя (better-auth). |
| `Account` | Связанные auth-аккаунты/провайдеры для входа (better-auth). |
| `Verification` | Токены/коды верификации (OTP/email подтверждения и служебные проверки). |
| `ApiToken` | API-токены для системных интеграций и агентских вызовов API. |

## Организации и каталог продуктов

| Модель | Зачем нужна |
| --- | --- |
| `Company` | B2B-компания: юридический и операционный контекст клиента. |
| `CompanyEmployee` | Кадровый список сотрудников компании (включая ещё не зарегистрированных: `userId = null`). |
| `CompanyRequest` | Заголовок B2B-заявки на обучение: договор/продукт/поток, сумма, статус исполнения, обязательная связь с `Invoice`. |
| `CompanyRequestEmployee` | Строка B2B-заявки: выбранный сотрудник + snapshot контактов + статус исполнения (user/order/enrollment). |
| `Product` | Продукт/курс/подписка как верхний уровень коммерческого предложения. |
| `Tier` | Тариф продукта (цена, лимиты, доступы, условия). |
| `Cohort` | Поток/когорта для запуска курса с датами и статусом. |

## Обучение (LMS)

| Модель | Зачем нужна |
| --- | --- |
| `Enrollment` | Зачисление пользователя в продукт/когорту; создаётся только после подтверждённой оплаты и всегда связано с `Order` (`orderId` обязательный). |
| `Module` | Учебный модуль внутри курса. |
| `Lesson` | Урок внутри модуля (контейнер текущей версии контента). |
| `LessonRevision` | Версия урока для истории изменений и контролируемых обновлений контента. |
| `CohortContent` | Снапшот/переопределение контента для конкретной когорты (fork/синхронизация). |
| `Progress` | Прогресс студента по урокам/модулям в рамках зачисления. |
| `Homework` | Домашние задания, привязанные к урокам/модулям. |
| `HomeworkSubmission` | Сдачи ДЗ студентами и цикл проверки/ревизий. |
| `Schedule` | Расписание живых активностей (лекции, воркшопы, Q&A). |
| `Attendance` | Фиксация посещаемости по расписанию. |
| `CuratorGroup` | Группы кураторов для распределения и сопровождения студентов. |
| `Certificate` | Сертификат по завершению обучения и факту выдачи. |

## CRM и продажи

| Модель | Зачем нужна |
| --- | --- |
| `Lead` | Лид в воронке продаж от источника до конверсии/потери. |
| `Order` | Коммерческий заказ (B2C/B2B), центральная сущность для запуска `Enrollment` после статуса `paid`. |
| `Payment` | Транзакция оплаты по заказу (провайдер, статус, сумма, чек). |
| `PromoCode` | Промокоды и бонусные механики в продажах. |

## Биллинг и документы

| Модель | Зачем нужна |
| --- | --- |
| `Invoice` | Счет на оплату (чаще B2B), обязательная связь с `Contract`, жизненный цикл от draft до paid/overdue; может покрывать несколько `Order` через `Order.invoiceId`. |
| `Contract` | Договор с клиентом (условия, статус подписания/действия). |
| `Act` | Акт выполненных работ для закрывающих документов; в текущем контуре инвариант: один `Act` на один `Invoice` (1:1). |
| `Subscription` | Подписка (например, Maker/Founder) со статусом и периодами. |

## Telegram и коммуникации

| Модель | Зачем нужна |
| --- | --- |
| `TelegramChat` | Реестр Telegram-чатов/каналов, куда управляется доступ. |
| `TelegramAccount` | Связь пользователя системы с его Telegram-аккаунтом. |
| `ChatMembership` | Состояние участия пользователя в конкретном Telegram-чате. |
| `Notification` | Очередь/история отправки уведомлений (email/telegram) и их статусы. |

## Feedback и аналитика опыта

| Модель | Зачем нужна |
| --- | --- |
| `FeedbackCampaign` | Кампания сбора обратной связи (триггеры, каналы, окно проведения). |
| `FeedbackResponse` | Ответ пользователя в рамках кампании обратной связи. |

## События, процессы, аудит и интеграции

| Модель | Зачем нужна |
| --- | --- |
| `Event` | Бизнес-события системы для аналитики, трассировки действий и автоматизаций. |
| `Process` | Оркестрация многошаговых бизнес-процессов (onboarding/offboarding и др.). |
| `AuditLog` | Журнал изменений для compliance: кто, когда и что изменил. |
| `WebhookLog` | История входящих webhook-вызовов от внешних интеграций и статус обработки. |

## Как использовать глоссарий

- Для изменения модели сначала сверяться с `docs/01-domain-model.md` и `prisma/schema.prisma`.
- Для инцидентов по очередям проверять данные в `pgboss`, но доменные причины искать в `public`.
- Для постановки задач в Plan Mode использовать названия сущностей из этого глоссария как единый словарь команды.
